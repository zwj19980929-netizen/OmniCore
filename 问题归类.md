# 代码层面
1. core/graph.py 职责过载与遗留代码清理
现状：graph.py 目前有将近 1800 行代码。里面不仅包含了 LangGraph 的 DAG 图定义，还塞满了大量的辅助函数（如处理乱码的 _repair_mojibake_text、处理数据提取的 _extract_structured_findings），甚至还保留了大量 _legacy_ 前缀的废弃/旧版节点函数。
优化建议：

模块拆分：将纯文本处理、数据格式化的辅助函数抽离到 utils/formatters.py 或 utils/state_helpers.py 中。

清理历史包袱：如果 _legacy_replanner_node 等代码已经不再被图结构使用，建议果断删除（可以依赖 Git 记录），或者将新旧版本放在不同的模块中，保持 graph.py 的纯粹性——它应该只负责“定义节点”和“定义边”。

2. agents/browser_agent.py 中的内联 JavaScript 字符串
现状：在 _build_fallback_semantic_snapshot 和 _extract_interactive_elements 等方法中，直接在 Python 字符串里硬编码了长达上百行的 JavaScript 代码。
优化建议：

分离 JS 代码：这会让 JS 失去语法高亮和代码检查。建议在项目中建立一个 scripts/js_injections/ 文件夹，将这些脚本保存为独立的 .js 文件。

实现说明：在 Python 中读取它们：
```python
# 例如在 utils 某处实现一个加载器
import os
def load_js_script(script_name: str) -> str:
    path = os.path.join("scripts", "js_injections", f"{script_name}.js")
    with open(path, "r", encoding="utf-8") as f:
        return f.read()

# 在 BrowserAgent 中使用：
js_code = load_js_script("extract_elements")
r = await self._call_toolkit("evaluate_js", js_code)```
3. core/llm.py 的代码重复与 JSON 解析脆弱性
现状：

同步/异步代码重复：chat 和 achat 两个方法中，关于网络错误重试、Token 降级重试、API Fallback 的逻辑几乎是复制粘贴的。

JSON 解析嵌套过深：parse_json_response 方法使用了一层又一层的 try-except 结合正则表达式来“强行”提取 JSON，这在 LLM 返回不可控时很容易出 Bug。
优化建议：

提取重试装饰器：将重试和降级逻辑抽象为一个装饰器，或者一个通用的内部执行函数，消除 chat 和 achat 之间的重复代码。

利用结构化输出：现在绝大多数 LLM（包括 OpenAI, DeepSeek 等）都已经原生支持了 response_format={"type": "json_object"} 甚至 json_schema 强制结构化输出。可以进一步加强提示词约束，并考虑引入轻量级的 Pydantic 解析库，替代手写的正则猜测逻辑。

4. BrowserAgent.run 方法过于庞大（God Method）
现状：run 方法是整个浏览器 Agent 的心脏，处理了初始化、绕过验证码、意图推断、无限重试循环、视觉卡点检测、以及最终的截图保存等，逻辑极度耦合。
优化建议：

虽然你已经开始了三层架构（Perception 感知, Decision 决策, Execution 执行）的重构，但主循环依然在 run 里。建议将 for step_no in range(1, max_steps + 1): 这个大循环重构为一个有限状态机 (FSM)。

将诸如“绕过反机器人验证”、“提取数据并判断是否满足”这些逻辑抽离为独立的单步处理函数。

5. core/router.py 中的“魔法数字” (Magic Numbers)
现状：在 _score_registered_tool 方法中，有很多硬编码的分数权重（如 score += 10, score += 7, score += 4 * ...）。
优化建议：

将这些计分权重提取为类的常量或者配置项。这样未来如果要微调路由匹配的准确度，只需修改顶部的常量字典，而不需要深入到代码逻辑中。



# 网页操作问题
1. 脆弱的搜索引擎硬编码 (Hardcoded Selectors)
问题所在：
在 _extract_search_results_data 和相关等待逻辑中，你硬编码了大量搜索引擎的 CSS 选择器（虽然代码中提到委托给了 get_search_result_selectors，但内部逻辑和 JS 注入中依然高度依赖特定结构，比如查找 div.g, .b_ans 等）。
现代搜索引擎（Google, Bing 等）的 DOM 结构变化极其频繁（通常带有反爬混淆）。这种依赖 DOM 选择器硬刮搜索引擎的做法，极其脆弱，维护成本极高。

2. “像素级”视觉校验的误报率极高 (Pixel Diffing Issues)
问题所在：
在动作校验 _verify_action_effect 和 等待校验中，你使用了图像差异对比（screenshots_differ 结合 VISION_PIXEL_DIFF_THRESHOLD）来判断页面是否发生了变化。
这在真实的 Web 页面中会是个巨大的坑：轮播图（Carousels）、动态广告、GIF、甚至输入框内光标的闪烁，都会导致像素级 Diff 检测到“页面有变化”，从而使 Agent 陷入逻辑误判（例如认为点击已经生效，或者无限等待页面静止）。

3. JavaScript 提取逻辑无法穿透复杂的 Shadow DOM 和 iframe
问题所在：
你通过注入长达上百行的 JS 代码（evaluate_js）来获取页面交互元素。目前的 JS 逻辑主要是 document.querySelectorAll 结合一些可见性判断。
现代前端项目（React/Vue 复杂组件，尤其是 Web Components）大量使用 Shadow DOM。此外，很多关键交互（如登录框、支付模块）会嵌套在 iframe 中。你目前的提取方式会丢失这部分内容。


4. 昂贵且可能导致超时的 Vision Fallback (视觉回退)
问题所在：
在 run 循环中，如果 DOM 层无法找到可操作元素，代码会频繁 Fallback 到 _decide_action_with_vision，将页面截图发给视觉大模型（Vision LLM）。
视觉模型的延迟通常在几秒到十几秒不等，而且成本高昂。如果 Agent 在复杂页面上迷失，它可能会在死循环中不断截图发送，消耗大量 Token 和时间。
