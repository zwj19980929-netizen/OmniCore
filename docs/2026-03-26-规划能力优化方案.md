# OmniCore 规划能力优化方案

日期: 2026-03-26

## 1. 现状总结

当前规划流水线：

```
用户输入 → Router(LLM) → 任务DAG → PolicyEngine → HumanConfirm
  → ParallelExecutor → Validator(硬规则) → Critic(LLM)
  → [失败] → Replanner(LLM, ≤3次) → 重新执行
  → [通过] → Finalize → 用户输出
```

**核心组件与职责：**

| 组件 | 文件 | 职责 |
|------|------|------|
| Router | `core/router.py` + `prompts/router_system.txt` | 意图识别、任务分解、路径选择 |
| TaskPlanner | `core/task_planner.py` | 任务标准化、策略映射 |
| TaskExecutor | `core/task_executor.py` | 依赖推断、就绪批次调度 |
| Validator | `agents/validator.py` | 确定性硬规则验证 |
| Critic | `agents/critic.py` | LLM审查 + 视觉复核 |
| Replanner | `core/graph.py:793-1052` | 失败分析、策略切换 |
| AdaptiveReroute | `core/graph.py:2491-2547` | 轻量级提前终止 |
| PAOD | `agents/paod.py` | 失败分类、安全表达式求值 |

**已有的优点：**
- 三层失败分析（执行层/路径层/方向层）在 Replanner 提示词中清晰定义
- 路径选择思维在 Router 提示词中强调"阻力最小"
- 自适应跳过（单答案意图/数量达标/依赖阻塞）
- 策略历史追踪（replan_history）防止重复失败路径
- 确定性快速通道（终端命令跳过 LLM）

## 2. 发现的问题

### 2.1 规划前缺少可行性预检（高优先级）

**问题**：Router 产出的任务 DAG 直接进入执行，没有任何静态验证。

**具体表现**：
- 循环依赖无检测：`task_1 depends_on task_2, task_2 depends_on task_1` 会导致两个任务永远无法就绪
- 孤立依赖无检测：`task_2 depends_on task_99`，但 `task_99` 不存在于队列中
- 不可能的 Worker 组合不拦截：例如 `file_worker` 写文件的 `data_source` 指向一个不会产出数据的 `system_worker` 任务

**影响**：浪费一整轮执行 + 验证 + 重规划的时间，才发现计划本身就有结构错误。

**建议**：在 `route_node` 之后、`parallel_executor_node` 之前插入一个轻量级 `plan_validator` 阶段（纯 Python，零 LLM 调用），检查：
1. 依赖图无环（拓扑排序检测）
2. 所有 `depends_on` 引用的 `task_id` 存在于队列中
3. `data_source` / `data_sources` 引用的任务类型确实能产出数据
4. 检测到结构问题时立即修复（删除无效依赖）或重新调用 Router

### 2.2 Replanner 缺少结构化失败上下文（高优先级）

**问题**：Replanner 给 LLM 的上下文是拼接的自然语言字符串，关键信息容易被截断或淹没。

**具体表现（`graph.py:860-910`）**：
- `failure_summary` 是 `"\n".join(failures)` 的纯文本拼接
- 执行轨迹只取最后 3 步，且截断到 80 字符
- `replan_history` 只记录 `strategies` 和 `urls`，不记录失败类型和错误原因
- PAOD 的 `classify_failure()` 在 `task_executor.py` 中调用了，但分类结果（`failure_type`）未传入 Replanner 的结构化上下文

**影响**：Replanner LLM 做决策时信息不充分，容易选择本质上相同的失败路径。

**建议**：
1. `replan_history` 扩展为：
   ```python
   {
       "round": 1,
       "strategies": [...],
       "urls": [...],
       "failure_types": ["blocked_or_captcha", "selector_not_found"],
       "failure_layer": "path",  # 执行层/路径层/方向层
       "error_summaries": ["403 Forbidden", "No matching elements"]
   }
   ```
2. 给 Replanner 的 prompt 改用结构化 JSON 而非拼接文本：
   ```json
   {
       "user_request": "...",
       "attempt_number": 2,
       "is_final": false,
       "failed_tasks": [
           {
               "description": "...",
               "worker_type": "browser_agent",
               "url": "...",
               "failure_type": "blocked_or_captcha",
               "failure_layer": "path",
               "error": "...",
               "last_3_steps": [...]
           }
       ],
       "history": [...],
       "authoritative_url": "..."
   }
   ```

### 2.3 两套 Replanner 实现共存（中优先级）

**问题**：`graph.py` 中存在 `_legacy_replanner_node_v2`（793-1052）和 `_legacy_replanner_node`（1056-1230+）两套实现，逻辑高度重叠但细节不同。

**具体表现**：
- `_legacy_replanner_node_v2` 使用中文提示词 `replanner_system`，集成了 PUA Coach
- `_legacy_replanner_node` 使用英文提示词 `replanner_system_en`，逻辑更干净
- 实际注册到 graph 的是哪个取决于 `replanner_node` 的赋值（需确认）
- `_legacy_replanner_node_v2` 第 882 行有一个 bug：`error = result.get("error", "未知错误")` 中的 `result` 引用了上下文中已被覆盖的变量

**影响**：维护成本高，两套实现可能行为不一致；有未触发的 bug。

**建议**：
1. 确认当前活跃的 Replanner 是哪个，删除另一个
2. 修复 `_legacy_replanner_node_v2:882` 的变量引用 bug
3. 统一为一套实现，保留中英文提示词的选择能力

### 2.4 Validator 和 Critic 职责边界模糊（中优先级）

**问题**：Validator（硬规则）和 Critic（LLM）各自都做"数据质量判断"，标准可能冲突。

**具体表现**：
- Validator 在 `browser_agent` 任务中检查 `data` 是否为空（`validator.py:150-152`）
- Critic 的 `_deterministic_review_result` 也检查数据量和质量（`critic.py:113-168`）
- 一个场景：Validator 通过（data 非空），Critic 拒绝（数量不够）→ 进入 Replanner
- 另一个场景：Validator 拒绝（data 为空）→ 直接进 Replanner，Critic 完全不参与

**影响**：相同类型的失败在不同路径下可能有不同的处理方式，Replanner 收到的上下文不一致。

**建议**：
- Validator 只做结构性验证（类型正确、非 None、非空、文件存在）
- Critic 做语义性验证（数量达标、内容相关、列表项有效）
- 明确：Validator 的 `passed=False` 意味着"执行层失败"（可重试），Critic 的 `approved=False` 意味着"结果层不足"（需换策略）
- 在 Replanner 中根据是 Validator 失败还是 Critic 失败来选择不同的重规划策略

### 2.5 Router 缺少成本/复杂度预估（低优先级）

**问题**：所有任务只用 `priority`（1-10）排序，没有预估执行复杂度。

**具体表现**：
- `browser_agent` 任务（需要启动浏览器、多步交互）和 `web_worker` 任务（单次 HTTP 请求）被同等对待
- 3 个可并行的 `web_worker` 和 1 个 `browser_agent` 组成 4 任务的批次，但 `browser_agent` 需要 30 秒而 `web_worker` 只需 2 秒

**影响**：轻任务被重任务拖慢；无法做出"先试快路径，失败再走慢路径"的智能选择。

**建议**：
- 给 TaskItem 添加 `estimated_cost` 字段（`"low"/"medium"/"high"`）
- Router 提示词中要求标注每个任务的预估成本
- 执行器在批次调度时考虑成本：先执行低成本任务，如果已经满足需求则跳过高成本任务
- 与 `should_skip_remaining_tasks` 联动：低成本任务完成且满足条件 → 跳过高成本任务

### 2.6 成功路径记忆未充分利用（低优先级）

**问题**：Router 接收 `successful_paths` 和 `failure_patterns`，但这些信息只是追加到 prompt 中，没有在 Replanner 中使用。

**具体表现**：
- `router.py:903-946` 把成功路径和失败模式作为提示注入 Router
- Replanner 完全不接收这些信息
- 当 Replanner 需要"换一条完全不同的路径"时，没有历史成功路径作为参考

**影响**：Replanner 每次重规划都是"从零开始想新策略"，而不是"参考历史上成功过的策略"。

**建议**：
- 把 `successful_paths` 也传入 Replanner 的上下文
- 在 Replanner 提示词中增加："以下是类似任务曾经成功的路径，优先考虑这些已验证的方案"

### 2.7 `_legacy_replanner_node_v2` 中的编码问题（低优先级）

**问题**：`graph.py` 中大量中文字符串出现 mojibake（例如 `鎵€鏈変换鍔℃墽琛屽畬鎴?`），表明文件经过了错误的编码转换。

**具体表现**：`graph.py:1062, 1092-1096, 1122-1127` 等位置的中文字符串已损坏。

**影响**：`_legacy_replanner_node` 中的日志和错误消息不可读；如果损坏字符串被传入 LLM prompt，会浪费 token 并干扰理解。

**建议**：
- 修复损坏的字符串，或直接删除已废弃的 Replanner 实现（参见 2.3）
- 如果保留，使用 UTF-8-BOM 重新保存受影响的代码段

## 3. 优化实施优先级

| 优先级 | 编号 | 优化项 | 预估工作量 | 预期收益 |
|--------|------|--------|-----------|----------|
| P0 | 2.1 | 规划前可行性预检 | 小（纯 Python，~100行） | 避免结构性死循环，省下无效执行轮次 |
| P0 | 2.2 | 结构化失败上下文 | 中（改 Replanner 上下文构建 + 提示词） | 显著提升重规划成功率 |
| P1 | 2.3 | 统一 Replanner 实现 | 小（删除冗余代码） | 消除 bug，降低维护成本 |
| P1 | 2.4 | Validator/Critic 职责明确化 | 小（调整判断条件和路由逻辑） | 重规划策略更精准 |
| P2 | 2.5 | 任务成本预估 | 中（Router 提示词 + 调度逻辑） | 执行效率提升，减少无谓等待 |
| P2 | 2.6 | 成功路径传入 Replanner | 小（数据传递 + 提示词调整） | 重规划命中率提升 |
| P2 | 2.7 | 编码问题修复 | 小（文件重新编码） | 日志可读性 |

## 4. P0 优化项的具体设计

### 4.1 规划前可行性预检 (Plan Pre-Validator)

**新增位置**：`core/plan_validator.py`（新文件）

**注册为 Stage**：order=15（在 Router 的 10 之后、human_confirm 的 20 之前）

```python
def validate_plan(task_queue: List[TaskItem]) -> PlanValidationResult:
    """
    纯 Python 静态检查，零 LLM 调用。

    检查项：
    1. 循环依赖检测（Kahn 拓扑排序）
    2. 孤立依赖检测（depends_on 引用不存在的 task_id）
    3. data_source 引用合法性（目标任务能产出数据）
    4. 空任务队列但非 direct_answer 场景

    返回：
    - passed: bool
    - issues: List[str]  # 发现的问题描述
    - auto_fixes: List[str]  # 已自动修复的问题
    """
```

**自动修复策略**：
- 孤立依赖 → 自动删除无效的 `depends_on` 条目
- 循环依赖 → 删除形成环的最低优先级边
- 不可修复 → 返回 `passed=False`，图路由到 finalize 并报错

### 4.2 结构化失败上下文 (Structured Replan Context)

**修改位置**：`core/graph.py` 的 Replanner 节点

**新增辅助函数**：

```python
def _build_structured_replan_context(
    state: OmniCoreState,
    replan_history: List[dict],
    authoritative_url: str,
) -> dict:
    """
    构建给 Replanner LLM 的结构化 JSON 上下文。
    替代当前的字符串拼接方式。
    """
    failed_tasks = []
    for task in state["task_queue"]:
        if task["status"] == "failed" or (
            task["status"] == "completed"
            and not task.get("critic_approved", False)
        ):
            failed_tasks.append({
                "description": task["description"],
                "worker_type": task["task_type"],
                "failure_type": task.get("failure_type", "unknown"),
                "error": _extract_error_summary(task, max_len=200),
                "url": _extract_task_url(task),
                "last_steps": _extract_last_steps(task, count=3),
            })

    return {
        "user_request": state["user_input"],
        "attempt_number": state["replan_count"],
        "is_final": state["replan_count"] >= MAX_REPLAN,
        "failed_tasks": failed_tasks,
        "history": replan_history,
        "authoritative_url": authoritative_url,
        "successful_paths": state.get("shared_memory", {}).get("successful_paths", []),
    }
```

**Replanner Prompt 改造**：把 `user_message` 从拼接文本改为 `json.dumps(context, ensure_ascii=False, indent=2)`，让 LLM 更容易解析结构化输入。

## 5. 不建议做的事

以下优化看起来有价值，但在当前阶段**不建议实施**：

| 不建议做 | 原因 |
|----------|------|
| Router 输出多个候选计划让用户选择 | 增加交互复杂度，通用场景下用户期望"一键完成" |
| 实时执行成本计量（token/时间） | 需要侵入所有 Worker，收益不明确 |
| 持久化策略知识库（跨 session 的"策略记忆"） | 当前 `successful_paths` / `failure_patterns` 已提供基础能力，额外持久化过早优化 |
| 给每个 Stage 加 timeout 强制推进 | 可能截断正在执行的浏览器操作，风险大于收益 |
