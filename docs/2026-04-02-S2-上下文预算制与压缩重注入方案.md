## 问题分析

R1 实现了两层成本控制（tool result 截断 8000 字符 + history snip 保留最近 10 条），但缺少：

1. **全局 token 预算**：不知道当前上下文总共用了多少 token，各部分占比多少
2. **无预留机制**：如果上下文已经 90% 满了，下一次 LLM 调用可能直接 OOM
3. **压缩后失忆**：snip 后丢失了当前 plan 状态、artifact 引用、工具声明，agent 需要重新"热身"
4. **无熔断**：如果 compact 操作本身反复失败（如单条消息就超窗口），系统会持续浪费 API 调用
5. **R7 session memory 注入时机固定**：只在 snip 时注入，不在 compact 后重注入

## 核心设计决策

1. **预算是建议而非强制**：超预算时触发 compact/截断，但不会拒绝 LLM 调用。这是 soft limit 而非 hard limit
2. **compact 用低成本模型**：与 R7 session memory 类似，compact 摘要调用 `complexity=0.2` 路由到最低成本模型
3. **重注入内容格式**：结构化 SystemMessage，用 `[上下文恢复]` 前缀标识，让模型知道这是压缩后恢复的上下文
4. **不改变 R1 的 snip 逻辑**：S2 的 compact 是 snip 之外的额外压缩层，两者共存。触发顺序：snip（每轮） → compact（接近窗口上限时）

## 验收标准

- 每轮 LLM 调用前能输出各 slot 的 token 使用量
- 上下文使用率超过 85% 时自动触发 compact
- Compact 连续失败 3 次后自动熔断
- Compact 后 plan 状态、artifact 引用、session memory 自动重注入
- 长会话（>30 轮）不再出现 context window exceeded 错误
