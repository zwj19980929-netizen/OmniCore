# OmniCore 通用 Agent 第二阶段落地说明

日期：2026-03-03

本文记录在第一阶段基础上继续推进的第二阶段改造，重点是让运行时更接近真正的 tool-first 执行模型。

## 已完成

1. Router 输出协议升级
   - `core/router.py` 现在会在系统提示里明确要求优先输出 `tool_name` 和 `tool_args`
   - 同时保留对旧 `task_type / params` 输出的兼容
   - Router 在解析结果后会做一次 shape 归一化，把旧格式补齐为 tool-first 结构

2. 权限规则继续下沉到工具执行层
   - `agents/file_worker.py` 在真正写文件前会依据 `requires_confirmation` 做确认
   - 这覆盖了 txt 以及结构化导出前的统一写入入口
   - `agents/system_worker.py` 在执行命令前增加了命令安全校验，并在任务执行入口再次执行确认

3. 风险边界进一步收紧
   - 系统命令会拦截常见 shell 控制符（如 `&&`、`||`、`;`、管道和重定向）
   - 文件写入即使经过规划层判断，也会在工具层再次确认

4. 并发调度开始使用工具元数据
   - `Tool Registry` 现在声明每个工具的并发上限与是否串行
   - 调度器会优先按工具元数据限制批次选择
   - 浏览器工具默认并发 2，系统工具默认串行

## 当前状态

运行时主路径已经是：

- Router 输出 tool-first 计划
- 任务规范化层补齐兼容字段
- TaskExecutor 通过 Tool Registry 做实际分发
- 工具层执行前再次做权限与边界控制

## 收口结果

第二阶段原本遗留的工程性尾巴已经清理完成：

1. `core/task_executor.py` 中原先保留的 `_execute_single_task_async_legacy` 历史快照已移除
   - 当前只保留新的 `_execute_single_task_async` 作为执行入口
   - 运行时分发路径已经完全收敛到 Tool Registry 主链路

2. `core/router.py` 的主 JSON 示例已改成 tool-first 结构
   - 示例会直接展示 `tool_name` 和 `tool_args`
   - 旧格式仍可被归一化兼容，但不再是主示例

## 如何添加新工具

当前版本接入一个新工具，建议按下面 4 步做。

1. 在 `core/tool_registry.py` 注册工具
   - 新增一个 `RegisteredTool(...)`
   - 至少补齐这些字段：
   - `spec.name`
   - `spec.task_type`
   - `spec.description`
   - `spec.risk_level`
   - `adapter_name`
   - `max_parallelism`
   - `serialized`

2. 让规划层知道这个工具
   - 在 `core/router.py` 的 Router 提示词里补充这个工具的用途
   - 如果希望 LLM 优先产出这个工具，还要补充对应的 `tool_name` 示例
   - 如果 Router 仍输出旧格式，运行时会做兼容归一化，但最好直接输出 `tool_name / tool_args`

3. 在执行层接入对应适配器
   - 实际执行入口是 `core/task_executor.py` 的 `_execute_registered_tool_async()`
   - 如果复用现有能力，可以直接使用已有 `adapter_name`
   - 如果是全新能力，通常需要：
   - 新建一个执行模块（例如新的 worker）
   - 在 `WorkerPool` 中补实例管理
   - 在 `_execute_registered_tool_async()` 中增加 `adapter_name` 分发分支

4. 在 `core/policy_engine.py` 增加确定性权限规则
   - 有写入、本地副作用、外部提交、系统操作的工具，必须在这里补规则
   - 不要只依赖 LLM 的高风险判断

### 最小接入清单

一个新工具最少需要同时经过这 4 个文件：

- `core/tool_registry.py`
- `core/router.py`
- `core/task_executor.py`
- `core/policy_engine.py`

如果是全新执行器，通常还会新增或修改：

- `agents/...`
- `WorkerPool`

### 示例思路

如果要加一个 `api.call` 工具，推荐做法是：

1. 在 `core/tool_registry.py` 注册 `api.call`
2. 新增一个专门的执行模块，例如 `agents/api_worker.py`
3. 在 `WorkerPool` 中补 `api_worker`
4. 在 `_execute_registered_tool_async()` 中增加 `adapter_name == "api_worker"` 的执行分支
5. 在 `core/policy_engine.py` 里规定：
   - `GET` 默认低风险
   - `POST / PUT / DELETE` 默认需要确认

这样接入后，新工具就能同时进入：

- 规划
- 调度
- 执行
- 权限治理
