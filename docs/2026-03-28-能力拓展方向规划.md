# OmniCore 能力拓展方向规划

> 编写时间：2026-03-28
> 基于当前架构（7 层 + 5 执行 Agent + LangGraph DAG 编排）的下一步拓展方向。

---

## 当前能力边界

| 维度 | 已具备 | 尚未覆盖 |
|------|--------|----------|
| 任务规划 | LLM 路由 + 自动分解 + 3 次 replan | 跨 Agent 协商、动态子图 |
| 执行能力 | Browser / Web / File / System / Terminal 五种 Agent | 外部工具生态（MCP）、多模态输入输出 |
| 记忆 | ChromaDB 向量检索 + 会话/任务/制品持久化 | 主动经验沉淀（Skill Library）、深度 RAG |
| 交互 | CLI + Streamlit Web UI | 语音、移动端、IM Bot |
| 自动化 | 定时任务 + 目录监控 + 后台 Worker | 事件驱动（网页变化监控、消息监听） |
| 成本控制 | 多模型支持、models.yaml 配置 | 基于复杂度的智能路由 |

---

## 拓展方向

### P0-1: MCP (Model Context Protocol) 工具生态接入

**价值**：投入产出比最高。一行配置接入社区数百个现成工具（数据库查询、Slack/飞书/邮件、日历、GitHub API 等），避免为每个外部服务单独写 Agent。

**当前基础**：
- `core/tool_registry.py` + `core/tool_protocol.py` 已有插件化框架
- `config/agents.yaml` 支持声明式工具注册
- Router 已具备向不同 tool 分发任务的能力

**实现路径**：
1. 在 `core/tool_adapters.py` 新增 `MCPToolAdapter`，通过 stdio/SSE 与 MCP Server 通信
2. MCP Server 工具列表在启动时自动发现，动态注入 Router prompt（复用 `{{AGENT_CAPABILITIES}}` 机制）
3. MCP 工具的输入/输出统一映射为 `TaskItem` / `ArtifactItem`
4. `config/mcp_servers.yaml` 声明式配置 MCP Server 列表

**预估工作量**：2-3 天核心实现 + 1 天联调

### P0-2: 自主学习与经验复用（Skill Library）

**价值**：让系统越用越聪明，核心差异化能力。成功完成的任务流自动沉淀为可复用 Skill，下次遇到类似任务直接执行。

**当前基础**：
- ChromaDB 记忆系统已就绪
- `finalize` 阶段已有结果汇总逻辑
- `ArtifactStore` 可存储 Skill 产物

**实现路径**：
1. 定义 `Skill` 数据结构：名称、触发条件（语义描述）、参数模板、任务流模板
2. 在 `finalize` 阶段增加 Skill 提炼逻辑：LLM 判断当前任务是否值得沉淀 → 抽象为 Skill → 存入 ChromaDB 专用 collection
3. Router 阶段先检索 Skill Library：命中（相似度 > 阈值）→ 参数适配 → 直接执行；不命中 → 走完整规划
4. Skill 支持版本管理和失败自动降级（连续失败 N 次后标记为 deprecated）

**预估工作量**：3-4 天

📄 **详细方案**：[2026-03-28-P0-2-Skill-Library经验复用方案.md](2026-03-28-P0-2-Skill-Library经验复用方案.md)

### P1-1: 远程访问 — IM Bot 接入

**价值**：个人助手必须随时可达。手机上下发任务，后台执行，完成后推送通知。

**当前基础**：
- `submit_job` + `RuntimeStateStore` 已支持异步任务提交和状态查询
- 后台 Worker 已能独立消费队列
- 通知机制（`workflow_automation_store`）已有基础

**实现路径**：
1. 在 Layer 1（交互层）新增 `interactions/telegram_bot.py`（或微信/飞书 Bot）
2. Bot 接收消息 → 调用 `submit_job` → 返回 job_id
3. Worker 完成后通过 Bot API 推送结果
4. 支持 `/status`、`/cancel`、`/approve` 等交互命令
5. 高风险操作的审批通过 Bot 内联按钮完成

**预估工作量**：2-3 天（Telegram 最快，微信需额外适配）

📄 **详细方案**：[2026-03-28-P1-1-IM-Bot远程访问方案.md](2026-03-28-P1-1-IM-Bot远程访问方案.md)

### P1-2: 知识库 & RAG 深度集成

**价值**：已有 ChromaDB 基础设施，补全最后一公里。将浏览/下载/执行产出的信息自动索引，任务执行时自动检索相关知识。

**当前基础**：
- `memory/scoped_chroma_store.py` 已有向量存储和检索
- `memory/manager.py` 已有实体提取
- `ArtifactStore` 记录所有产物

**实现路径**：
1. Artifact 写入时自动触发文本提取 + 向量索引（PDF/HTML/文本文件）
2. 浏览器 Agent 完成后，将页面关键内容自动索引
3. Router 阶段增加 RAG 检索步骤：检索 top-K 相关知识 → 注入任务上下文
4. 提供 `/learn <url_or_file>` 命令让用户主动喂文档
5. 知识条目支持过期标记和手动管理

**预估工作量**：3-4 天

### P2-1: 多模态输入/输出

**价值**：提升交互体验。用户贴图问"帮我找到这个产品"、上传 PDF 提取信息后执行后续任务、语音下发指令。

**当前基础**：
- Vision Model 已接入（`gpt-4o`）
- Browser Agent 已有截图分析能力
- LiteLLM 支持多模态 API

**实现路径**：
1. **图片/文档输入**：CLI 和 Streamlit 支持文件上传 → 自动调用 Vision Model 提取意图 → 注入 Router
2. **语音输入**：接入 Whisper API → 转文本 → 走正常流程
3. **语音输出**：结果文本 → TTS API → 音频播放/推送
4. **文档理解**：PDF/Word/Excel → 文本提取 → LLM 摘要 → 作为任务上下文

**预估工作量**：3-5 天（图片/文档优先，语音可后置）

### P2-2: 成本感知与智能路由

**价值**：长期运行显著省钱。简单任务用便宜模型，复杂推理用强模型。

**当前基础**：
- `config/models.yaml` 已有多模型 + cost_preference 配置
- `core/llm.py` 已有模型切换能力
- Router 已有意图分类

**实现路径**：
1. Router 增加 complexity scoring（基于任务描述长度、关键词、历史相似任务的 token 消耗）
2. `core/llm.py` 基于 score 动态路由：低复杂度 → DeepSeek/Haiku，高复杂度 → Claude/GPT-4o
3. 记录每次调用的模型 + token + 成本到 `RuntimeMetricsStore`
4. 提供 `/cost` 命令查看历史成本统计
5. 支持用户设置月度预算上限

**预估工作量**：2-3 天

### P3-1: 实时信息流 & 事件驱动

**价值**：从被动响应升级为主动监控。网页价格变动、新文章发布、邮件到达时自动触发任务。

**当前基础**：
- `workflow_automation_store.py` 已有目录监控和定时任务
- 后台 Worker 已能消费事件队列
- 通知基础设施已就绪

**实现路径**：
1. 抽象通用 `EventSource` 接口（目录监控、网页变化、邮件、Webhook 皆为实现）
2. `WebChangeMonitor`：定期 Browser/Web Agent 访问指定 URL → diff 内容 → 变化超阈值触发任务
3. `EmailMonitor`：IMAP 轮询 → 新邮件触发任务
4. `WebhookReceiver`：HTTP endpoint 接收外部事件
5. 事件 → 自动创建 Job → Worker 消费 → 结果通知

**预估工作量**：3-5 天（网页监控优先）

📄 **详细方案**：[2026-03-28-P3-1-事件驱动信息流方案.md](2026-03-28-P3-1-事件驱动信息流方案.md)

### P3-2: 多 Agent 协作（Agent-to-Agent）

**价值**：支持复杂的跨 Agent 联动场景。架构复杂度最高，等场景驱动。

**当前基础**：
- `message_bus.py` 已有类型化消息传递
- `graph.py` 已有 DAG 编排能力
- `StageRegistry` 支持条件跳转

**实现路径**：
1. 在 `message_bus.py` 增加 Agent-to-Agent 消息类型（`MSG_AGENT_REQUEST` / `MSG_AGENT_RESPONSE`）
2. `graph.py` 支持动态子图：执行阶段可按需展开为子 DAG
3. 典型场景实现：Browser 下载 → Terminal 解压处理 → File Worker 整理
4. Planner Agent 与 Executor Agent 分离：Planner 生成高层计划，Executor 独立执行并汇报，Planner 动态调整
5. 引入 Agent 间的协商协议（请求-响应-确认三步握手）

**预估工作量**：5-7 天

📄 **详细方案**：[2026-03-28-P3-2-多Agent协作方案.md](2026-03-28-P3-2-多Agent协作方案.md)

## 优先级总览

| 优先级 | 方向 | 核心理由 |
|--------|------|----------|
| **P0** | MCP 工具生态接入 | 投入产出比最高，立刻获得数百个工具能力 |
| **P0** | Skill Library 经验复用 | 让系统越用越聪明，核心差异化 |
| **P1** | IM Bot 远程访问 | 个人助手必须随时可达 |
| **P1** | 知识库 RAG 深度集成 | 已有基础设施，补全最后一公里 |
| **P2** | 多模态输入/输出 | ✅ 已实施（2026-03-28） |
| **P2** | 成本感知智能路由 | 长期运行省钱，渐进优化 |
| **P3** | 事件驱动信息流 | 已有基础，按需扩展 |
| **P3** | 多 Agent 协作 | 架构复杂度高，等场景驱动 |
