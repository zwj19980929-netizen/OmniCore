# 2026-03-17 Web Agent架构落地方案

## 1. 背景

当前 `BrowserAgent` 已经具备以下能力：

- 语义快照和交互元素提取，入口在 `agents/browser_agent.py`
- 页面评估 LLM、动作决策 LLM、vision fallback
- 动作执行、结果校验、循环检测、人工审批

但主链路仍然缺一层显式的“页面诊断”。结果就是 Agent 往往能看到输入框，却不一定能先判断：

- 当前是否处于正确的交互模式
- 给定凭证类型是否和当前模式匹配
- 当前报错是页面阻塞、校验失败，还是单纯未加载完成

典型表现是登录页默认停在“邮箱登录”，Agent 看到输入框后直接输入 `admin`，却没有先得出“`admin` 更像用户名，不像邮箱，因此应该先切到用户名登录”。

这说明问题不在“能不能识别元素”，而在“有没有先诊断页面状态，再规划子目标”。

## 2. 目标

把当前链路升级为真正的：

`observe -> diagnose -> generate candidates -> choose -> act -> verify -> remember`

原则如下：

- 不依赖站点级硬编码
- 不让规则树主导下一步动作
- 保留少量通用执行规则，只负责安全、校验和 fallback
- 优先让模型基于结构化页面状态做决策

## 3. 目标架构

### 3.1 Observation Layer

输入来自当前已有的语义快照、元素提取、URL、标题、最近步骤。

建议新增统一结构 `PageObservation`，至少包含：

```json
{
  "url": "",
  "title": "",
  "page_kind_hint": "auth|list|detail|serp|unknown",
  "interactive_elements": [],
  "visible_tabs": [],
  "active_tab": "",
  "form_groups": [],
  "inline_validation_errors": [],
  "global_alerts": [],
  "loading_signals": [],
  "recent_state_changes": []
}
```

重点不是“抓更多 HTML”，而是把模型真正需要的页面状态显式提取出来。

对认证页，最重要的 observation 信号是：

- 当前激活的登录模式
- 可切换的其他模式
- 输入框对应的标签、placeholder、type、autocomplete
- 即时校验错误
- 主提交控件

### 3.2 Diagnosis Layer

新增 `PageDiagnosis`。这是当前系统最缺的一层。

推荐输出：

```json
{
  "page_kind": "auth",
  "stage": "interactive",
  "active_mode": "email_password",
  "available_modes": ["email_password", "username_password", "phone_password"],
  "credential_kind": "username",
  "mode_mismatch": true,
  "blocking_issue": "inline_validation",
  "blocking_evidence": ["请输入正确格式的邮箱"],
  "next_subgoal": "switch_auth_mode"
}
```

这样，模型不需要靠“猜”。它会先得出页面诊断，再决定下一步是切换模式、填表、提交，还是等待。

### 3.3 Candidate Generation Layer

不要让 LLM 直接在原始元素列表里随意输出动作。先由代码根据 observation 和 diagnosis 生成一组 typed candidates，例如：

- `switch_mode(username_password)`
- `fill_field(username, admin)`
- `fill_field(password, AiAgent#2025)`
- `submit(primary_form)`
- `wait_for_hydration`
- `recover_from_validation`

这一层仍然是通用逻辑，但它只负责生成“可执行候选”，不负责替模型做业务判断。

### 3.4 Policy Layer

LLM 在以下输入上做选择：

- `TaskIntent`
- `PageObservation`
- `PageDiagnosis`
- `candidate_actions`
- `recent_steps`
- `last_action_result`

这里的 prompt 应从“自由规划动作”切换为“在候选动作中选一个，并说明为什么”。

这样可以显著降低：

- 重复动作
- 错误点击
- 在活跃表单上过早 `done`
- 因页面元素太多导致的决策漂移

### 3.5 Verification Layer

动作执行后必须验证状态迁移，而不是只看调用是否成功。

推荐校验项：

- URL 是否变化
- 激活 tab 是否变化
- 表单是否离开当前模式
- 错误提示是否消失、变化或新增
- 主按钮是否从 disabled 变为 enabled 或反过来
- 是否出现登录后内容、重定向或新的受保护页面元素

对于 auth/form 场景，“动作成功”应定义为“页面状态发生了可解释变化”，而不是“点击 API 没报错”。

### 3.6 Workflow Memory

把成功轨迹按“交互模式”存储，不按站点存储。

例如：

- 多模式登录页先切 tab 再填表
- 表单提交后等待局部错误提示
- 搜索结果页先点最相关卡片再抽取

推荐记忆粒度：

- 页面诊断模式
- 子目标序列
- 动作序列
- 成功/失败证据

这样能帮助相似页面复用套路，但不会退化成站点硬编码。

## 4. 对当前代码的映射

### 4.1 BrowserAgent

当前可复用部分：

- `TaskIntent`
- `PageElement`
- `_get_semantic_snapshot`
- `_extract_interactive_elements`
- `_assess_page_with_llm`
- `_decide_action_with_llm`
- `_plan_next_action`
- `run`

建议新增：

- `PageObservation` dataclass
- `PageDiagnosis` dataclass
- `_build_page_observation(...)`
- `_diagnose_page_with_llm(...)`
- `_generate_candidate_actions(...)`
- `_verify_state_transition(...)`

建议把 `run()` 主循环调整为：

1. 观察页面  
2. 生成 `PageObservation`  
3. 生成 `PageDiagnosis`  
4. 生成 `candidate_actions`  
5. LLM 选择下一步  
6. 执行动作  
7. 校验状态迁移  
8. 写入 workflow memory

### 4.2 Prompt

建议新增或改造两个 prompt：

- `browser_page_assessment.txt`
  目标：输出诊断 JSON，而不只是 page type
- `browser_action_decision.txt`
  目标：从候选动作中选择一个最合理动作

Prompt 的重点应从“看页面像什么”变成：

- 当前页面处于什么交互模式
- 当前模式是否与任务字段匹配
- 当前阻塞点是什么
- 下一子目标是什么

### 4.3 WebWorker

`agents/web_worker.py` 里的天气判断、站点补丁、`common_selectors` 等启发式，本质上是抽取 fallback，不应该继续扩展为主策略。

建议定位为：

- 只服务于读页、列表页、搜索页抽取
- 只在 LLM 诊断无法给出稳定交互路径时兜底
- 后续逐步收敛为 profile 或 fallback library

不要再把交互式问题修到 `WebWorker` 里。

## 5. 实施顺序

### 阶段一：补诊断层

先落 `PageObservation` 和 `PageDiagnosis`，不改现有 executor 和 graph。

交付标准：

- auth/form 页面能输出 `active_mode`、`available_modes`、`credential_kind`
- 能区分 `loading`、`validation_failed`、`mode_mismatch`、`ready_to_submit`

### 阶段二：补候选动作层

在 `BrowserAgent` 内部新增 typed candidates，限制 LLM 的选择空间。

交付标准：

- 对登录、多 tab、带局部校验的表单，动作质量明显提升
- 重复点击和错误提交次数下降

### 阶段三：替换主规划链

把 `_plan_next_action` 调整为：

`diagnosis -> candidates -> policy choice -> sanitize`

本地规则只做：

- 动作格式校验
- 缺失 target 的拦截
- 重复失败动作抑制
- 安全约束

### 阶段四：沉淀 workflow memory

对成功任务保存：

- 页面诊断摘要
- 子目标序列
- 动作序列
- 最终成功证据

## 6. 成功标准

如果架构升级有效，应该出现这些结果：

- 登录页不再“看到输入框就直接填”
- 模型能先判断当前模式是否匹配任务字段
- 页面出现即时校验错误后，能主动重规划
- `WAIT` 不再只是拖时间，而是和页面状态相关
- `repeated action loop detected` 的触发率下降
- 相似页面的成功率随着 workflow memory 累积而提升
