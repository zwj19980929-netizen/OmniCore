## 二、P0: Vision Fallback 预算与限流

### 2.1 现状分析

**问题**: `_decide_action_with_vision`（line 1646）在 DOM 层无法决策时被调用，将截图发给视觉大模型。当前无任何限流机制：

- 无单次 run 调用次数上限
- 无调用间隔冷却
- 无 token 预算控制
- 仅依赖全局 LLM 超时（60s），无视觉专用超时

**后果**: Agent 在复杂页面迷失时，可能在死循环中不断截图发送，消耗大量 token 和时间。

**视觉调用入口**（3 处）:
1. line 2646: DOM 决策返回 WAIT/None 时的主回退路径
2. line 2816: 动作执行失败且本地恢复也失败时
3. line 2622: `_vision_check_page_relevance` 页面相关性检查

### 2.2 Step 1: 新增 VisionBudget 追踪器

在 `agents/browser_agent.py` 中添加：

```python
@dataclass
class VisionBudget:
    max_calls_per_run: int = 5          # 单次 run 最多 5 次视觉调用
    max_total_tokens: int = 20000       # token 上限
    cooldown_seconds: float = 3.0       # 调用间隔冷却
    calls_made: int = 0
    tokens_used: int = 0
    last_call_time: float = 0.0

    def can_call(self) -> bool:
        if self.calls_made >= self.max_calls_per_run:
            return False
        if time.time() - self.last_call_time < self.cooldown_seconds:
            return False
        return True

    def record_call(self, tokens: int = 0):
        self.calls_made += 1
        self.tokens_used += tokens
        self.last_call_time = time.time()

    def remaining(self) -> int:
        return max(0, self.max_calls_per_run - self.calls_made)
```

在 `BrowserAgent.__init__` 中初始化 `self._vision_budget = VisionBudget(...)`。

## 三、P0: 像素级视觉校验误报优化

### 3.1 现状分析

**当前实现**（`utils/image_diff.py`）:
- `compute_pixel_diff`（line 15）: 灰度转换 → 4x 降采样 → 逐像素比较（阈值 20）
- `screenshots_differ`（line 56）: 差异超过 2% 即判定"页面变化"
- `VisualProgressTracker`（line 97）: 感知哈希检测卡住

**误报场景**:
| 场景 | 像素变化 | 实际含义 |
|------|---------|---------|
| 光标闪烁 | 0.1%-0.5% | 无意义变化 |
| 轮播图切换 | 5%-15% | 自动播放，非用户操作导致 |
| 动态广告刷新 | 2%-8% | 无关区域变化 |
| GIF 动画 | 1%-3% | 持续变化 |
| 真实页面跳转 | 30%+ | 有意义变化 |
| 弹窗出现 | 10%-25% | 有意义变化 |

当前 2% 阈值太敏感，会将光标闪烁和广告刷新误判为"页面有变化"。

### 3.5 Step 4: 分层校验策略

更新 `screenshots_differ` 为分层判断：

```python
def screenshots_differ(img_a, img_b,
                       threshold=None,
                       use_block_diff=True) -> bool:
    """
    分层判断页面是否发生有意义变化：
    1. 像素级差异 > threshold → True（大范围变化，如页面跳转）
    2. 块级差异 > block_threshold → True（结构变化，如弹窗）
    3. 否则 → False（噪声，如光标/广告）
    """
    threshold = threshold or settings.VISION_PIXEL_DIFF_THRESHOLD

    pixel_diff = compute_pixel_diff_roi(img_a, img_b)
    if pixel_diff > threshold:
        return True

    if use_block_diff:
        block_diff = compute_block_diff(img_a, img_b)
        if block_diff > 0.08:  # 8% 的块发生变化
            return True

    return False
```

### 3.6 Step 5: 动作验证专用函数

为 `_verify_action_effect`（browser_agent.py line 1879）提供更严格的比较：

    """
    用于动作效果验证的严格比较。
    排除输入框光标区域，使用块级比较。
    """

## 四、P1: Shadow DOM / iframe 穿透增强

### 4.1 现状分析

| 能力 | 状态 | 说明 |
|------|------|------|
| Shadow DOM | **完全不支持** | 所有 JS 使用 `document.querySelectorAll`，无法穿透 Shadow Root |
| iframe（单层） | **基础支持** | `switch_to_iframe`（browser_toolkit.py line 904）支持单层切换 |
| iframe（嵌套） | **不支持** | 无嵌套 iframe 切换能力 |
| iframe 内容提取 | **不支持** | 元素提取不扫描 iframe 内部 |

**依赖**: 本节的 JS 修改依赖代码重构方案中 "JS 分离"（第四节）先完成。

### 4.2 Step 1: Shadow DOM 穿透 — JS 工具函数

在 `utils/perception_scripts.py` 的 `SCRIPT_COMMON_UTILS` 中添加：

```javascript
/**
 * 递归穿透 Shadow DOM 的 querySelectorAll。
 * 遍历所有元素的 shadowRoot，合并结果。
 */
const querySelectorAllDeep = (selector, root = document) => {
    const results = [...root.querySelectorAll(selector)];
    const allElements = root.querySelectorAll('*');
    for (const el of allElements) {
        if (el.shadowRoot) {
            results.push(...querySelectorAllDeep(selector, el.shadowRoot));
        }
    }
    return results;
};

/**
 * 递归穿透 Shadow DOM 的 querySelector（返回第一个匹配）。
 */
const querySelectorDeep = (selector, root = document) => {
    const result = root.querySelector(selector);
    if (result) return result;
    const allElements = root.querySelectorAll('*');
    for (const el of allElements) {
        if (el.shadowRoot) {
            const found = querySelectorDeep(selector, el.shadowRoot);
            if (found) return found;
        }
    }
    return null;
};
```

### 4.3 Step 2: 更新元素提取脚本

将 `SCRIPT_EXTRACT_INTERACTIVE_ELEMENTS` 和 `SCRIPT_FALLBACK_SEMANTIC_SNAPSHOT` 中的 `document.querySelectorAll` 替换为 `querySelectorAllDeep`，使元素提取能穿透 Shadow DOM。

**关键替换点**:
- 交互元素提取: `querySelectorAll('a, button, input, select, textarea, [role="button"], ...')`
- 可见性检查: 需考虑 Shadow Root 中的元素可能有不同的样式作用域
- 选择器生成: Shadow DOM 内的元素需用组合选择器（如 `host-element >>> inner-selector`）

### 4.4 Step 3: iframe 内容提取

在 `BrowserPerceptionLayer`（`agents/browser_perception.py`）中添加：

```python
async def _extract_iframe_elements(self) -> List[PageElement]:
    """
    遍历所有可见 iframe，提取其中的交互元素。
    元素标记 region="iframe"，selector 添加 iframe 定位前缀。
    """
    iframe_elements = []
    frames = self.page.frames  # Playwright 提供的所有 frame

    for frame in frames:
        if frame == self.page.main_frame:
            continue
        try:
            # 在每个 iframe 中执行元素提取 JS
            elements = await frame.evaluate(SCRIPT_EXTRACT_INTERACTIVE_ELEMENTS)
            for el in elements:
                el["region"] = "iframe"
                el["frame_url"] = frame.url
                iframe_elements.append(el)
        except Exception:
            continue  # iframe 可能跨域不可访问

    return iframe_elements
```

在 `observe()` 方法中合并 iframe 元素到主元素列表。

async def switch_to_iframe(self, selector: str = "") -> ToolkitResult:
    """
    支持嵌套 iframe 切换。
    selector 格式: "outer_selector >>> inner_selector" 表示嵌套层级。
    """
    if ">>>" in selector:
        parts = [s.strip() for s in selector.split(">>>")]
        current_frame = self._page.main_frame
        for part in parts:
            handle = await current_frame.locator(part).first.element_handle()
            current_frame = await handle.content_frame()
            if not current_frame:
                return ToolkitResult(success=False, error=f"iframe not found: {part}")
        self._current_frame = current_frame
        self._in_iframe = True
        return ToolkitResult(success=True)

async def list_iframes(self) -> ToolkitResult:
    """列出当前页面所有可见 iframe 及其属性"""
    frames_info = []
    for frame in self._page.frames:
