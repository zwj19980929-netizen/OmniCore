## 问题分析

当前 `agent_registry.py` 支持按 `agents.yaml` 配置注册不同类型 agent，`router.py` 基于 intent 选择 agent 类型，但：

1. **单层调度**：router → planner → executor 是固定流水线，不支持一个 agent 派发子 agent
2. **无并行子任务**：复杂任务（如"调研 A 和 B 并对比"）只能串行执行
3. **无结果汇总**：没有 coordinator 角色来汇总多个子任务的结果
4. **无 agent 间通信**：agents 之间不能传递消息或共享中间结果
5. **无递归控制**：缺少拓扑约束防止 agent 无限 spawn

## Claude Code 参考实现

```
// 三级模式：
// 1. Subagent: AgentTool.runAgent(), 继承父 prompt bytes (cache stability)
// 2. Coordinator: system prompt 替换为编排者，派发 worker，<task-notification> 返回
// 3. Swarm: 持久化 team file, shared task list, mailbox, permission bridge
//
// 拓扑约束：teammate 不能 spawn teammate, in-process teammate 不能 spawn background
// 权限桥接：leaderPermissionBridge.ts, 子 agent 权限请求路由到主 agent
// 上下文继承：fork child 直接拿 parent 的 rendered prompt bytes
```

## 核心设计决策

1. **先做 Subagent + Coordinator，不做 Swarm**：Swarm 模式（持久化团队、mailbox）复杂度高、OmniCore 场景暂不需要。Coordinator 已能覆盖"调研并对比"类复杂任务
2. **子 agent 跑独立 graph**：不在主 graph 内嵌套子图，而是独立创建 state + 跑完整 graph 循环。简单但隔离性好
3. **上下文继承非全量**：只继承 system prompt + session memory + plan 概要，不继承完整 history。避免子 agent 上下文膨胀
4. **Coordinator 是 graph 节点，不是 agent 类型**：Coordinator 是主 graph 中的一个路由分支，不是独立 agent。这样复用现有 finalize / policy 逻辑
5. **不引入进程隔离**：所有子 agent 在同一进程内异步执行（asyncio），不做进程 spawn。OmniCore 是单用户系统，进程隔离收益低

## 验收标准

- 复杂任务（比较/调研/并行信号）自动路由到 coordinator 模式 ✅
- Coordinator 能并行派发 2-3 个子 agent 并汇总结果 ✅
- 子 agent 受 max_turns / timeout / depth 限制 ✅
- 子 agent 的审批请求能路由到主 agent 展示给用户（消息类型已定义，实际桥接待 S6）
- 简单任务不受影响（不走 coordinator）✅
- 全量测试通过 ✅

### 与原方案的偏差
1. **路由判定方式**：原方案计划基于 complexity_scorer（P2-2）的 complexity > 0.7 阈值，实际改为 LLM 驱动判定 — router LLM 调用新增 `needs_coordinator` 输出字段，复用同一次 LLM 调用零额外成本。比硬编码正则更准确，也不依赖 P2-2
2. **Agent 间通信**：原方案计划新增 agent_channel 概念，实际复用 MessageBus 现有的 source/target 过滤机制（更简洁，无需扩展 bus API）
3. **权限桥接**：消息类型已定义（MSG_SUBAGENT_APPROVAL_*），但实际审批流打通需要 S6 安全分层的 pipeline 支持，标记为待集成
4. **Coordinator 定位**：按原方案设计，是 graph 节点而非独立 agent。coordinator → finalize 一步到位，不经过 validator/critic（coordinator 内部已完成合成质量把控）

### 关键设计决策
- SubagentRunner 使用 `asyncio.to_thread(graph.invoke)` 桥接同步 graph 到异步执行
- `_run_subagents_sync` 处理了"已在事件循环中"的边界情况（用 ThreadPoolExecutor 隔离）
- 所有子 agent 共享同一进程，通过 asyncio.Semaphore 控制并发度
- 失败策略 fail_fast 通过 asyncio.Event 实现任务取消通知
