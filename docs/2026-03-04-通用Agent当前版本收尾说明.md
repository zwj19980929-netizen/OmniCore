# 通用Agent当前版本收尾说明

## 1. 文档目的

这份文档用于对当前版本做一次明确收尾，回答 3 个问题：

1. 这一版到底改了什么
2. 为什么要这样改，业务目标是什么
3. 现在项目已经推进到什么边界，后面应该怎么用、怎么收口

这不是新的路线图，而是对当前代码状态的阶段性结论。

---

## 2. 当前版本的定位

当前版本的 OmniCore，不再只是一个“能跑几个工具的任务执行器”，而是一个面向个人场景的、可长期运行的通用 Agent Runtime 雏形成品。

它的目标不是做一个对外平台，也不是做一个炫技型自治系统，而是尽可能贴近“个人数字员工”这个使用场景：

- 你给出任务，它能拆解、执行、校验、交付
- 你不盯着它的时候，它也能后台运行、定时触发、等待审批、等待事件
- 它能记住工作上下文，复用已有成果和成功路径
- 它开始具备“围绕一个持续事项推进工作”的能力，而不只是执行一次性命令

一句话概括：当前版本的重点，是把系统做成“自己长期敢用”的个人通用智能体底座。

---

## 3. 这一版改了什么

### 3.1 工具层与调度层：从固定 Worker 走向 Tool-First

这一版把系统的核心执行逻辑，从“按固定 Worker 类型分发”，推进到了“按工具协议和工具注册中心调度”。

主要变化：

- 建立了 `Tool Protocol`
- 引入了 `Tool Registry`
- 执行器优先按 `tool_name` 调度，而不是只按 `task_type`
- Router 侧开始以 `tool_name / tool_args` 为主要输出结构
- 新工具接入从“改一圈中央分发逻辑”收敛成“注册工具 + 接 adapter + 补策略”

这一步的意义是：系统不再被最初那几类 Worker 绑死，后续扩能力的成本明显下降。

### 3.2 插件系统：从内建工具扩展到可注入工具

这一版把插件这条线做到了“当前阶段闭环”。

已经具备的能力：

- adapter 自注册
- 通过模块名加载插件
- 通过目录扫描发现插件
- 插件可自带 `ToolSpec`
- 插件会自动注入 `ToolRegistry`
- Router 的可用工具目录会从注册中心动态生成，所以插件工具会自动进入 AI 可见范围
- 插件支持基础 `PLUGIN_MANIFEST`
- 支持启用、禁用、卸载
- UI 中可查看插件状态和管理插件

这一步的目的不是做一个插件平台，而是让你的个人 Agent 在后续增加能力时，不至于越改越散、越改越难维护。

### 3.3 性能与可观测性：浏览器池、LLM 缓存、运行时指标

这一版完成了两项高收益底层优化，并把它们做成了可观察、可调优的形态。

包括：

- 引入 `Browser Runtime Pool`
  - 复用浏览器进程
  - 隔离 `BrowserContext`
  - 补了容量控制、获取超时、熔断/降级

- 引入 `LLM Cache`
  - 覆盖 URL 分析和页面结构分析
  - 支持 TTL、LRU、单飞去重
  - 增加命名空间配额和主动失效

- 增加运行时指标
  - 缓存命中率
  - 浏览器复用率
  - 任务级指标历史
  - 推荐配置导出
  - UI 调试面板和独立指标页

这一步的本质，是让系统在你长期使用时更稳、更省、更容易发现真实瓶颈。

### 3.4 长期运行能力：Session / Job / Artifact / Queue / Worker / Checkpoint

这一版把运行时从“一次执行一个函数”推进成了真正的长期运行底座。

主要能力：

- `Session / Job / Artifact` 持久化
- 轻量任务队列
- 后台 worker（线程模式 + 进程模式）
- 计划任务（`once / interval / daily`）
- checkpoint 持久化
- 从 checkpoint 恢复
- 失败任务重跑 / 恢复入口
- 通知机制

这部分改造的核心目的，是让 Agent 具备“持续挂着替你做事”的基础，而不是你每次都盯着一次次发命令。

### 3.5 工作闭环：Goal / Project / Todo / Artifact Store / 经验复用

这一版已经不只是在“执行任务”，而是开始围绕“推进工作”来建模。

新增的工作层能力：

- `Goal / Project / Todo` 工作上下文
- 独立 `Artifact Store`
- 工作经验沉淀
- 成功工具路径复用
- Workbench 页面
- 交付结果开始关联工作上下文和未完成事项

这一步的意义是：系统开始能承接持续事项，而不是把每个 Job 都当成互相无关的一次性操作。

### 3.6 真实工作接入与协作：审批态、事件态、API、模板、目录监听

这是当前版本最后一个主功能阶段，也是本轮收尾前最关键的一步。

已经完成：

- `api.call` 工具接入
- 变更型 API 操作进入审批态
- `waiting_for_approval / waiting_for_event / blocked` 三种等待态进入主链路
- 审批后可继续执行，而不是重跑整条链
- 工作模板
- 目录监听
- 基于目录事件自动创建任务
- `Daily Dashboard`

这一步的目标，是让系统真正进入你的工作流，而不只是活在自己的内部系统里。

### 3.7 稳定收口：统一状态、修丢任务风险、统一等待态展示

在功能阶段完成后，这一版又做了一轮“稳定收口”，避免继续无休止加能力。

这轮收口主要做了：

- 新增统一状态常量层，收敛等待态和运行态字符串
- 修复目录监听事件在“模板被删除”时的静默丢任务问题
  - 现在会自动降级为直接使用文件事件 prompt 入队
  - 同时写入自动化通知
- UI 中等待态不再被错误显示成失败态

这一步的目的很直接：减少真实使用中的误判、误丢、误操作。

## 4. 为什么要这么改

### 4.1 核心目标不是“功能越多越好”，而是“你自己长期敢用”

如果只追求功能数量，这个项目很容易变成一堆看起来很强、实际难维护的 Agent 实验功能集合。

当前版本的设计选择，核心不是“做大而全”，而是围绕下面几个判断：

- 个人自用场景里，稳定比炫技重要
- 长期运行能力比临时演示能力重要
- 工作闭环比工具数量更重要
- 经验复用比每次都重新规划更重要
- 可维护性比过度平台化更重要

所以很多改动看起来是在“补基础设施”，本质上是在降低你以后长期使用时的维护成本和认知负担。

### 4.2 为什么先做运行时，再做工作闭环，再做真实接入

这个顺序是刻意设计的，不是随机推进。

原因是：

1. 如果没有长期运行能力，Agent 就不可能像员工一样“持续替你干活”
2. 如果没有工作上下文和产物沉淀，它就只能做孤立任务，无法持续推进事情
3. 如果没有真实工作接入，它仍然只是一个内部系统里自转的智能体

因此本轮推进的逻辑是：

- 先让它能长期活着
- 再让它能围绕事情工作
- 最后再让它真正接入你的工作流

这个顺序，符合“个人数字员工”这个目标，而不是“研究演示 Agent”这个目标。

### 4.3 为什么在插件做到可用后就不继续深挖“平台化”

插件系统现在已经够用了，继续深挖下去当然还能做很多事，但那些事对你个人使用的边际收益已经不高。

比如：

- 更复杂的热重载
- 更重的插件平台治理
- 更复杂的依赖系统

这些并不是不能做，而是在当前阶段不值得优先做。

所以后面的决策是刻意收敛的：插件做到“可接入、可治理、可禁用”就停，把精力转回真实使用价值更高的地方。

---

## 5. 这一版是为了解决什么问题

从业务目标看，这一版主要解决了 5 类问题：

### 5.1 解决“只能手动盯着跑”的问题

通过队列、后台 worker、计划任务、通知和 checkpoint，系统开始能持续运行，而不是只适合你手动发起一次性任务。

### 5.2 解决“每次都像第一次做”的问题

通过工作上下文、Artifact Store、成功路径复用和模板，系统开始积累经验，不再每次都从零摸索。

### 5.3 解决“工具越来越多后会失控”的问题

通过 Tool Registry、adapter、插件发现和插件治理，系统开始有能力管理扩展能力，而不是把新增工具变成越来越乱的硬编码。

### 5.4 解决“进了真实工作流后缺协作机制”的问题

通过审批态、等待态、目录监听和 API 工具，系统具备了更接近真实工作过程的中间态处理能力，不再只有“成功 / 失败”两种粗糙结果。

### 5.5 解决“看起来完成了，其实不好用”的问题

通过交付工作包、工作台、每日工作台、通知和稳定收口，系统开始更像一个能给你清晰交付结果的工作助手，而不是只吐一串执行状态。

## 6. 当前版本的边界

到当前版本为止，**主功能阶段已经完成**。

这意味着：

- 不建议再继续无上限地开新功能阶段
- 不建议再把项目往“更复杂的通用平台”方向扩
- 当前更合理的节奏，是进入长期试运行和稳定收口

换句话说，现在最该做的不是“再想一个新阶段做什么”，而是：

- 真机连续使用
- 看真实卡点
- 修真实不稳定问题
- 简化复杂度

这才符合“我要一个自己长期用的通用智能体”，而不是“我要永远在开发中”的目标。

---

## 7. 当前版本已知边界与限制

虽然当前版本已经到可用阶段，但还存在一些明确边界：

### 7.1 仍然是单机、自维护、轻量级架构

这套系统现在非常适合你个人本地长期使用，但它不是为大规模多用户、多租户、分布式部署设计的。

### 7.2 UI 主要做了代码和测试级验证

本轮大量功能都做了代码级和测试级回归，但并不是所有 Streamlit 页面都做过完整的人工点击验收。

这意味着：真实使用过程中，UI 层仍可能暴露一些交互细节问题，这属于正常的收口期问题，不属于方向性问题。

### 7.3 pytest 缓存仍有权限 warning

当前测试已经收敛到统一目录，不会污染项目根目录，但 `pytest` 写缓存时仍有一个权限 warning。

这不会影响功能正确性判断，但仍然是环境层的一个小噪音。

### 7.4 仍然存在历史编码差异

仓库里部分旧文件存在历史编码问题，虽然当前功能链路已经能正常工作，但后续如果继续维护旧文件，仍建议逐步统一编码，降低后续 patch 风险。

---

## 8. 当前版本建议的使用方式

这一版最适合的使用方式，不是继续疯狂开发，而是这样用：

1. 先用真实日常工作跑一段时间
2. 优先观察：
   - 哪些任务最常用
   - 哪些等待态最常出现
   - 哪些恢复链路最容易卡
   - 哪些模板最值得保留
3. 把真实出现的问题记录下来
4. 后续只做“影响长期使用体验”的修正

也就是说，后面建议进入：

- 使用驱动的稳定收口
- 而不是功能驱动的继续扩张

---

## 9. 当前版本的结论

当前版本已经完成了从“任务执行器原型”到“个人通用智能体可用版”的关键跃迁。

它已经具备：

- 可扩展的工具调度能力
- 可治理的插件接入能力
- 可长期运行的任务底座
- 可持续推进工作的上下文模型
- 可接入真实工作流的协作能力
- 基本足够的稳定收口

因此，这个版本最合理的定位是：

**它已经可以进入长期试运行，而不是继续无限期大开发。**

后续的重点，不再是“还能再做多少新能力”，而是：

**把它变成你自己真正愿意长期依赖的个人数字员工。**
