# OmniCore 通用 Agent 路线图状态

日期: 2026-03-04
参考: `docs/2026-03-03-通用Agent演进路线图.md`

这份文档用于补充路线图的当前落地状态，避免原始路线图只描述目标、不反映真实进展。

## 当前结论

- `Phase 0` 已进入后期收口，但未完全完成。
- `Phase 1` 已有可用基础设施，但仍是轻量版，不是完整形态。
- `Phase 2` 只完成了部分前置能力。
- `Phase 3` 基本未开始。

## Phase 0 状态

### 已完成

- 已有统一的 `ToolSpec / ToolContext / ToolResultEnvelope` 抽象。
- 已建立 `ToolRegistry`，并把现有四类能力注册为内建工具。
- `TaskExecutor` 主执行路径已改为通过注册工具解析后执行。
- 执行层已从 `task_executor` 大分支中拆出独立 `tool_adapters` 模块。
- `Router` 已偏向输出 `tool_name + tool_args`。
- `task_type` 已降级为兼容字段，不再是新的推荐主路径。
- 已接入确定性权限策略，文件写入、系统命令、交互式浏览器操作会被规则层标记。
- 已接入 `BrowserRuntimePool`。
- 已接入 `LLM Cache`，并覆盖 URL 分析与页面结构分析。
- 最终输出已从“任务状态列表”升级为“结果摘要 + artifact + issue”。

### 部分完成

- 新增工具仍然需要在 adapter 层注册，尚未达到“完全无需触碰执行层”的插件化状态。
- 权限治理已收紧，`policy_decisions` 已进入状态与 job 持久化，但还没有独立的审计视图和更完整的审批流程。
- `TaskType` 虽已降级，但兼容链路仍保留，尚未完全退出。

### 未完成

- 还没有独立的工具插件目录与更彻底的动态发现机制。
- 还没有完整的权限白名单/资源级授权模型。

## Phase 1 状态

### 已完成

- 已有 `Session / Job / Artifact` 持久化基础结构。
- 已有轻量 `Job Queue`。
- 已有后台 worker，可自动消费队列。
- `run_task()` 已是“提交后立即执行”的兼容入口。
- 已有 `Session History` 页面，可查看历史、重跑、恢复失败任务。
- 最近 session 的 artifact 已注入规划上下文。

### 部分完成

- 当前恢复能力更接近“重排队 / 重跑”，不是严格的断点续跑。
- 当前后台 worker 仍是单进程内线程模型，不是独立长期运行执行器。

### 未完成

- 没有真正的 checkpoint / resume-from-checkpoint。
- 没有完整的审计日志模型和恢复点模型。

## Phase 2 状态

### 已完成

- 已有基础 artifact 持久化。
- 最终交付层已具备初步的产物展示能力。

### 部分完成

- `Artifact` 仍挂在 runtime state store 中，尚未演进成独立 `Artifact Store`。

### 未完成

- 记忆尚未拆分为工作记忆 / 情景记忆 / 用户记忆 / 资源记忆。
- 用户偏好尚未真正进入运行时主流程。
- 尚未把历史成功路径稳定用于规划优化。

## Phase 3 状态

### 未完成

- 第三方工具插件化接入
- 定时任务与事件触发
- 长期目标管理
- 更强的世界模型
- 主动提醒与建议

## 本轮新增收口

- `system_worker` 已切换为 `shell=False` 执行，阻断 shell/interpreter 跳板。
- `file_worker.write_file()` 不再允许通过 `require_confirm=False` 直接绕过确认；只有内部预确认路径才可跳过重复确认。
- `policy_decisions` 已进入 LangGraph state，并会随 job 一起落盘，确认节点会把待确认记录更新为 `approved` 或 `rejected`。

## 下一步建议

1. 把 `policy_decisions` 挂入状态与持久化层，形成可审计记录。
2. 继续把工具执行从“注册中心 + 适配器注册”推进到更彻底的插件化发现。
3. 开始做真正的 checkpoint 恢复，而不是只做重跑和重排队。

## 2026-03-04 最新补充

- `policy_decisions` 已经接入可视化审计入口。
- `ui/pages/2_Session_History.py` 现在会按 Session 汇总所有授权记录，并支持：
- `decision` 筛选
- `risk_level` 筛选
- `tool_name` 筛选
- `pending / approved / rejected` 聚合统计
- 按 decision / tool 的聚合视图

- 工具适配器插件化继续推进。
- `core/tool_adapters.py` 现在支持通过 `TOOL_ADAPTER_PLUGIN_MODULES` 按模块路径动态导入外部插件。
- `core/tool_adapters.py` 现在也支持通过 `TOOL_ADAPTER_PLUGIN_DIRS` 扫描插件目录。
- 外部插件只要在导入时通过 `@tool_adapter(...)` 自注册，就可以被运行时发现。
- 外部插件现在也可以通过 `@tool_plugin(...)` 自带 `ToolSpec`，直接进入 `ToolRegistry`。
- 插件现在支持模块级 `PLUGIN_MANIFEST`，可声明版本、依赖和默认启用状态。
- 插件 manifest 已加入基础校验，非法 `plugin_id / version / dependencies` 会被拦截。
- `Session History` 页面会显示插件模块的配置、已加载状态和加载错误。
- `Session History` 页面也会展示插件 manifest 和启用状态。
- `Session History` 页面已具备基础插件管理能力：安装源、启用、禁用、卸载。
- `Session History` 页面现在也会显示 checkpoint，并支持 `Resume From Checkpoint`。
- `core/graph.py` 和 `core/runtime.py` 已开始在关键阶段落盘 checkpoint，用于恢复。
- `core/router.py` 现在会从 `ToolRegistry` 动态生成工具目录，插件工具会自动进入 Router 提示词。
- `ENABLED_TOOL_PLUGIN_IDS / DISABLED_TOOL_PLUGIN_IDS` 已可控制插件工具是否真正进入运行时。
- 同时引入了持久化插件管理配置，运行时会自动读取 UI 安装/禁用/卸载过的插件状态。

- 这意味着当前 `Phase 0` 的“工具插件化”已经从“代码内自注册”推进到“配置驱动的外部模块发现”。
- 同时，工具插件已经不再只扩展 adapter，而是能一体化扩展 “ToolSpec + adapter + Router 可见性”。
- 插件现在也具备了最基础的版本与启停治理能力。
- 但仍未达到完全独立插件目录、自动扫描、热插拔的最终形态。
- 当前的 checkpoint 恢复已经进入可用阶段，但仍属于单进程、同图结构下的轻量恢复，不是完整的跨版本工作流迁移方案。

## 2026-03-04 稳定性收口补充

- `resume_job_from_checkpoint()` 现在会优先选择最近的非 `finalize` checkpoint，避免恢复时直接跳过整个工作流。
- 恢复执行时会刷新最新的会话 artifact 和相关记忆，而不是完全依赖旧 checkpoint 中的上下文快照。
- 运行结果现在会附带：
- `delivery_package`
- `checkpoint_summary`
- `resumed_from_checkpoint / resume_checkpoint_id / resume_checkpoint_stage`

- Router 的稳定性也补了一层确定性兜底：
- 如果 LLM 产出的任务缺少明确 `tool_name`，运行时会按任务描述、参数和工具元数据做确定性工具猜测。
- Router 在请求 LLM 前还会注入一段 `Deterministic tool hints`，降低选错工具的概率。

- 最终交付从“状态汇总”进一步收口到更像工作产出的结构：
- `headline`
- `completed_tasks`
- `deliverables`
- `issues`
- `recommended_next_step`

## 2026-03-04 长期运行阶段补充

- 已引入基础任务调度器：
- 支持 `once / interval / daily` 三种计划任务类型
- 到点任务会自动释放到队列，而不是只能手工提交
- 支持暂停 / 恢复 / 删除计划任务

- 后台执行器已从“仅线程内 worker”扩展为“可独立进程启动的 worker”：
- `QUEUE_WORKER_MODE=process` 时，后台队列可由独立进程消费
- Worker 状态会持久化 `mode / pid / heartbeat`
- `main.py worker --process-loop` 可作为独立前台 worker 入口

- checkpoint 恢复进一步增强：
- `resume_job_from_checkpoint()` 现在支持指定 `checkpoint_id`
- UI 可以选择具体 checkpoint 进行恢复
- 恢复结果会返回 `resume_strategy`

- 用户偏好已进入运行时主流程：
- 支持按 Session 保存 `default_output_directory`
- 支持保存 `preferred_tools / preferred_sites`
- 支持 `auto_queue_confirmations`
- Router 会把这些偏好注入规划上下文
- `file_worker` 在未指定路径时会优先使用偏好输出目录

- 已补上基础通知闭环：
- 任务完成 / 失败 / 带问题完成后会自动生成通知
- 通知支持未读筛选与标记已读
- `Session History` 页面可查看通知、批量标记已读
- 主聊天页侧边栏会显示最近未读通知

- 这意味着当前系统已经从“可用的任务执行器”推进到“具备基础长期运行能力的个人 Agent Runtime”：
- 有计划任务
- 有后台独立执行
- 有恢复入口
- 有偏好注入
- 有结果通知

## 2026-03-04 工作闭环阶段补充

- 已引入轻量 `Goal / Project / Todo` 持久化模型：
- 可按 Session 创建长期目标、工作项目和待办事项
- Job 现在可带 `goal_id / project_id / todo_id`
- Todo 会随成功执行自动推进到 `done`，失败时推进到 `in_progress`

- 已引入独立 `Artifact Store`：
- 运行时产物会在 job 完成后同步进入独立 catalog
- 支持按 `session / goal / project / todo` 检索
- 这使得 artifact 不再只是 runtime state 的附属记录，而是可复用资源层

- 规划层已开始复用工作上下文与成功经验：
- Runtime 会把当前 `Goal / Project / Todo` 上下文注入 `shared_memory`
- Runtime 会把可复用的资源产物注入 `resource_memory`
- Runtime 会从历史成功经验中提取 `successful_paths`
- Router 会把这些信息显式拼进规划上下文，优先复用已有成果和已验证路径

- 已新增 Workbench 视图：
- 可创建和查看 `Goal / Project / Todo`
- 可手动更新 Todo 状态
- 可查看独立 Artifact Store 中的可复用资源
- 主聊天页也支持为当前会话选择活动的 `Goal / Project / Todo`

- 最终交付已升级为更贴近“工作包”：
- `delivery_package` 现在会带：
- `goal / project / todo`
- `open_todos`
- 前端会展示当前工作上下文和未完成事项

- 这意味着当前系统已经从“长期运行的个人 Agent Runtime”进一步推进到“具备基础工作闭环和经验复用能力的个人数字员工雏形”。

## 2026-03-04 真实工作接入与协作阶段补充

- 已引入 `api.call` 工具：
- 可执行标准 HTTP API 请求
- `GET / HEAD / OPTIONS` 会直接执行
- 变更型请求（如 `POST / PUT / PATCH / DELETE`）会先进入 `waiting_for_approval`
- 用户批准后可基于 checkpoint 继续执行，而不是整条任务从头重跑

- 已补齐等待态：
- Job 现已支持 `waiting_for_approval`
- 任务层已支持 `waiting_for_approval / waiting_for_event / blocked`
- `finalize` 会把这些状态转成明确的交付结果和下一步建议

- 已引入轻量目录监听：
- 可按目录创建 watch
- 新文件出现时会自动触发任务
- 目录监听可绑定模板或直接绑定 prompt
- watch 本身以 `waiting_for_event` 挂起，直到检测到新文件

- 已引入工作模板：
- 可手动创建模板
- 可从成功 Job 快速保存模板
- 可直接从模板创建新任务
- 模板也可以被目录监听复用，作为事件触发后的标准工作流

- 已补齐每日工作台：
- 新增 `Daily Dashboard`
- 可查看：
- 今日完成
- 等待审批
- 等待事件
- 被阻塞任务
- 打开的 Todo
- 推荐下一步

- 这意味着当前系统已经进入“真实工作接入与协作”阶段的可用形态：
- 能接真实 API
- 能等你审批后继续
- 能等文件事件后继续
- 能把成功流程沉淀成模板
- 能用工作台方式管理当天事项
