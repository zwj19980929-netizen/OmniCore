# OmniCore v0.2 智能化改造实施文档（可执行版）

> 目标：把当前“可执行但偏死板”的系统，升级为“可自适应、可反思、可学习”的智能体系统。  
> 原则：小步快跑，不重写，不破坏现有 MVP 闭环。

---

## 0. 改造目标（必须达成）

### 0.1 总体目标
1. 提升任务成功率（尤其是网页结构变化、超时、交互不确定场景）。
2. 提升任务智能感（不是固定流程，而是执行中动态修正策略）。
3. 提升长期效果（系统可以“学会”什么策略在什么场景有效）。

### 0.2 成功标准（验收指标）
- 端到端任务成功率：提升 20%+
- 首次执行成功率：提升 15%+
- 平均重试次数：下降 30%
- 执行时长：不增加超过 20%
- 高危误执行：0（必须保持）

---

## 1. 架构升级总览

从当前：
- Router（一次性计划）→ Worker 串行执行 → Critic → 结束/重规划

升级为：
- Router（任务图 + 成功判据 + 备选策略）  
→ Scheduler（依赖驱动调度）  
→ Worker（每步微反思）  
→ Validator（硬规则）  
→ Critic（语义审查）  
→ Memory（沉淀策略模板）  
→ 下一次 Router 优先复用历史高成功策略

---

## 2. 分阶段实施（按周推进）

### Phase 1（Week 1-2）：先提升“智能感”和稳定性

#### 2.1 Router 输出升级（必须）
把任务输出从“任务列表”升级为“任务图节点”结构，每个任务包含：

- `task_id`
- `task_type`
- `description`
- `params`
- `depends_on`
- `success_criteria`（明确成功判据）
- `fallbacks`（失败后的备选方案）
- `abort_conditions`（必须停止条件）
- `priority`

**要求：**
- 老格式兼容（若无新字段，自动补默认值）
- 不允许仅靠自然语言模糊描述“成功”

#### 2.2 Worker 内部加“微反思循环”（必须）
每个 Worker 执行任务时，从一次性执行改为：

1. 计划当前一步（Step Plan）
2. 执行动作（Action）
3. 观察结果（Observation）
4. 判断是否满足 `success_criteria`
5. 若不满足，尝试 `fallbacks`
6. 达到最大步数后失败返回

**建议参数：**
- `max_steps_per_task = 6`
- `max_fallback_attempts = 2`

#### 2.3 Critic 两级化（必须）
拆成：
1. **Validator（硬规则）**  
   - 文件是否存在  
   - 数据是否为空  
   - URL 格式是否合法  
   - 必填字段完整
2. **Critic（LLM 审查）**  
   - 语义质量  
   - 结果与任务目标一致性

**执行顺序：** Validator 先跑，不通过直接判失败，不进 LLM 审查。

### Phase 2（Week 3-4）：核心调度智能升级

#### 2.4 调度器从“串行扫描”改为“依赖驱动”
实现 `ready queue` 机制：
- 仅当 `depends_on` 全部完成时，任务可进入 ready 状态
- ready 任务按 `priority` 选择执行
- 支持并发执行（先做同类型并发上限控制）

**并发建议：**
- web/browser: 2
- file/system: 1

#### 2.5 失败分类与恢复策略库（必须）
建立统一 `failure_type`：
- `timeout`
- `selector_not_found`
- `blocked_or_captcha`
- `permission_denied`
- `invalid_input`
- `unknown`

每类故障映射固定恢复动作，例如：
- `selector_not_found` → 改用语义定位/相邻节点定位
- `timeout` → 降低等待条件+重试一次
- `blocked_or_captcha` → 切手动确认流程

#### 2.6 模型路由（建议）
按任务类型分配模型：
- 规划/反思/审查：强推理模型
- 抽取/格式化/改写：便宜模型
- 高风险判断：稳定保守模型

### Phase 3（Week 5-8）：长期学习能力

#### 2.7 记忆三层化（必须）
新增 memory schema：

1. `episodic_memory`：完整任务轨迹（输入-计划-执行-结果）
2. `semantic_memory`：领域知识片段
3. `procedural_memory`：可复用策略模板（重点）

`procedural_memory` 示例字段：
- `scenario_signature`（场景签名）
- `strategy_steps`
- `applicable_conditions`
- `success_rate`
- `avg_cost`
- `last_verified_at`

#### 2.8 Router 接入“策略优先复用”
Router 规划前先检索 procedural memory：
- 若匹配到高成功率策略，优先生成“策略实例化任务图”
- 若无匹配，再走自由规划
- 执行结束后回写策略成功率（在线更新）

## 3. 数据结构定义（建议落地）

### 3.1 TaskItem 扩展
```json
{
  "task_id": "task_1",
  "task_type": "web_worker",
  "description": "抓取前5条新闻",
  "params": {},
  "depends_on": [],
  "priority": 10,
  "success_criteria": [
    "result.success == true",
    "len(result.data) >= 5"
  ],
  "fallbacks": [
    {"type": "retry", "param_patch": {"wait_ms": 5000}},
    {"type": "switch_worker", "to": "browser_agent"}
  ],
  "abort_conditions": [
    "risk_score > 0.8",
    "manual_cancelled == true"
  ]
}
```

### 3.2 ExecutionTrace（新增）
```json
{
  "task_id": "task_1",
  "steps": [
    {
      "step_no": 1,
      "plan": "打开目标页面并等待内容加载",
      "action": "goto(url)",
      "observation": "页面加载成功",
      "decision": "继续"
    }
  ],
  "failure_type": null,
  "fallback_used": false
}
```

## 4. 安全机制（不得退化）

1. 高危动作仍需 human confirm。  
2. 新增风险分值 `risk_score`（0~1）：
   - 删除/转账/发送消息等动作 >= 0.7 必须确认
3. fallback 不得绕过安全确认（强约束）

## 6. 测试计划（必须自动化）

### 6.1 单元测试
- Router schema 完整性测试
- depends_on 调度正确性测试
- failure_type 映射测试
- fallback 执行路径测试

### 6.2 集成测试
- Hacker News 抓取 + 保存文件（主链路）
- 页面结构变化模拟（selector 失效）
- 超时场景重试后成功
- 高危操作确认拦截

### 6.3 回归标准
- 所有旧测试通过
- 新增测试通过率 100%
- 无安全回归

---

## 7. 交付顺序（严格执行）

1. 先交 Router 新 schema + 兼容层  
2. 再交 Scheduler depends_on 改造  
3. 再交 Worker 微反思 + fallback  
4. 再交 Validator 两级审查  
5. 最后交 procedural memory 与策略复用

---

## 8. 禁止事项（非常重要）

- 禁止一次性大重构全部模块
- 禁止移除 human confirm 安全机制
- 禁止只改 prompt 不改执行逻辑
- 禁止写“看起来智能”的伪逻辑（必须有可观测状态变化）

---

每一阶段输出必须包含：
1. 修改文件列表

3. 新增/修改测试清单
4. 运行命令与结果

## 10. 最终目标一句话

把 OmniCore 从“可执行脚本编排器”，升级为“能在执行中思考、失败后自适应、成功后会学习”的智能体系统。
