# 通用Agent当前架构说明

日期: 2026-03-04

## 1. 这份文档的作用

这份文档用于替代早期“阶段推进式”的说明口径，作为当前版本的统一架构说明。

它回答的是当前时点的 4 个问题：

1. OmniCore 现在到底是什么
2. 系统当前的核心架构是什么
3. 各层分别负责什么
4. 现在项目处于什么阶段，后面该怎么维护

这份文档是“当前权威口径”。
早期的阶段性路线图和落地说明，仍保留在 `docs/` 中，但只作为历史快照，不再作为当前主参考。

## 2. 当前定位

当前版本的 OmniCore，已经不再只是“一个能调用几个工具的任务执行器”。

它现在更准确的定位是：

**一个面向个人场景的、可长期运行的通用 Agent Runtime 可用版。**

这个定位有 3 个关键点：

1. 面向个人，而不是多租户平台  
目标是做“你自己长期使用的数字员工”，不是对外平台产品。

2. 可长期运行，而不是一次性脚本  
它已经具备队列、后台 worker、计划任务、通知、checkpoint、恢复能力。

3. 通用 Agent Runtime，而不是单功能工具  
它已经具备工具调度、工作上下文、产物沉淀、经验复用和真实工作接入能力。

因此，当前系统的目标不是“继续无上限堆新功能”，而是：

**进入长期使用，并用真实工作把稳定性收出来。**

## 3. 当前架构总览

当前架构可以分成 7 层。

### 3.1 交互层

负责接收输入、展示结果、呈现工作台和运维视图。

包括：

- CLI 入口
- 主聊天页
- Session History
- Workbench
- Daily Dashboard

这一层的职责是“交互”和“可见性”，不负责核心业务决策。

### 3.2 运行时与编排层

这是当前系统的中枢。

负责：

- 接收任务
- 提交 Job
- 载入上下文
- 调用 LangGraph
- 写回结果
- 驱动队列、调度器、后台 worker、checkpoint

这一层把“单次调用”升级成了“长期运行的任务系统”。

### 3.3 规划与决策层

负责把自然语言目标变成可执行计划。

包括：

- Router
- Task Planner
- Validator
- Critic
- Policy Engine

这一层的职责不是直接执行，而是：

- 识别意图
- 生成任务计划
- 选择工具
- 做确定性规则兜底
- 判断是否需要人工确认

### 3.4 工具调度层

负责把计划里的工具调用真正变成执行动作。

包括：

- Tool Protocol
- Tool Registry
- Tool Adapters
- Task Executor

这是当前系统从“固定 Worker 分发”转向“Tool-First 调度”的关键层。

### 3.5 工具能力层

负责具体执行动作。

当前核心能力包括：

- web / browser
- file
- system
- api.call

插件工具在这一层扩展，但前提是经过注册和权限控制。

### 3.6 状态、记忆与工作上下文层

负责让系统不再只是执行一次性任务，而是围绕“持续工作”运行。

包括：

- Session / Job / Artifact
- Goal / Project / Todo
- Runtime State Store
- Artifact Store
- Work Context Store
- 用户偏好
- 历史成功路径

这一层是“像员工一样持续推进事情”的基础。

### 3.7 自动化与协作层

负责让系统进入真实工作流。

包括：

- 计划任务
- 目录监听
- 工作模板
- 审批态
- 等待态
- 通知

这一层让系统从“被动执行”开始转向“主动等待条件、主动继续推进”。

## 4. 当前核心数据模型

当前系统最重要的业务对象，不再只是一个任务列表。

### 4.1 Session

表示一次持续工作会话。

作用：

- 聚合多个 Job
- 绑定用户偏好
- 绑定通知、历史产物、工作上下文

### 4.2 Job

表示一次任务提交和执行生命周期。

作用：

- 从 `queued` 到执行完成/等待/阻塞
- 记录输入、状态、结果、错误、checkpoint、policy decisions

### 4.3 Task

表示 Job 内部的子任务。

作用：

- 描述一个具体执行步骤
- 绑定工具、参数、状态、依赖、结果

### 4.4 Artifact

表示执行后产生的可复用产物。

包括：

- 文件
- 下载内容
- 图片
- 结构化结果

它的作用不是“顺手记一下输出”，而是作为后续规划和执行可复用的资源层。

### 4.5 Goal / Project / Todo

表示持续工作的业务上下文。

作用：

- 把多个 Job 串成一个长期事项
- 让系统知道“我现在在推进什么”
- 让交付结果和后续动作有连续性

### 4.6 Policy Decision

表示一次权限和风险判断。

作用：

- 记录某个任务是否需要确认
- 记录是批准、拒绝还是待处理
- 提供后续审计依据

### 4.7 Template / Directory Watch / Schedule

表示自动化入口。

作用：

- 模板让成功流程可复用
- 目录监听让文件事件可触发任务
- 计划任务让时间可触发任务

## 5. 当前关键业务流

### 5.1 主动提交任务

主链路是：

`用户输入 -> submit_job -> 构造上下文 -> Router/Planner -> Tool Dispatch -> Validator/Critic -> Delivery Package -> 持久化`

它已经不是单纯的“函数调用”，而是一套完整的任务生命周期。

### 5.2 后台消费任务

主链路是：

`Queue -> Worker -> run_next_queued_task -> 执行 -> 回写 Job/Artifact/通知`

这让系统具备长期挂起执行的能力。

### 5.3 计划任务触发

主链路是：

`Schedule 到点 -> 释放到队列 -> Worker 消费`

这让系统开始具备时间驱动的主动执行能力。

### 5.4 目录监听触发

主链路是：

`目录出现新文件 -> 生成文件事件 -> 自动创建任务 -> Worker 消费`

如果监听关联的模板失效，当前版本会自动降级为直接使用 fallback prompt，而不是丢事件。

### 5.5 审批与等待态

主链路是：

`高风险动作 -> waiting_for_approval -> 用户批准/拒绝 -> 恢复执行或进入 blocked`

这让系统的状态不再只有“成功 / 失败”两种粗糙结果。

### 5.6 Checkpoint 恢复

主链路是：

`运行中关键阶段落 checkpoint -> 失败/等待后选择 checkpoint -> 恢复执行`

当前已是“可用恢复”，但仍属于轻量恢复，不是跨版本、跨复杂工作流迁移系统。

## 6. 当前版本为什么这样设计

当前架构不是按“功能越多越好”设计的，而是按“个人长期可用”设计的。

核心设计原则是：

1. 稳定优先于花哨  
先把能长期运行、能恢复、能追踪做出来，再谈更复杂的智能能力。

2. 工作闭环优先于工具数量  
比起再接更多工具，更重要的是让现有能力真的能围绕事情推进。

3. 可维护优先于过度平台化  
插件已经做到够用即可，不把项目拖进无底洞式的平台建设。

4. 真实使用优先于架构想象  
当前架构已经到一个足以长期试运行的节点，后续应该让真实使用暴露问题，而不是继续凭空扩张。

这意味着：  
**主功能阶段已经完成。**

## 8. 当前版本的边界

虽然当前版本已经可用，但要明确边界，不要误把它当成另一个目标。

### 8.1 它是个人 Agent，不是多租户平台

当前设计是单机、自维护、个人长期使用优先。

### 8.2 它是可用版，不是最终成熟版

它已经足够进入长期试运行，但仍需要通过真实使用收出稳定性问题。

### 8.3 它是“够用的插件治理”，不是完整插件平台

插件现在已经够接入和管理，但不值得继续在这个阶段深挖平台化。

### 8.4 它是“轻量恢复”，不是复杂分布式恢复系统

checkpoint 已经有实际价值，但当前目标不是做企业级复杂工作流恢复框架。

---

## 9. 当前阶段的正确维护方式

现在项目不应该再继续按“开新大阶段”推进。

当前最合理的维护方式是：

1. 进入长期试运行  
用真实工作来跑，观察它到底哪里卡。

2. 只修真实问题  
优先修：
- 稳定性问题
- 恢复问题
- 错误状态展示问题
- 默认行为不合理的问题

3. 控制新功能冲动  
没有真实使用证据支持时，不再默认开新的大功能线。

4. 保持文档口径统一  
以后优先维护这份文档和“收尾说明 / 状态文档”，不要再把旧阶段文档当当前说明。
