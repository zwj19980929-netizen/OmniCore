# 增强网页感知能力 - 实施方案

## 问题诊断

你的 agent 网页感知能力差的根本原因：

1. **信息损失严重**：HTML 被暴力清洗后只保留前 15000 字符，LLM 看到的是残缺片段
2. **缺少结构化理解**：直接把原始 HTML 扔给 LLM，没有提取语义化的页面结构
3. **没有"理解"阶段**：直接跳到"提取数据"，没有先让 LLM 理解"这个页面是干什么的"
4. **元素提取不完整**：只提取可交互元素，忽略了内容结构（标题、列表、表格等）

## 解决方案：三层感知架构

### Layer 1: 页面感知器 (Page Perceiver)
**作用**：将原始 HTML 转换为结构化的语义描述

**核心思路**：
- 在浏览器中执行 JS，提取页面的**语义结构**（标题、列表、表格、段落、导航等）
- 不是给 LLM 看 HTML 源码，而是给它看"页面说明书"
- 包含：主要内容区域、导航区域、可交互元素、元数据

**实现文件**：`utils/page_perceiver.py`

### Layer 2: 页面理解 (Page Understanding)
**作用**：让 LLM 理解页面的功能和内容组织方式

**核心思路**：
- 基于 Layer 1 的结构化描述，让 LLM 回答：
  - 这是什么类型的页面？（列表页/详情页/搜索页等）
  - 主要功能是什么？
  - 目标数据在哪里？
  - 数据是如何组织的？
  - 需要什么交互操作？

**实现文件**：`prompts/page_perception.txt` (第一阶段)

### Layer 3: 选择器生成 (Selector Generation)
**作用**：基于页面理解，生成精确的数据提取策略

**核心思路**：
- LLM 已经理解了页面结构，现在生成具体的 CSS 选择器
- 不是盲目猜选择器，而是基于理解做出决策
- 包含：数据项选择器、字段选择器、分页配置、预操作等

**实现文件**：`prompts/page_perception.txt` (第二阶段)

## 新增文件说明

### 1. `utils/page_perceiver.py`
页面感知器核心实现，包含：
- `PageBlock`: 页面内容块数据结构
- `PageStructure`: 页面结构化表示
- `PagePerceiver`: 感知器类，执行 JS 提取页面结构
- `get_page_understanding()`: 便捷函数，返回 LLM 友好的页面描述

### 2. `prompts/page_perception.txt`
三阶段 Prompt 定义：
- 第一阶段：页面理解 - 分析页面类型和功能
- 第二阶段：选择器生成 - 生成提取配置
- 第三阶段：操作规划 - 规划下一步操作

### 3. `agents/enhanced_web_worker.py`
增强版 Web Worker，集成三层感知：
- `_understand_page()`: Layer 1 - 页面理解
- `_generate_selectors()`: Layer 2 - 选择器生成
- `_execute_extraction()`: Layer 3 - 执行提取
- `plan_next_action()`: 操作规划（用于多步骤任务）

### 4. `tests/test_enhanced_perception.py`
测试脚本，验证三层感知能力

在 `agents/web_worker.py` 的 `analyze_page_structure` 方法中：

```python
async def analyze_page_structure(self, tk: BrowserToolkit, task_description: str):
    # 使用新的页面感知器
    from utils.page_perceiver import get_page_understanding

    page_structure = await get_page_understanding(tk, task_description)

    # 使用新的 prompt
    response = self.llm.chat_with_system(
        system_prompt=PAGE_UNDERSTANDING_PROMPT.format(
            task_description=task_description,
            page_structure=page_structure
        ),
        user_message="请分析页面并生成选择器配置",
        temperature=0.2,
        json_mode=True
    )

    return self.llm.parse_json_response(response)
```

## 核心优势

### 对比旧方案：
**旧方案**：HTML → 清洗 → 截断 → 扔给 LLM → 猜选择器
- LLM 看到的是残缺的 HTML 片段
- 没有上下文，不知道页面是干什么的
- 盲目猜测选择器，成功率低

**新方案**：HTML → JS 提取结构 → 语义化描述 → LLM 理解 → 生成策略
- LLM 看到的是完整的页面"说明书"
- 先理解页面功能，再决定如何提取
- 基于理解生成选择器，准确率高

### 关键改进：

1. **信息完整性**：不再截断 HTML，而是提取结构化信息
2. **语义化表示**：给 LLM 看的是"页面有什么"，而不是"HTML 长什么样"
3. **分层决策**：理解 → 规划 → 执行，每一层都有明确目标
4. **可解释性**：每一步都有推理过程，便于调试

## 下一步建议

1. **先测试页面理解层**：运行测试模式 2，看看 LLM 能否正确理解页面结构
2. **调整 Prompt**：根据测试结果，优化 `prompts/page_perception.txt` 中的 prompt
3. **集成到现有代码**：选择方式 2 或 3，将新架构集成到你的 web_worker 或 browser_agent
4. **逐步替换**：不要一次性替换所有代码，先在一个场景中验证效果

## 注意事项

1. **性能开销**：三层感知会增加 LLM 调用次数（理解 + 选择器生成），但准确率会大幅提升
2. **缓存策略**：页面理解结果已经做了缓存，相同页面不会重复分析
3. **降级方案**：如果三层感知失败，可以回退到原有的简单提取逻辑
4. **Prompt 调优**：根据你的实际场景，调整 prompt 中的示例和规则

## 总结

这套方案的核心思想是：**让 LLM 先"看懂"网页，再决定怎么操作**。

就像人类浏览网页一样：
1. 先看看这是什么页面（理解）
2. 找到目标信息在哪里（定位）
3. 决定如何获取（操作）

而不是盲目地猜测和尝试。
