# OmniCore 文档中心

## 目录结构

```
docs/
├── README.md                  # 本文件：文档索引与规范
├── architecture/              # 核心架构参考（当前有效）
├── design/                    # 功能设计与规划方案
├── archive/                   # 历史文档、修复报告、实施总结
├── claude code详细的设计/      # Claude Code 架构研究笔记
├── solution/                  # 解决方案参考
└── img/                       # 文档配图
```

## 各目录说明

| 目录 | 放什么 | 不放什么 |
|---|---|---|
| `architecture/` | 当前系统架构说明、路线图、版本收尾说明 | 单个功能的设计文档 |
| `design/` | 功能设计方案、优化规划、技术选型 | 临时修复记录、实施流水账 |
| `archive/` | 已完成的阶段总结、修复报告、过时的设计文档 | 仍在参考的活跃文档 |
| `claude code详细的设计/` | Claude Code 架构研究与代码细读 | OmniCore 自身的设计文档 |
| `solution/` | 通用解决方案、技术方案参考 | — |

## 文档规范

以下规范适用于所有新建和更新的文档，**智能体在写文档时必须遵守**。

### 1. 命名规则

```
YYYY-MM-DD-<简短标题>.md
```

- 日期使用 ISO 格式（`2026-04-03`），以创建日期为准
- 标题使用中文或英文均可，但要简洁明确，能通过文件名判断内容
- 使用 `-` 连接词，不使用空格
- 如果属于系列文档，使用统一前缀编号，如 `R1-`、`S2-`、`P0-1-`

### 2. 文档头部模板

每篇文档开头必须包含以下信息：

```markdown
# 标题

> 状态：草稿 | 进行中 | 已完成 | 已归档
> 创建：YYYY-MM-DD
> 最后更新：YYYY-MM-DD

## 背景与目标

（为什么要做这个，解决什么问题）

## 方案概述

（核心思路，一段话说清楚）
```

### 3. 存放规则

- **新的设计方案** → `design/`
- **架构级变更**（影响整体分层或核心数据模型） → `architecture/`
- **已完成且不再更新的文档** → `archive/`
- **修复报告、实施流水账** → `archive/`（或者不写，代码提交记录本身就是最好的记录）

### 4. 生命周期

1. 新方案创建在 `design/`，状态标记为 `草稿` 或 `进行中`
2. 方案实施完成后，将状态改为 `已完成`
3. 当文档内容已被代码完全体现、不再有参考价值时，移入 `archive/`，状态改为 `已归档`

### 5. 禁止事项

- **不要在 docs 根目录直接放文档文件**，必须放入对应子目录
- **不要写纯流水账式的"完成总结"**，如果只是记录"做了什么"，commit message 和 PR description 已经足够
- **不要重复造文档**，更新已有文档优于新建文档
- **不要在文档中硬编码配置值**（如模型名、token 数、超时时间），应引用配置文件或环境变量
