## 问题 1: 视网膜脱落 - 只看"交互"，不看"内容" ✅ **确认存在**

### 代码证据

**文件：** `agents/browser_agent.py:739`

```javascript
const nodes = Array.from(document.querySelectorAll('a, button, input, textarea, select, [role="button"], [role="link"], [contenteditable="true"]'));
```

### 问题分析

1. **硬编码的选择器限制**
   - 只抓取：`a, button, input, textarea, select` 等交互元素
   - **完全忽略**：`p, h1, h2, h3, div, span, text` 等内容元素

2. **致命后果**
   - Agent 看不到页面上的普通文本、标题、段落
   - 无法理解按钮/链接的上下文语义
   - 例如：看到"进入官网"按钮，但看不到上方的"2024年最新高危漏洞"标题

3. **实际影响**
   - 在 CNNVD 这类复杂网站上，漏洞描述、严重等级、影响范围等关键信息都在普通文本中
   - Agent 只能看到"搜索"、"查询"、"详情"等按钮，但不知道这些按钮对应什么内容

### 对比：有限的内容提取

虽然代码中有 `_maybe_extract_data()` 方法（line 1811-1875）可以提取部分内容：

```javascript
const contentSelectors = [
  'main li', 'article li', '[role="main"] li', 'section li',
  'main p', 'article p', '[role="main"] p', 'section p',
  'main h1', 'article h1', 'main h2', 'article h2',
  'main h3', 'article h3', 'tr'
].join(', ');
```

**但问题是：**
- 这个方法只在特定场景下调用（`_extract_data_for_intent`）
- 提取的内容和交互元素是**分离的**，LLM 无法建立它们之间的关联
- 最多只提取 12 条内容（`slice(0, 12)`），对于复杂页面远远不够

## 问题 2: 暴力截断 - 简单粗暴的 HTML 阉割 ✅ **部分修复，但仍存在问题**

### 代码证据

**文件：** `agents/web_worker.py:1361-1362`
```python
if len(html) > 100000:
    html = html[:100000] + "\n... (truncated)"
```

**文件：** `agents/web_worker_singleflight.py:213-214`
```python
if len(html) > 100000:
    html = html[:100000] + "\n... (truncated)"
```

### 问题分析

1. **已经从 15,000 提升到 100,000**（根据之前的修复）
   - 这是一个改进，但仍然是**暴力截断**

2. **根本问题未解决**
   - 截断是**字符级别**的，不考虑 HTML 结构
   - 可能在标签中间截断：`<div class="important-data">重要内容...</div>` → `<div class="important-data">重要内`
   - 导致 HTML 结构破坏，后续标签全部失效

3. **更好的方案**
   - 应该使用 HTML 清洗（移除 `<script>`, `<style>`, 冗余属性）
   - 或者使用 DOM 树截断（保留完整的节点）
   - 或者提取关键区域（`<main>`, `<article>`, `[role="main"]`）

### 实际影响

- 复杂页面（如政府网站、企业系统）的 HTML 通常 50,000-150,000 字符
- 100,000 字符的限制仍然可能截断关键内容
- 截断后的 HTML 可能导致 CSS 选择器失效

## 问题 3: Token 灾难 - 用最笨的方法喂给大模型 ✅ **确认存在**

### 代码证据

**文件：** `agents/web_worker.py:1364-1372`

```python
response = self.llm.chat_with_system(
    system_prompt=PAGE_ANALYSIS_PROMPT.format(
        task_description=task_description,
        html_content=html,  # ← 直接传入原始 HTML
        current_url=url_r.data or "",
    ),
    user_message="请分析页面结构并返回选择器配置",
    temperature=0.2, json_mode=True,
)
```

### 问题分析

1. **原始 HTML 充满噪音**
   - `class="container mx-auto px-4 py-8 flex flex-col items-center justify-center"`
   - `style="margin: 0; padding: 0; font-family: Arial, sans-serif;"`
   - 大量的 `<div>` 嵌套，只有最内层才是真正的内容

2. **信噪比极低**
   - 估计 90% 以上是排版标记（class, style, id, data-* 属性）
   - 只有不到 10% 是有效信息（文本内容、href、name）

3. **Token 浪费严重**
   - 100,000 字符的 HTML ≈ 25,000 tokens
   - 其中可能只有 2,500 tokens 是有用的
   - 浪费了 22,500 tokens 的成本和上下文窗口

4. **大模型注意力分散**
   - "Lost in the Middle" 现象：关键信息被噪音淹没
   - 大模型需要在海量无用标记中寻找有效信息
   - 导致提取准确率下降

### 更好的方案

应该先清洗 HTML：
```python
# 移除 script, style 标签
html = re.sub(r'<script[^>]*>.*?</script>', '', html, flags=re.DOTALL)
html = re.sub(r'<style[^>]*>.*?</style>', '', html, flags=re.DOTALL)

# 移除冗余属性（保留 id, name, href, type, role）
html = re.sub(r'\s(class|style|data-[^=]+)="[^"]*"', '', html)

# 压缩空白
html = re.sub(r'\s+', ' ', html)
```

## 问题 4: 逻辑跨越 - 缺乏"先理解，后决策"的思考期 ✅ **确认存在**

### 代码证据

**文件：** `prompts/browser_page_assessment.txt:1-30`

```
You are evaluating whether the current browser page already advances the task.
Return JSON only with keys:
- page_relevant: boolean
- goal_satisfied: boolean
- reason: string
- evidence_indexes: array of integers
- confidence: number
- action: object with keys type, element_index, target_selector, value, description, fallback_selector, use_keyboard, keyboard_key

Task: {task}
Intent: {intent}
Query: {query}
Current URL: {url}
Page title: {title}
Last action: {last_action}
Visible data:
{data}

Candidate elements:
{elements}

Rules:
- Use only the visible data and candidate elements provided above.
- If the visible data already likely answers the task, choose action.type="extract".
- If the task is already complete without further extraction, choose action.type="done".
- If the page is a search results page and the snippets are relevant enough to answer the task, prefer "extract" over clicking through.
- If the current page is relevant but incomplete and a candidate likely leads to the best source or detail page, choose "click" and reference the element_index.
- Choose "input" only if the current page is not already showing relevant results or the search box clearly needs a different query.
- If you cannot justify a meaningful action from the evidence, choose "wait".
```

### 问题分析

1. **直接要求决策，没有理解阶段**
   - Prompt 直接要求：`choose action.type="extract"` 或 `choose "click"`
   - 没有先问："这是什么类型的页面？数据是如何组织的？"

2. **只给交互元素，不给页面结构**
   - `Candidate elements: {elements}` 只包含按钮、链接、输入框
   - 没有页面的整体结构、内容区域、数据表格等信息

3. **缺少页面理解（Page Understanding）阶段**
   - 人类浏览网页的流程：
     1. **理解页面**：这是搜索结果页？详情页？列表页？表单页？
     2. **定位目标**：我要找的信息在哪个区域？
     3. **决策操作**：我应该点击哪个链接？填写哪个表单？

   - 当前系统的流程：
     1. ~~理解页面~~（跳过）
     2. ~~定位目标~~（跳过）
     3. **直接决策**：从 10 个按钮里选一个点击（瞎猜）

4. **导致幻觉式决策**
   - LLM 没有足够的上下文来做出正确判断
   - 只能基于按钮的文本（"搜索"、"查询"、"详情"）来猜测
   - 经常生成不存在的 CSS 选择器，或者点击错误的链接

### 更好的方案

应该分两阶段：

**阶段 1：页面理解（Page Understanding）**
```
Analyze this page and answer:
1. Page type: search results / detail page / list page / form page / other
2. Main content area: where is the primary information located?
3. Data structure: how is the information organized? (table / list / cards / paragraphs)
4. Key elements: what are the most important interactive elements?
5. Navigation: how can I access more detailed information?
```

**阶段 2：操作决策（Action Decision）**
```
Based on your understanding of the page:
- Task: {task}
- Page type: {page_type}
- Main content: {main_content_summary}
- Available actions: {elements}

Choose the best action to advance the task.
```
