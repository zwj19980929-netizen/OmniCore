# OmniCore 通用 Agent 第三阶段落地说明

日期：2026-03-03

本文记录在第二阶段基础上继续推进的第三阶段改造。这个阶段完成了可恢复运行时第一版：`Session / Job / Artifact` 已落地，并补上了轻量 `Job Queue` 与 `Session History` 页面。

## 本阶段目标

第三阶段的目标不是让 OmniCore 一步变成长期自治 Agent，而是先把“一次任务调用”提升为“有可追踪身份的运行时记录”，并为后续后台执行留出队列入口：

- 同一轮连续交互可以归属到同一个 `Session`
- 每一次 `run_task()` 都生成一个独立的 `Job`
- 任务执行过程中产生的文件类产物可以沉淀为 `Artifact`
- `run_task()` 不再直接假定自己是唯一入口，而是走“提交后立即执行”的兼容路径

这一步的价值在于：后续要做会话恢复、后台队列、任务重试、产物索引，必须先有稳定的运行时对象。

## 已完成

1. 增加了运行时状态持久化模块
   - 新增 `utils/runtime_state_store.py`
   - 提供 `RuntimeStateStore`
   - 提供单例入口 `get_runtime_state_store()`
   - 默认把数据写到 `data/runtime_state/`

2. 引入了三类运行时对象
   - `Session`：连续交互的归属容器
   - `Job`：一次 `run_task()` 的执行记录
   - `Artifact`：任务产出的可引用对象（当前先覆盖文件类产物）

3. `run_task()` 已接入 Session / Job
   - `core/runtime.py` 现在支持传入可选的 `session_id`
   - 每次调用会先创建或复用 `Session`
   - 然后先提交 `Job` 到队列，再立即消费执行（兼容模式）
   - 返回结果中会附带 `session_id`、`job_id`、`artifacts`
   - 新增了 `submit_task()` 和 `run_next_queued_task()` 两个运行时入口

4. 任务结果会自动抽取 Artifact
   - 当前会优先从任务结果里的这些字段抽取：
   - `file_path`
   - `path`
   - `output_path`
   - `download_path`
   - `screenshot_path`
   - 对 `file_worker` 的写入任务，也会回退读取 `params.file_path`
   - 额外会从 `data / items / content` 中抽取结构化结果，记为 `structured_data` 类型 Artifact

5. LangGraph 状态已补运行时标识
   - `core/state.py` 现在增加了：
   - `session_id`
   - `job_id`
   - `artifacts`

6. UI 和 CLI 会复用同一个 Session
   - `ui/app.py` 在 Streamlit 会话态里保存 `agent_session_id`
   - 连续任务会传回同一个 `session_id`
   - `main.py` 的交互式 CLI 也会复用同一个 Session

7. 增加了独立的 Session History 页面
   - 新增 `ui/pages/2_Session_History.py`
   - 可查看队列摘要、Session 列表、Job 历史、Artifact 历史
   - 可直接提交排队任务
   - 可手动触发 `Run Next Queued Job`

## 当前边界

这阶段是“可恢复运行时基础设施”的第一版，还没有做这些能力：

- Job 中断后恢复执行
- Artifact 的统一下载/打开入口
- 非文件类 Artifact（如结构化网页数据、API 响应对象）

说明：

- 轻量 `Job Queue` 已有，但仍是单进程、手动消费模型
- `Session History` 已有，但目前是浏览级页面，不是完整运维台

也就是说，第三阶段现在完成的是“先把对象立住”，还没有把这些对象全面运营起来。

## 数据落盘方式

当前持久化方式保持轻量：

- `data/runtime_state/sessions.json`
  - 保存 Session 的当前快照
- `data/runtime_state/jobs.jsonl`
  - 追加式记录 Job，并在收口时重写更新状态
- `data/runtime_state/artifacts.jsonl`
  - 保存已注册的 Artifact
- `data/runtime_state/job_queue.jsonl`
  - 保存队列状态（queued/running/completed/error/cancelled）

这种方式足够支撑当前单机单进程使用，也方便后面迁移到数据库。

## 对后续阶段的意义

这一阶段完成后，下一步就可以更稳地推进：

1. 把当前轻量 `Job Queue` 升级为真正后台 worker 模型
   - 让 `run_task()` 逐步退化为“提交入口”

2. 把 `Session History` 从浏览页升级成运维台
   - 支持筛选、恢复、重跑、按 Job 下钻

3. 增加更完整的 `Artifact Index`
   - 让 Agent 能基于历史产物继续工作，而不只是记住文本摘要

## 本次补充收口

在第三阶段第一版基础上，又补齐了这几项：

1. 后台队列 worker
   - `submit_task()` 现在默认会拉起后台队列线程
   - 新增 `start_background_worker()` / `stop_background_worker()` / `get_background_worker_status()`
   - 队列任务不再只能依赖手动点击 `Run Next Queued Job`

2. Session History 运维能力增强
   - 增加后台 worker 状态展示
   - 增加队列明细表
   - 增加 Job 状态筛选
   - 增加 Job 下钻详情
   - 增加 `Rerun Job` 和 `Resume Failed Job`

3. 历史 Artifact 参与规划
   - `core/runtime.py` 会把当前 Session 最近的 Artifact 注入到 `shared_memory`
   - `core/router.py` 会把这些 Artifact 摘要加入规划上下文
   - 这意味着规划器现在不仅看对话和向量记忆，也会看近期产物
