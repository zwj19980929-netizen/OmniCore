# OmniCore 通用 Agent 第二阶段落地说明

日期：2026-03-03

本文记录在第一阶段基础上继续推进的第二阶段改造，重点是让运行时更接近真正的 tool-first 执行模型。

## 已完成

1. Router 输出协议升级
   - `core/router.py` 现在会在系统提示里明确要求优先输出 `tool_name` 和 `tool_args`
   - 同时保留对旧 `task_type / params` 输出的兼容
   - Router 在解析结果后会做一次 shape 归一化，把旧格式补齐为 tool-first 结构

2. 权限规则继续下沉到工具执行层
   - `agents/file_worker.py` 在真正写文件前会依据 `requires_confirmation` 做确认
   - 这覆盖了 txt 以及结构化导出前的统一写入入口
   - `agents/system_worker.py` 在执行命令前增加了命令安全校验，并在任务执行入口再次执行确认

3. 风险边界进一步收紧
   - 系统命令会拦截常见 shell 控制符（如 `&&`、`||`、`;`、管道和重定向）
   - 文件写入即使经过规划层判断，也会在工具层再次确认

4. 并发调度开始使用工具元数据
   - `Tool Registry` 现在声明每个工具的并发上限与是否串行
   - 调度器会优先按工具元数据限制批次选择
   - 浏览器工具默认并发 2，系统工具默认串行

5. 工具适配器支持按模块动态加载
   - `core/tool_adapters.py` 会读取 `TOOL_ADAPTER_PLUGIN_MODULES`
   - 配置中的模块会在运行时按模块名导入
   - 外部模块只要在导入时用 `@tool_adapter(...)` 自注册，就能进入适配器注册表
   - 已加载模块与加载失败信息可在 `Session History` 页面查看
   - 现在也支持 `TOOL_ADAPTER_PLUGIN_DIRS`
   - 运行时会扫描目录中的 `.py` 文件并尝试导入
   - 这样外部插件既可以按模块名配置，也可以按目录批量发现

6. 插件现在可以自带 ToolSpec
   - `core/tool_registry.py` 新增了 `@tool_plugin(...)`
   - 插件模块在导入时可以同时完成：
   - `@tool_adapter(...)` 注册执行适配器
   - `@tool_plugin(...)` 注册工具定义
   - 这样插件不再只扩展执行层，也能自带可调度的工具能力
   - 插件还可以导出模块级 `PLUGIN_MANIFEST`
   - 用于声明 `plugin_id / version / description / dependencies / enabled`
   - `PLUGIN_MANIFEST` 现在会做基础校验：
   - `plugin_id` 必须是合法标识符
   - `version` 必须是可接受的版本字符串
   - `dependencies` 必须是合法依赖标识列表

7. ToolRegistry 会自动注入插件工具
   - 内建注册中心在构建和获取时，都会先触发插件加载
   - 已注册的插件工具会自动合并进 `ToolRegistry`
   - 不再需要手工去改 `build_builtin_tool_registry()`
   - `ENABLED_TOOL_PLUGIN_IDS` 和 `DISABLED_TOOL_PLUGIN_IDS` 会控制哪些插件工具真正进入运行时注册中心
   - 现在还会读取持久化的插件管理配置
   - 被 UI 安装的模块/目录源会自动参与加载
   - 被 UI 禁用或卸载的插件会被持久化过滤

8. Router 的可用工具列表改为从注册中心动态生成
   - `core/router.py` 不再写死 `tool_name` 列表
   - Router 会从当前 `ToolRegistry` 生成 `Registered Tool Catalog`
   - 插件工具只要完成注册，就会自动进入 Router 提示词

## 当前状态

运行时主路径已经是：

- Router 输出 tool-first 计划
- 任务规范化层补齐兼容字段
- TaskExecutor 通过 Tool Registry 做实际分发
- 工具层执行前再次做权限与边界控制

## 收口结果

第二阶段原本遗留的工程性尾巴已经清理完成：

1. `core/task_executor.py` 中原先保留的 `_execute_single_task_async_legacy` 历史快照已移除
   - 当前只保留新的 `_execute_single_task_async` 作为执行入口
   - 运行时分发路径已经完全收敛到 Tool Registry 主链路

2. `core/router.py` 的主 JSON 示例已改成 tool-first 结构
   - 示例会直接展示 `tool_name` 和 `tool_args`
   - 旧格式仍可被归一化兼容，但不再是主示例

## 如何添加新工具

当前版本接入一个新工具，建议按下面 4 步做。

1. 注册工具定义
   - 如果是内建工具，仍然可以直接在 `core/tool_registry.py` 注册 `RegisteredTool(...)`
   - 如果是插件工具，推荐在插件模块里直接使用 `@tool_plugin(...)`
   - 至少要定义这些字段：
   - `spec.name`
   - `spec.task_type`
   - `spec.description`
   - `spec.risk_level`
   - `adapter_name`
   - `max_parallelism`
   - `serialized`

2. 让规划层知道这个工具
   - 现在 Router 的工具清单来自 `ToolRegistry` 动态生成
   - 也就是说，插件工具只要完成注册，就会自动出现在 Router 提示词里
   - 只有当你想补充更具体的业务语义时，才需要再调整 `core/router.py` 的静态提示词说明
   - 如果 Router 仍输出旧格式，运行时会做兼容归一化，但最好直接输出 `tool_name / tool_args`

3. 在执行层接入对应适配器
   - 实际执行入口仍由 `core/task_executor.py` 统一调度，但具体适配器已经拆到 `core/tool_adapters.py`
   - 如果复用现有能力，可以直接使用已有 `adapter_name`
   - 如果是全新能力，通常需要：
   - 新建一个执行模块（例如新的 worker）
   - 在 `WorkerPool` 中补实例管理
   - 新增一个继承 `BaseToolAdapter` 的适配器类，并用 `@tool_adapter("your_adapter_name")` 自注册
   - `ToolAdapterRegistry` 会按 `adapter_name` 惰性实例化，不需要再改中央 `build_tool_adapter_registry()`
   - 如果希望做成外部插件，而不是仓库内直接导入：
   - 把模块路径加入环境变量 `TOOL_ADAPTER_PLUGIN_MODULES`
   - 模块在导入时完成 `@tool_adapter(...)` 自注册即可
   - 运行时会自动导入这些模块并记录加载状态

4. 在 `core/policy_engine.py` 增加确定性权限规则
   - 有写入、本地副作用、外部提交、系统操作的工具，必须在这里补规则
   - 不要只依赖 LLM 的高风险判断

### 最小接入清单

一个新工具最少需要同时经过这 4 个文件：

- `core/tool_registry.py`
- `core/router.py`
- `core/tool_adapters.py`
- `core/policy_engine.py`

如果是全新执行器，通常还会新增或修改：

- `agents/...`
- `WorkerPool`

### 示例思路

如果要加一个 `api.call` 工具，推荐做法是：

1. 新增一个专门的执行模块，例如 `agents/api_worker.py`
2. 在 `WorkerPool` 中补 `api_worker`
3. 在插件模块或仓库代码中新增 `ApiWorkerAdapter(BaseToolAdapter)`，并用 `@tool_adapter("api_worker")` 装饰
4. 用 `@tool_plugin(...)` 为 `api.call` 注册 `ToolSpec`
5. 如果作为外部插件接入：
   - 把模块加入 `TOOL_ADAPTER_PLUGIN_MODULES`
   - 或把目录加入 `TOOL_ADAPTER_PLUGIN_DIRS`
6. 在 `core/policy_engine.py` 里规定：
   - `GET` 默认低风险
   - `POST / PUT / DELETE` 默认需要确认

这样接入后，新工具就能同时进入：

- 规划
- 调度
- 执行
- 权限治理

## 审计与运维补充

1. `Session History` 页面现在有 `Policy Audit`
   - 会按当前 Session 汇总所有 job 的 `policy_decisions`
   - 支持按 `decision / risk_level / tool_name` 筛选
   - 支持查看 `pending / approved / rejected` 聚合计数
   - 会展示按 decision 和按 tool 的聚合表

2. `Session History` 页面现在也会显示 `Tool Adapter Plugins`
   - 可查看配置中的插件模块
   - 可查看配置中的插件目录
   - 可查看通过 UI 安装的插件模块/目录源
   - 可查看被禁用的插件 ID
   - 可查看被卸载后屏蔽的模块/文件
   - 可查看已成功加载的模块
   - 可查看按目录扫描加载的文件
   - 可查看插件 manifest（版本、依赖、启用状态）
   - 可查看加载失败的模块和错误信息
   - 可以直接在 UI 中：
   - 安装模块源
   - 安装目录源
   - 启用插件
   - 禁用插件
   - 卸载插件

3. 运行时现在会记录 checkpoint
   - `core/graph.py` 会在 `route / human_confirm / parallel_executor / validator / critic / replanner / finalize` 后落 checkpoint
   - `core/runtime.py` 也会在 graph 启动前落一个 `initial_state` checkpoint
   - `Session History` 页面可查看某个 job 的 checkpoint 列表
   - `Resume From Checkpoint` 会基于最新 checkpoint 继续跑后续流程
