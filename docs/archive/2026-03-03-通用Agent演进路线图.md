# OmniCore 通用 Agent 演进路线图

> 日期：2026-03-03
> 适用范围：OmniCore 当前代码库
> 文档目的：基于当前实现，明确 OmniCore 从“可执行任务的个人数字助理原型”演进为“面向个人用户的通用 Agent Runtime”的目标形态、架构方向、分阶段改造方案与落地顺序。

---

## 1. 背景与目标

### 1.1 当前产品定位

结合当前代码与产品目标，OmniCore 的现阶段定位可以概括为：

- 面向个人用户的 AI 数字助理原型
- 通过自然语言接收任务，并自动拆解、执行、校验、交付
- 重点覆盖网页获取信息、浏览器交互、文件生成、基础系统操作

这意味着，OmniCore 当前并不是一个单纯的聊天机器人，也不是一个单一垂直业务工具，而是一个正在形成中的“个人数字执行内核”。

### 1.2 最终目标

最终目标不是让系统只会执行几类固定任务，而是让它成为一个真正的“通用 Agent”：

- 用户像给员工下达工作一样下达目标
- 系统能够理解目标、制定计划、调用工具、执行任务、处理失败、交付结果
- 系统可以持续扩展能力，而不是每新增一个能力都要改一套主流程
- 系统具备长期记忆、权限边界、任务追踪和持续自治能力

简化定义如下：

> OmniCore 的终局形态应当是一个面向个人用户、可长期运行、可持续扩展、可授权执行的通用 Agent Runtime。

### 1.3 本文要解决的问题

本文不讨论单次功能修补，而重点回答以下问题：

1. 当前代码距离“通用 Agent”还有多远
2. 哪些现有模块值得保留，哪些应当重构
3. 通用 Agent 的核心架构应该如何分层
4. 应当按什么优先级推进，才能避免陷入只堆 Prompt 和 Worker 的局面

---

## 2. 当前系统现状评估

### 2.1 当前已经具备的能力

从代码实现看，OmniCore 已经具备通用 Agent 的部分基础设施：

- 自然语言任务入口：CLI 与 Streamlit UI
- 任务编排框架：基于 `LangGraph` 的状态机流转
- 意图分析与任务拆解：`RouterAgent`
- 多执行器模型：`WebWorker`、`BrowserAgent`、`FileWorker`、`SystemWorker`
- 失败反思重规划：`Replanner`
- 双层校验：`Validator` 与 `Critic`
- 简单长期记忆：`ChromaMemory`
- 多模型调用抽象：`LLMClient`、`ModelRegistry`
- 浏览器原子能力封装：`BrowserToolkit`

这些能力说明，当前系统已经跨过了“纯问答 AI”的阶段，具备了“行动系统”的初始骨架。

### 2.2 当前更像什么

尽管具备上述基础，当前系统更接近：

- 一个“任务编排器 + 几类固定 Worker + Prompt 驱动路由”的执行框架

而不是：

- 一个长期运行、可扩展、可治理的通用 Agent 平台

换句话说，当前更像“可执行的原型 Agent”，不是“成熟的通用 Agent Runtime”。

### 2.3 当前最明显的架构边界

现有代码的主链路是：

`用户输入 -> run_task -> Router -> HumanConfirm -> TaskExecutor -> Validator -> Critic -> Finalize`

这个链路对“单次任务执行”是成立的，但其边界也很明显：

- 以单次请求为中心，任务完成后上下文结束
- 工具类型由固定枚举定义，不适合无限扩展
- 权限控制主要依赖 LLM 判断，而非确定性策略
- 记忆能力以向量检索为主，缺少结构化状态
- 最终交付更偏执行状态回报，而非真正面向用户的结果交付

## 3. 与通用 Agent 目标之间的差距

### 3.1 持续自治能力不足

当前系统的执行入口是 `run_task()`，本质上是一次性调用：

- 用户发起请求
- 系统执行
- 返回结果
- 生命周期结束

这适合“单次代办”，但不适合真正的通用 Agent。通用 Agent 至少应支持：

- 可恢复会话
- 后台任务队列
- 长周期目标跟踪
- 定时或事件驱动执行
- 未完成事项的延续处理

当前这些能力基本不存在。

### 3.2 工具模型不够通用

当前系统把执行能力固化为 4 类任务类型：

- `web_worker`
- `browser_agent`
- `file_worker`
- `system_worker`

这在 MVP 阶段是合理的，但对通用 Agent 来说限制过强：

- 每加一种新能力，往往要新增一种 Worker 或扩展现有 Worker 的特殊分支
- 调度器理解的是“任务类型”，而不是“工具能力”
- 上层规划直接耦合具体实现类别，不利于长期扩展

通用 Agent 更合理的抽象应当是“工具协议”而不是“固定 Worker 类型”。

### 3.3 长期记忆不够结构化

当前 `ChromaMemory` 的主要能力是：

- 保存任务摘要
- 检索相似历史

这属于“语义记忆”的初级形态，但通用 Agent 还需要：

- 用户偏好与工作习惯
- 当前任务上下文
- 任务产物索引
- 长期目标状态
- 对外部实体和资源的结构化引用

如果只有向量检索，系统能“想起类似的事”，但很难“管理持续的工作状态”。

### 3.4 权限与安全模型过于松散

对通用 Agent 来说，最危险的问题不是“不会做事”，而是“误做事”。

当前系统已经有人工确认机制，但存在几个天然风险：

- 高风险判断主要来自 Router 的 LLM 输出
- 文件写入确认并未在所有写入路径中被严格执行
- 系统命令执行使用 `shell=True`
- 缺少稳定的权限分级和资源白名单

这意味着当前更像“有提醒的执行器”，而不是“有治理能力的可授权代理”。

### 3.5 结果交付层过弱

当前系统最终输出常常是：

- 哪些任务完成了
- 哪些任务失败了

这对于调试有价值，但对于用户而言不够像“员工交付结果”。真正的通用 Agent 应具备：

- 面向用户的结果总结
- 产物列表和引用
- 可追溯的执行摘要
- 失败的明确原因和下一步建议

否则系统做了很多事，但用户感知不到真正的业务价值。

## 4. 通用 Agent 的目标架构

### 4.1 总体架构原则

OmniCore 演进为通用 Agent 时，应遵循以下原则：

1. 保留现有可用能力，避免推倒重来
2. 把固定角色抽象成通用工具协议
3. 把单次任务执行升级为持续运行的 Runtime
4. 把 LLM 负责“理解与生成”，而不是承担全部治理逻辑
5. 把安全、权限、状态、交付都程序化，而不是纯 Prompt 化

### 4.2 建议的目标分层

建议将未来的 OmniCore 架构稳定为以下 6 层：

#### A. 交互层（Interaction Layer）

职责：

- 接收自然语言目标
- 显示执行状态
- 展示最终结果、产物与审计信息

典型形态：

- CLI
- Web UI
- 后续可扩展到桌面托盘、语音入口、IM 入口

#### B. Agent Runtime 层（Runtime Layer）

职责：

- 管理任务生命周期
- 管理状态机流转
- 承载计划、执行、重试、恢复
- 管理后台队列与会话恢复

这是未来最核心的一层，当前 `core/graph.py` 与 `core/runtime.py` 属于这层的雏形。

#### C. 规划与决策层（Planning Layer）

职责：

- 理解用户目标
- 生成任务计划
- 评估执行结果
- 在失败后调整策略

建议拆成三个角色，而不是继续把职责混在一起：

- `Goal Interpreter`：理解用户真实目标
- `Planner`：生成可执行计划
- `Replanner`：失败后调整路线

`Critic` 和 `Validator` 继续保留，但定位为“质量与规则守门层”。

#### D. 工具与执行层（Tool Execution Layer）

职责：

- 提供统一的工具协议
- 所有动作都通过标准接口调用
- 屏蔽底层差异（浏览器、文件、系统、API）

未来不应再以“固定 Worker 类型”为核心，而应变成：

- 工具注册中心
- 统一输入输出协议
- 工具能力元数据
- 风险等级声明

当前的 `WebWorker`、`BrowserAgent`、`FileWorker`、`SystemWorker` 应逐步退化为“能力提供者”，不再是调度主语。

#### E. 记忆与状态层（Memory Layer）

职责：

- 持久化会话
- 保存任务历史
- 保存用户偏好
- 保存产物引用
- 管理长期目标与外部实体

建议未来分为四类：

- 工作记忆：单次任务内的上下文
- 情景记忆：历史任务摘要与相似经验
- 用户记忆：偏好、习惯、长期约束
- 资源记忆：文件、网页、下载内容、截图、外部对象引用

#### F. 权限与治理层（Policy Layer）

职责：

- 决定是否允许执行
- 定义哪些动作需要确认
- 记录审计日志
- 控制高风险动作的白名单与审批

这一层必须尽可能程序化，不能主要依赖 LLM。

## 5. 推荐的核心抽象

### 5.1 从 TaskType 迁移到 Tool Protocol

当前系统是：

- 计划直接指定 `task_type`
- 执行器按 `task_type` 做分支判断

建议迁移为统一工具协议：

```python
class ToolSpec:
    name: str
    description: str
    input_schema: dict
    output_schema: dict
    risk_level: str
    tags: list[str]


class Tool:
    spec: ToolSpec

    def execute(self, args: dict, context: ToolContext) -> ToolResult:
        ...
```

这样做的好处：

- 新增能力时无需改主执行器逻辑
- 规划层可以基于工具能力做选择
- 权限层可以基于工具元数据做治理
- 审计层可以统一记录调用

### 5.2 增加会话与任务实体

当前状态以单次 `OmniCoreState` 为主，建议新增两个长期实体：

#### Session

表示一个持续会话，包含：

- 用户身份或设备身份
- 当前活跃目标
- 最近上下文
- 权限偏好
- 最近产物引用

#### Job

表示一次独立任务执行，包含：

- job_id
- 原始目标
- 计划版本
- 执行状态
- 产物
- 审计日志
- 恢复点

这样后续才能支持后台任务、失败恢复、分阶段继续执行。

### 5.3 增加 Artifact 抽象

当前大量结果只是塞进 `shared_memory`。这适合短生命周期，但不利于长期管理。

建议增加 `Artifact` 抽象，统一表示：

- 文件
- 网页结构化数据
- 截图
- 下载内容
- 报告
- 中间结果

推荐最小字段：

- artifact_id
- type
- path 或 uri
- producer_job_id
- created_at
- summary
- metadata

这样系统才能像员工一样“交付成果”，而不是只返回一句文本。

### 5.4 增加 Policy Decision 抽象

未来每个高风险动作都不应仅仅做“是否确认”的即时判断，而应形成标准化决策记录：

- action
- target_resource
- risk_level
- decision
- reason
- requires_human_confirm
- approved_by
- approved_at

这能让系统具备真正的治理能力和审计能力。

---

### 5.5 引入 Browser Runtime Pool 抽象

当前浏览器能力虽然已经由 `BrowserToolkit` 做了统一封装，但浏览器实例的生命周期仍偏向“按任务创建、按任务销毁”。

对于通用 Agent 来说，这种模式在功能上可用，但在运行效率上有明显损耗：

- Playwright 启动浏览器进程有固定冷启动开销
- 高频任务会重复创建和销毁浏览器实例
- 并发任务增多时，浏览器初始化成本会被放大

因此，建议在未来架构中引入 `Browser Runtime Pool`（浏览器运行时池）抽象。

建议的设计原则：

- 复用的是浏览器进程或浏览器实例，而不是直接复用页面上下文
- 每个任务仍应创建独立的 `BrowserContext`，避免会话污染
- 池化对象应支持按模式分组，例如 `headless`、`fast_mode`、`with_storage_state`
- 需要有最大并发上限、空闲回收、健康检查和异常重建机制

推荐最小职责：

- 获取可用浏览器实例
- 为任务创建隔离上下文
- 释放上下文并归还实例
- 对失效实例自动重建
- 记录浏览器资源占用与命中率

需要强调的边界：

- 不能为了复用而复用 `BrowserContext`
- 不能让不同任务默认共享登录态、Cookie 或本地存储
- 池化应提升性能，但不能破坏隔离性

正确的池化方式应是：

> 复用浏览器底层进程，隔离任务级上下文。

### 5.6 引入 LLM Cache 抽象

当前系统中有一类非常典型的重复性 LLM 调用：

- URL 分析
- 页面结构分析
- 数据质量判断
- 某些重复的结果审查

其中最适合优先缓存的是：

- 对相同目标页面的 URL 推断结果
- 对相同页面结构的选择器分析结果

这是因为这些调用往往具有以下特点：

- 语义重复度高
- 成本高于普通程序逻辑
- 在短时间内结果稳定

因此，建议引入 `LLM Cache` 抽象，将“高重复、低时效敏感”的分析结果缓存起来。

推荐缓存范围：

- `determine_target_url()` 的分析结果
- `analyze_page_structure()` 的页面结构配置结果
- 后续也可逐步扩展到部分低风险、稳定的分析型调用

推荐缓存键设计：

- 不能只使用 URL 作为键
- 应至少包含：规范化 URL、任务意图摘要、页面指纹、Prompt 版本、模型标识

推荐的缓存键构成示例：

- `normalized_url`
- `task_signature`
- `page_fingerprint`
- `prompt_version`
- `model_name`

其中：

- `task_signature` 用于区分同一 URL 下不同业务目标
- `page_fingerprint` 用于识别页面结构是否变化
- `prompt_version` 用于在 Prompt 升级后自动失效旧缓存

推荐失效策略：

- 基于 TTL 的自动过期
- 基于页面结构变化的失效
- 基于 Prompt 版本变化的失效
- 支持手动清理和按域名清理

需要强调的边界：

- 不能对所有 LLM 调用一刀切缓存
- 不能仅按 URL 粗暴命中，否则极易产生错误复用
- 缓存应优先用于“分析结果”，而不是“最终结论”

正确的方向应是：

> 缓存稳定的中间分析结果，而不是缓存所有模型输出。

## 6. 现有模块的保留与重构建议

### 6.1 建议保留并继续演进的模块

#### `utils/browser_toolkit.py`

这是当前最有平台价值的基础设施之一：

- 抽象层级对
- 原子操作完整
- 屏蔽了 Playwright 细节
- 已经接近“通用浏览器工具层”

建议继续沉淀，把它作为未来 Web 工具族的底座。

同时建议在其外围新增一个浏览器池管理层，而不是把池化逻辑继续塞进工具本体。更合适的结构是：

- `BrowserToolkit` 负责单个浏览器上下文内的原子操作
- `BrowserRuntimePool` 负责浏览器实例复用、隔离与资源治理

#### `core/task_executor.py`

这个模块已有明显的平台潜力：

- 支持依赖判断
- 支持 ready task 批量调度
- 支持并行执行
- 支持结果写回共享上下文

建议保留其调度思想，但从“按固定 Worker 分派”升级为“按 Tool Registry 分派”。

#### `core/graph.py` 与 `core/state.py`

当前状态机方向是对的：

- 有清晰节点
- 有失败重规划
- 有校验与最终收尾

建议保留整体思路，但要继续扩展持久化状态、后台队列、恢复点和会话实体。

#### `core/llm.py`、`core/model_registry.py`

多模型抽象是通用 Agent 的必要基础能力，建议继续保留。

#### `memory/chroma_store.py`

适合作为第一阶段的情景记忆层保留，但后续不能只停留在这一种记忆形态。

### 6.2 建议优先重构的模块

#### `core/router.py`

当前是整个系统最关键、也最脆弱的模块之一。

主要问题：

- 由 LLM 直接产出执行计划，稳定性不够高
- 输出结果与底层执行器类型强耦合
- 风险判定也绑定在 LLM 输出里

建议演进为：

- LLM 生成计划草案
- 程序化计划修正与校验
- 工具选择改为基于 `ToolSpec`

#### `agents/system_worker.py`

这是当前风险最高的执行模块。

主要问题：

- 直接 `shell=True`
- 命令与权限边界不够硬
- 缺少资源白名单与命令策略

建议优先加：

- 命令白名单
- 参数级校验
- 更细的风险分级
- 审计记录

#### `agents/file_worker.py`

当前主要问题：

- 写入确认未在所有路径中强制一致
- 文件输出更像“执行工具”，还不是“产物管理器”

建议未来将其一部分职责迁入 `Artifact Store`，把写文件和管理产物分开。

#### `utils/human_confirm.py`

当前只是一个确认交互组件，不是完整的策略层。

建议未来让它退化成“策略执行器的一个交互终端”，而不是由业务代码直接调用。

## 7. 分阶段演进路线图

### Phase 0：从原型走向可持续演进的底座

这是最关键的阶段，目标不是“新增更多功能”，而是把架构打稳。

#### 目标

- 建立统一工具协议
- 从固定 Worker 类型迁移到 Tool Registry
- 建立确定性权限策略
- 让任务结果有标准化交付形式

#### 必做事项

1. 引入 `ToolSpec`、`Tool`、`ToolResult`、`ToolContext`
2. 建立 `ToolRegistry`
3. 把现有四类 Worker 先包装成四组 Tool
4. 修改 Planner 输出，改为工具调用计划而不是固定 `task_type`
5. 新增 `Artifact` 与 `Job` 基础结构
6. 将高风险策略从 Router Prompt 中部分下沉到规则层

7. 引入 `BrowserRuntimePool`，减少浏览器冷启动成本
8. 引入 `LLM Cache`，优先缓存 URL 分析与页面结构分析结果

补充说明：Phase 0 中第 7 项为 `BrowserRuntimePool`，第 8 项为 `LLM Cache`。两者都是性能优化项，但前提是不破坏隔离性与正确性。

#### 完成标志

- 新增一个工具不需要修改 `task_executor` 的分发逻辑
- 计划层不再直接依赖 `TaskType` 枚举
- 高风险操作不再只靠 LLM 标记

- 浏览器高频任务不再为每次执行都承担完整冷启动成本
- 相同页面的重复分析型 LLM 调用显著下降

### Phase 1：从“单次任务”升级为“持续 Agent Runtime”

#### 目标

- 引入持久化任务队列
- 支持后台执行与恢复
- 支持会话级状态管理

#### 必做事项

1. 增加 `Session` 与 `Job` 持久化
2. 支持任务断点恢复
3. 支持前台提交、后台执行、后续查询进度
4. 增加统一执行日志和审计记录
5. 对 `run_task()` 做兼容封装，使其变为提交入口而不是唯一执行形态

#### 完成标志

- 用户可以提交任务后离开，再回来查看结果
- 系统可以恢复上一次未完成的任务

### Phase 2：增强记忆与交付能力

#### 目标

- 让系统具备更像“员工”的持续工作体验

#### 必做事项

1. 将记忆拆分为工作记忆、情景记忆、用户记忆、资源记忆
2. 引入 `Artifact Store`
3. 为最终输出增加“结果总结 + 产物清单 + 执行摘要”
4. 建立用户偏好和常用任务模板
5. 将相似历史任务的成功路径作为规划参考

#### 完成标志

- 系统能引用上次生成的文件和结果
- 系统能在交付时明确指出做了什么、生成了什么、在哪里

### Phase 3：增强通用性与自治性

#### 目标

- 让 OmniCore 具备更强的广泛适配能力与主动执行能力

#### 可做事项

1. 插件化第三方工具接入
2. 定时任务与事件触发
3. 长期目标管理
4. 更强的外部世界模型
5. 主动提醒和建议

这一阶段不建议过早投入。在 P0 和 P1 没稳定前，继续扩展通用性只会放大架构负担。

## 8. 推荐的代码迁移顺序

### 8.1 第一批改造（优先）

建议先改造以下部分：

1. 新增 `core/tools/` 目录，定义统一工具协议
2. 将 `web_worker`、`browser_agent`、`file_worker`、`system_worker` 包装为工具适配器
3. 修改 `core/task_executor.py`，让它依赖 `ToolRegistry` 而不是 `TaskType`
4. 修改 `core/router.py`，让计划输出改为工具调用计划

原因：

- 这是最小但最关键的架构转向
- 不会直接破坏已有底层能力
- 能为后续所有扩展打基础

### 8.2 第二批改造（紧随其后）

1. 在 `core/state.py` 中扩展 `job_id`、`session_id`、`artifacts`、`policy_decisions`
2. 在 `core/graph.py` 中将最终收尾改为真正的交付层
3. 把 `HumanConfirm` 从直接调用式组件迁移为策略层的一部分
4. 为系统命令和文件写入增加确定性规则

### 8.3 第三批改造（稳定期）

1. 引入持久化任务队列
2. 引入 Artifact Store
3. 引入 Session Store
4. 重构 `run_task()` 为兼容入口，而不是唯一工作模式

## 9. 关键风险与避免方式

### 9.1 不要继续堆更多 Prompt

如果继续沿当前思路，只是在 Router、Critic、Replanner 上叠更多 Prompt，短期看会“更聪明”，长期会出现：

- 规划不稳定
- 扩展成本越来越高
- 行为难以治理
- 调试越来越困难

结论：

- Prompt 仍然重要
- 但 Prompt 应负责“理解和生成”
- 不应承担“系统治理”和“架构扩展”的主要职责

### 9.2 不要把更多能力继续塞进现有 Worker

继续往 `WebWorker` 或 `SystemWorker` 里堆更多分支，会导致：

- 单模块职责膨胀
- 调度层更难抽象
- 后续工具化迁移更难

正确方向是：

- 保留已有能力实现
- 逐步把能力拆成可注册工具

### 9.3 不要过早追求“多 Agent 协作”

当前阶段最容易让系统看起来更高级的做法，是继续加更多 Agent 角色。

但对于当前代码库，真正的瓶颈不是“Agent 数量”，而是：

- 工具协议还没统一
- 状态还不持久
- 权限还不够硬
- 交付还不够完整

在这些问题未解决前，多 Agent 协作只会放大复杂度。

---

### 9.4 不要用错误粒度做缓存和池化

浏览器池和 LLM 缓存在方向上是正确的，但如果实现粒度错误，会直接引入隐蔽问题。

典型错误包括：

- 复用 `BrowserContext` 导致任务之间共享登录态
- 不区分任务意图，只按 URL 命中页面分析缓存
- Prompt 升级后仍继续复用旧缓存
- 页面结构已变化，但缓存仍强制命中

因此需要明确：

- 浏览器池复用的是底层浏览器实例，不是任务上下文
- LLM 缓存命中的前提是“页面结构和任务语义都足够接近”
- 所有缓存都必须具备可失效、可观测、可手动清理的能力

如果这三个条件做不到，宁可先不缓存，也不要引入错误缓存。

## 10. 里程碑与验收标准

### 10.1 P0 验收标准

- 新工具接入无需修改核心分发逻辑
- Planner 输出不再直接绑定 `TaskType`
- 至少对系统命令、文件写入建立确定性风控规则
- 最终输出包含结构化结果与产物信息

- 浏览器池命中后，重复 Web 任务的平均冷启动时延明显下降
- 相同页面分析场景下，分析型 LLM 调用次数明显下降，且缓存错误命中可控

### 10.2 P1 验收标准

- 支持后台任务队列
- 支持任务恢复
- 支持会话级上下文
- 用户可查询历史任务及状态

### 10.3 P2 验收标准

- 支持结构化长期记忆
- 支持跨任务复用产物
- 系统能基于历史经验优化规划

## 11. 对当前 OmniCore 的结论

从产品方向看，OmniCore 走在正确的路上：

- 它不是普通聊天系统
- 它已经具备行动能力
- 它有明确的“个人数字员工”与“通用 Agent”潜力

但从系统形态看，它仍然处于“通用 Agent 的原型期”，主要原因在于：

- 执行能力已经出现，但统一工具层还未成型
- 编排能力已经出现，但持续 Runtime 还未成型
- 记忆能力已经出现，但结构化状态还未成型
- 风险控制已经出现，但治理体系还未成型

因此，接下来最重要的工作不是继续增加零散能力，而是完成一次架构层的“抽象升级”：

> 从“按固定 Worker 执行任务的智能编排器”，升级为“按统一工具协议运行的通用 Agent Runtime”。

这一步一旦完成，OmniCore 才真正具备向“通用 Agent”持续演进的基础。

---

## 12. 建议的下一步动作

建议按以下顺序推进后续工作：

1. 先定义统一 Tool Protocol，并确定如何映射现有四类 Worker
2. 再重构 `task_executor` 和 `router` 的计划与分发方式
3. 随后引入 `Job`、`Session`、`Artifact` 三个长期实体
4. 最后再做后台任务队列、长期记忆和更强自治能力

如果要控制风险，推荐遵循一个原则：

> 先统一抽象，再扩展能力；先做治理底座，再追求更强智能。
