# 代码细读：规划、任务、代理协作

## 先给结论

很多 agent 会“说自己有计划”，但实际上只是输出一段步骤列表。这个仓库不是。它把规划、执行拆解、代理协作做成了正式运行时对象：

- Plan Mode
- Plan 文件
- ExitPlan 审批
- Todo / Task
- Forked subagent
- Background task

所以它真正做的是 orchestration，不是聊天里临时列提纲。

---

## 1. 规划入口：`EnterPlanModeTool`

主要代码在 `tools/EnterPlanModeTool/EnterPlanModeTool.ts`。核心逻辑其实很干净：

```ts
const EnterPlanModeTool = buildTool({
  name: 'EnterPlanMode',
  shouldDefer: true,
  isReadOnly: true,

  async call(_input, context) {
    if (context.agentId) {
      throw new Error('cannot be used in agent contexts')
    }

    appState = context.getAppState()
    handlePlanModeTransition(appState.toolPermissionContext.mode, 'plan')

    context.setAppState(prev => ({
      ...prev,
      toolPermissionContext: applyPermissionUpdate(
        prepareContextForPlanMode(prev.toolPermissionContext),
        { type: 'setMode', mode: 'plan', destination: 'session' },
      ),
    }))

    return {
      message: 'Entered plan mode. focus on exploring and designing.'
    }
  }
})
```

这段代码特别重要，因为它说明 Plan Mode 不是 prompt 风格，而是 app state 切换。

进入 Plan Mode 后改变的不只是提示词，还有：

- permission mode
- 后续 attachment 提醒逻辑
- ExitPlan 工具的可用性和语义

这意味着规划在这里是一个正式状态机。

---

## 2. 为什么 `EnterPlanModeTool` 的 prompt 很关键

这个工具不仅切 mode，还会把行为准则塞回 `tool_result`：

```text
In plan mode:
1. explore codebase
2. identify patterns
3. compare approaches
4. ask clarification if needed
5. design implementation strategy
6. do not edit files yet
```

也就是说：

- mode 切换负责硬状态
- tool_result 负责给当前轮一个强约束

这是 runtime 约束和 prompt 约束并用的典型做法。

## 3. `plans.ts`：计划不是消息，而是文件资产

`utils/plans.ts` 这一层非常重要。核心函数包括：

- `getPlanSlug()`
- `getPlansDirectory()`
- `getPlanFilePath()`
- `getPlan()`
- `copyPlanForResume()`
- `copyPlanForFork()`
- `recoverPlanFromMessages()`

### 3.1 计划文件路径如何确定

```ts
function getPlanFilePath(agentId?) {
  planSlug = getPlanSlug(getSessionId())

  if (!agentId) {
    return join(getPlansDirectory(), `${planSlug}.md`)
  }

  return join(getPlansDirectory(), `${planSlug}-agent-${agentId}.md`)
}
```

这里已经体现出一个很成熟的想法：

- 主会话的计划是主计划
- 子代理也可以有自己的计划文件

### 3.2 为什么 `getPlansDirectory()` 值得看

它不仅是 mkdir，还做了路径防护：

```ts
function getPlansDirectory() {
  settingsDir = settings.plansDirectory
  resolved = resolve(cwd, settingsDir)

  if (resolved escapes project root) {
    fallback to configHome/plans
  }

  mkdirSync(plansPath)
  return plansPath
}
```

也就是说，连计划文件目录都考虑了 path traversal 风险。这种细节表明它不是 demo。

## 4. `copyPlanForResume()` 说明计划是可恢复工件

这段逻辑很能体现工程成熟度：

```ts
async function copyPlanForResume(log, targetSessionId) {
  slug = getSlugFromLog(log)
  if (!slug) return false

  setPlanSlug(targetSessionId, slug)

  if (planFile exists on disk) return true

  if (remote environment only) {
    recovered = fileSnapshotEntry(log.messages, 'plan')
      ?? recoverPlanFromMessages(log)

    if (recovered) {
      writeFile(planPath, recovered)
      return true
    }
  }

  return false
}
```

关键点：

- 恢复优先读磁盘
- 磁盘没有，再从 file snapshot 恢复
- 再不行，从消息历史恢复

这说明“计划”已经被当成正式运行资产，而不是一段 ephemeral 文本。

## 5. `recoverPlanFromMessages()` 很值得抄

它会从三类地方回捞计划：

1. `ExitPlanMode` tool_use 里的输入
2. 用户消息里的 `planContent`
3. `plan_file_reference` attachment

伪代码大概是：

```ts
function recoverPlanFromMessages(log) {
  for (msg from end to start) {
    if (assistant tool_use == ExitPlanMode && input.plan exists) {
      return input.plan
    }
    if (user message has planContent) {
      return planContent
    }
    if (attachment.type == 'plan_file_reference') {
      return attachment.content
    }
  }
  return null
}
```

这段逻辑体现的不是技巧，而是“计划在多个边界都被显式保留”。

## 6. `ExitPlanModeV2Tool` 是规划态到执行态的闸门

主要逻辑在 `tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts`。它大致可以拆成三步。

### 6.1 校验是否真的处于 plan mode

```ts
validateInput() {
  if (isTeammate()) return true
  if (currentMode !== 'plan') {
    return false with message('you are not in plan mode')
  }
}
```

这意味着 ExitPlan 不是普通命令，而是 mode-sensitive transition。

### 6.2 权限策略

```ts
checkPermissions(input, context) {
  if (isTeammate()) {
    return allow
  }
  return ask("Exit plan mode?")
}
```

也就是说：

- teammate 路径走邮箱/leader 审批流
- 普通用户路径走本地确认流

### 6.3 核心 `call()` 流程

可以抽成下面这段：

```ts
async function call(input, context) {
  filePath = getPlanFilePath(context.agentId)
  inputPlan = input.plan ?? undefined
  plan = inputPlan ?? getPlan(context.agentId)

  if (inputPlan exists) {
    writeFile(filePath, inputPlan)
    persistFileSnapshotIfRemote()
  }

  if (isTeammate() && isPlanModeRequired()) {
    requestId = generateRequestId(...)
    approvalRequest = {
      type: 'plan_approval_request',
      planFilePath: filePath,
      planContent: plan,
      requestId,
    }
    writeToMailbox('team-lead', approvalRequest)
    markAwaitingPlanApproval()
    return { awaitingLeaderApproval: true, requestId, plan }
  }

  context.setAppState(prev => {
    restoreMode = prev.toolPermissionContext.prePlanMode ?? 'default'
    if (restoreMode === 'auto' and autoGateOff) {
      restoreMode = 'default'
    }

    if (restoringToAuto) {
      baseContext = stripDangerousPermissionsForAutoMode(baseContext)
    } else if (strippedDangerousRules) {
      baseContext = restoreDangerousPermissions(baseContext)
    }

    return {
      ...prev,
      toolPermissionContext: {
        ...baseContext,
        mode: restoreMode,
        prePlanMode: undefined,
      },
    }
  })

  return { plan, filePath, hasTaskTool }
}
```

这里有几个很强的点：

- 退出时会把 plan 写回磁盘
- teammate 可以发 plan approval request 给 leader
- 退出时会恢复 prePlanMode
- 如果计划前是 auto mode，还会重新处理危险权限剥离/恢复

也就是说，它不是简单地把 `mode = default`，而是把整个 permission context 一起迁移。

## 7. `attachments.ts` 中的 plan reminders 才是维持规划状态的关键

只看 Enter/ExitPlan 还不够，真正让这个系统稳定的是中间的提醒逻辑：

- `getPlanModeAttachments()`
- `getPlanModeExitAttachment()`
- `getPlanModeAttachmentTurnCount()`
- `countPlanModeAttachmentsSinceLastExit()`

可以把它理解成：

```ts
if (currently in plan mode && many turns passed) {
  inject attachment reminding:
    - still planning
    - should keep plan updated
    - maybe time to exit with approval
}

if (recently exited plan mode) {
  inject sparse reminder:
    - plan already exists
    - continue implementation
}
```

这是 runtime 主动纠偏，而不是指望模型自律。

## 8. `TodoWriteTool`：轻量执行拆解器

这部分代码很实用，而且体现了产品经验。

```ts
async function TodoWriteTool.call({ todos }, context) {
  todoKey = context.agentId ?? getSessionId()
  oldTodos = appState.todos[todoKey] ?? []
  allDone = todos.every(t => t.status === 'completed')
  newTodos = allDone ? [] : todos

  verificationNudgeNeeded =
    VERIFICATION_AGENT enabled &&
    todos.length >= 3 &&
    !todos.some(t => /verif/i.test(t.content))

  saveTodos(todoKey, newTodos)

  return {
    oldTodos,
    newTodos: todos,
    verificationNudgeNeeded,
  }
}
```

最强的细节是 `verificationNudgeNeeded`。如果你一次性关了 3 个以上任务，但没有 verification step，系统会明确 nudging：

```text
spawn verification agent before final summary
```

这说明 todo 系统不是记账工具，而是执行质量控制器。

## 9. `TaskCreateTool` / `TaskUpdateTool`：正式任务对象

### 9.1 创建任务

从 `TaskCreateTool.ts` 可以抽出：

```ts
createTask({
  subject,
  description,
  status: 'pending',
  owner: undefined,
  blocks: [],
  blockedBy: [],
  metadata,
})
executeTaskCreatedHooks(...)
```

### 9.2 更新任务

`TaskUpdateTool.ts` 更丰富，支持：

- `subject`
- `status`
- `owner`
- `metadata`
- `addBlocks`
- `addBlockedBy`

可以抽成：

```ts
updateTask(id, patch) {
  if (subject changed) updates.subject = subject
  if (owner changed) updates.owner = owner
  if (metadata provided) mergeMetadata()
  if (status changed) {
    if (status === 'completed') runTaskCompletedHooks()
    updates.status = status
  }
  if (addBlocks) addDependencyEdges()
  if (addBlockedBy) addReverseDependencyEdges()
  if (ownership changed) maybeNotifyMailbox(newOwner)
}
```

这层的意义在于：

- Todo 解决“我现在要做哪几步”
- Task 解决“这件事是谁负责、依赖谁、状态是什么”

它开始具备小型任务图谱的味道了。

## 10. `createSubagentContext()`：代理协作为什么不会太乱

这段代码在 `utils/forkedAgent.ts`，值得认真看。它的本质是“克制地复制 parent context”。

伪代码几乎可以直接写成：

```ts
function createSubagentContext(parentContext, overrides?) {
  abortController =
    overrides.abortController
    ?? (overrides.shareAbortController
        ? parentContext.abortController
        : createChildAbortController(parentContext.abortController))

  getAppState =
    overrides.getAppState
    ?? (overrides.shareAbortController
        ? parentContext.getAppState
        : wrapGetAppStateToForceAvoidPermissionPrompts(parentContext))

  return {
    readFileState: cloneFileStateCache(overrides.readFileState ?? parentContext.readFileState),
    nestedMemoryAttachmentTriggers: new Set(),
    loadedNestedMemoryPaths: new Set(),
    dynamicSkillDirTriggers: new Set(),
    discoveredSkillNames: new Set(),
    contentReplacementState: cloneOrOverride(parentContext.contentReplacementState),
    abortController,
    getAppState,
    setAppState: overrides.shareSetAppState ? parentContext.setAppState : noOp,
    setAppStateForTasks: parentContext.setAppStateForTasks ?? parentContext.setAppState,
    localDenialTracking: sharedOrFresh(),
    setResponseLength: sharedOrNoOp(),
    updateAttributionState: parentContext.updateAttributionState,
    options: overrides.options ?? parentContext.options,
    messages: overrides.messages ?? parentContext.messages,
    agentId: overrides.agentId ?? createAgentId(),
    queryTracking: {
      chainId: randomUUID(),
      depth: parentContext.queryTracking.depth + 1,
    },
  }
}
```

这段实现告诉我们：

- 读文件状态会继承，但以 clone 形式隔离
- discovered skills 也继承同类结构
- 默认避免子代理乱弹权限 UI
- `setAppState` 默认 no-op，防止随便改父状态

这就是为什么它能开子代理，又不至于把主状态搞乱。

## 11. `buildForkedMessages()`：fork 复制的是执行现场

`tools/AgentTool/forkSubagent.ts` 很关键。它的目的不是“再开一个聊天”，而是“把父 agent 当前现场，按缓存友好方式复制给子 agent”。

核心伪代码：

```ts
function buildForkedMessages(directive, assistantMessage) {
  fullAssistantMessage = cloneAssistantMessageWithAllToolUses()
  toolUseBlocks = extractAllToolUseBlocks(assistantMessage)

  toolResultBlocks = toolUseBlocks.map(block => ({
    type: 'tool_result',
    tool_use_id: block.id,
    content: 'Fork started - processing in background',
  }))

  toolResultMessage = createUserMessage({
    content: [
      ...toolResultBlocks,
      { type: 'text', text: buildChildMessage(directive) },
    ]
  })

  return [fullAssistantMessage, toolResultMessage]
}
```

这里很妙的一点是：

- 所有 fork child 尽量共享字节级一致前缀
- 只有最后一小段 directive 不同

这就是在为 prompt cache 做工程设计。

## 13. 为什么这套规划体系比常见 agent 强

常见 agent：

- 计划只是说说
- 中途容易忘
- 子代理是裸 prompt
- 没有真正的审批边界

这个仓库：

- 有 mode 切换
- 有 plan 文件
- 有 plan 恢复
- 有 plan 审批
- 有 todo / task
- 有 fork child 隔离上下文

所以它能承接更长、更复杂、更多人协作式的任务。
