# 规划、任务与代理协作

## 核心判断

这个仓库里最不像“普通聊天式 agent”的地方，就是它把规划、任务管理、代理协作做成了正式运行时，而不是让模型在回答里口头列几个步骤。

关键代码主要在：

- `src/tools/EnterPlanModeTool/EnterPlanModeTool.ts`
- `src/tools/EnterPlanModeTool/prompt.ts`
- `src/tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts`
- `src/utils/plans.ts`
- `src/commands/plan/plan.tsx`
- `src/tools/TodoWriteTool/TodoWriteTool.ts`
- `src/tools/TaskCreateTool/TaskCreateTool.ts`
- `src/tools/TaskUpdateTool/TaskUpdateTool.ts`
- `src/utils/attachments.ts`
- `src/tools/AgentTool/AgentTool.tsx`
- `src/tools/AgentTool/runAgent.ts`
- `src/tools/AgentTool/forkSubagent.ts`
- `src/utils/forkedAgent.ts`
- `src/tasks/LocalMainSessionTask.ts`

这一层最核心的意义是：

> 它把“计划”从语言输出升级成了状态、文件、工具和审批流。

## 1. Plan Mode 不是一句提示语，而是明确的模式切换

`EnterPlanModeTool.ts` 说明 Plan Mode 是一等机制，不是模型随口说一句“我先规划一下”。

进入 Plan Mode 后，系统会发生几类变化：

- 模式上下文改变
- 权限语义改变
- 任务目标变成先探索、先设计、先制定方案
- 后续执行需要通过明确退出来衔接

这种设计非常关键。很多系统嘴上支持 planning，但执行层没有模式边界，结果“规划”和“修改代码”混在同一轮里，风险很高。

这里的做法是先把规划态单独显式化。

## 2. 计划被持久化成文件，而不是只存在上下文里

`utils/plans.ts` 很值得看。它会：

- 生成 plan slug
- 为 session 或 agent 创建 plan file
- 支持 resume / fork 时复制和恢复
- 从 transcript / tool input / attachments 恢复 plan

这有三个很强的含义：

### 2.1 计划是可恢复资产

计划不会因为一次 compact 或一次上下文切换就丢失。

### 2.2 计划可以跨 agent 传递

fork、resume、leader/subagent 协作时，计划不是纯口头信息，而是可复制的工件。

### 2.3 计划能成为后续执行的锚点

退出 Plan Mode、继续执行、恢复会话时，系统都可以围绕这个 plan file 继续。

## 3. Exit Plan Mode 体现了“计划到执行”的审批边界

`ExitPlanModeV2Tool.ts` 里最成熟的一点，是它不把退出规划态当成普通文本动作，而是一个受控转移：

- 可以从磁盘读取 plan
- 可以从当前输入接收 plan
- 退出时会写回 plan file
- 对非 teammate 场景需要审批
- teammate 路径还可以请求 leader approval

这背后的工程理念很强：

- 规划态和执行态不应无缝混同
- 从“想做什么”转入“开始做”时，应该有明确门槛
- 多代理协作时，计划要能被上级审阅

这在很多 agent 系统里是完全缺失的。

## 4. `/plan` 命令只是入口，真正的重点是计划被系统持续提醒

`commands/plan/plan.tsx` 只是本地入口。更关键的是 `attachments.ts` 中的：

- `plan_mode`
- `plan_mode_exit`
- `plan_mode_reentry`

这些附件提醒说明，系统不会把 plan mode 当一次性操作，而是持续追踪当前会话是否仍处于：

- 应该继续规划
- 应该退出规划
- 需要重新进入规划

也就是说，计划是持续状态，不是瞬时动作。

## 5. TodoWriteTool：把“执行分解”做成强约束

`TodoWriteTool.ts` 是这个系统很实用的一层。

它不只是让模型列个 checklist，而是提供一个正式工具去维护 todo 状态。更重要的是，它包含一些工程化行为：

- 全部 done 时可以清空
- 如果任务较多却没有 verification step，会提示补充验证代理

这个细节很强。说明项目组已经意识到：

- 复杂任务不只需要分步
- 还需要强制考虑验证闭环

所以 todo 在这里不是表演性输出，而是执行控制器。

## 6. TaskCreate / TaskUpdate：比 Todo 更正式的任务图谱

`TaskCreateTool.ts` 和 `TaskUpdateTool.ts` 进一步把任务结构化，支持：

- id
- subject
- description
- owner
- status
- blocks / blockedBy
- metadata
- hooks

这说明系统实际上支持两层任务抽象：

- Todo：轻量执行清单
- Task：正式任务对象

这比很多 agent 只会维护一个 markdown checklist 高一个层级，因为它开始具备：

- 依赖表达
- 所有权表达
- 状态转换
- 生命周期管理

## 7. Reminder Attachments：系统会主动纠偏，而不是等模型自觉

`attachments.ts` 中 todo/task reminder 的意义很大。很多 agent 框架把 todo 系统做出来后，仍然依赖模型“自觉更新”。这个项目不是，它会基于 turn 数和状态主动插 reminder。

这背后是很现实的认知：

- 模型会忘
- 模型会偷懒
- 模型会跳步

所以必须由 runtime 定期提醒它维护计划和任务状态。

这其实是把 agent 从“聪明助手”往“受控执行系统”推进了一步。

## 8. Agent Tool：子代理不是玩具能力，而是正式执行单元

`AgentTool.tsx`、`prompt.ts`、`runAgent.ts`、`forkSubagent.ts`、`utils/forkedAgent.ts` 共同构成代理协作层。

它支持的维度包括：

- description
- prompt
- subagent_type
- model
- run_in_background
- isolation
- cwd

这里真正的亮点不是“能开子 agent”，而是它把子 agent 当成受控资源来设计。

## 9. Forked Agent：上下文继承非常克制，也非常关键

`createSubagentContext()` 会继承一些关键状态：

- readFileState
- contentReplacementState
- nested memory sets
- discovered skills

同时又做一些隔离：

- `setAppState` 为空实现
- 默认 `shouldAvoidPermissionPrompts`
- 可以控制是否 background run

这说明它不是简单复制整个进程，而是在复制对当前任务最关键的上下文事实。

这特别重要，因为子代理如果完全没有上下文，会浪费大量时间重新探索；如果继承过多，又会带来状态污染和权限问题。这里的平衡做得很好。

## 10. Fork 模式的真正价值：复制的是“执行现场”，不是聊天记录

`forkSubagent.ts` 会尽量构造字节级一致的 cache-safe prefix，并要求 fork worker 不要继续无限委托。

这背后有两个核心目标：

- 让子代理共享高价值上下文前缀，提高缓存命中和理解一致性
- 控制代理树不要无限扩张

这比普通“spawn another model call”成熟很多。它更像是在复制一个工作现场。

## 11. 后台任务：主会话也被任务化了

`LocalMainSessionTask.ts` 说明一个很强的方向：连主会话查询都可以包装为 task，并拥有：

- sidechain transcript
- notification
- 生命周期管理

这意味着系统未来完全可以把：

- 前台交互
- 后台长任务
- 子代理异步任务

统一收敛到同一种任务运行时上。这是很有平台潜力的架构。

## 12. 这套规划体系和常见做法有什么本质不同

常见做法：

- 模型回复里先写“计划如下”
- 然后自己边做边忘
- 中途没有正式状态机
- 子代理只是辅助 prompt

这个仓库的做法：

- 有显式 Plan Mode
- 有 Plan 文件
- 有 Exit 审批边界
- 有 Todo / Task 两层执行结构
- 有 reminder 机制纠偏
- 有 forked subagent 的上下文继承
- 有后台 task runtime

也就是说，它把 planning 变成了 orchestration。

## 13. 这层设计的核心亮点

### 亮点一：计划被持久化和审批化

计划不是聊完就算，而是正式工件，可以继续、恢复、审阅、转交。

### 亮点二：任务拆解不仅服务表达，更服务控制

Todo 和 Task 系统不是为了看起来更有条理，而是为了约束 agent 的执行路径。

### 亮点三：代理协作建立在共享状态和受控权限之上

forked agent 真正解决的是多代理协作时的上下文复用和权限边界问题。

### 亮点四：系统会主动推动计划维护

提醒机制表明，这套系统不相信模型会持续自律，因此把维护计划本身做成 runtime 职责。

## 14. 结论

这个项目在规划和代理协作上的真正竞争力，是把“先想清楚再做、分工执行、持续同步状态、受控转入执行”这一套流程，落实成了：

- 模式
- 文件
- 工具
- 任务对象
- 附件提醒
- 子代理上下文

这使得它不仅能“回答问题”，而且能比较稳定地承接复杂、多步、长周期的软件工程任务。
