# 代码细读：扩展性、性能、安全

## 先给结论

这份附录要说明三件事：

1. 这个项目为什么不只是一个固定工具集，而是一层能力协议。
2. 它为什么能在长会话里尽量不被上下文成本拖垮。
3. 它为什么比很多 agent demo 更接近生产可用。

对应的核心文件：

- `Tool.ts`
- `tools.ts`
- `commands.ts`
- `utils/toolSearch.ts`
- `skills/loadSkillsDir.ts`
- `services/compact/autoCompact.ts`
- `services/compact/compact.ts`
- `utils/permissions/permissionSetup.ts`
- `tools/BashTool/bashPermissions.ts`
- `tools/PowerShellTool/pathValidation.ts`

## 1. `Tool.ts`：它定义的不是函数，而是能力协议

从 `Tool.ts` 的接口设计看，一个 tool 至少带这些语义：

- `schema`
- `validateInput`
- `checkPermissions`
- `isConcurrencySafe`
- `isReadOnly`
- `requiresUserInteraction`
- `shouldDefer`
- `searchHint`

可以把它抽成：

```ts
type Tool = {
  name: string
  description(): Promise<string>
  prompt?(): Promise<string>
  inputSchema: zodSchema
  outputSchema: zodSchema
  validateInput?(input, context)
  checkPermissions?(input, context)
  isConcurrencySafe?(): boolean
  isReadOnly?(): boolean
  shouldDefer?: boolean
  searchHint?: string
  call(input, context)
}
```

这意味着上层 runtime 对工具的理解不是“一个可调用函数”，而是：

- 能不能晚一点再暴露
- 能不能并发
- 算不算危险
- 输入是否合法
- 适合被什么查询召回

这就是为什么它比较容易做大。

---

## 2. `tools.ts`：工具注册中心也是能力装配层

`tools.ts` 的意义不是列举工具，而是按当前环境、模式和 feature gate 装配出真正可见的工具集合。

可以抽成：

```ts
function buildToolSet(runtimeState) {
  tools = allBuiltInTools()

  tools = filterByEnvironment(tools)
  tools = filterByFeatureFlags(tools)
  tools = filterByDenyRules(tools)
  tools = filterBySimpleModeOrReplMode(tools)

  return tools
}
```

这层很重要。因为一旦工具多起来，“全部永远可见”会造成：

- prompt 体积膨胀
- 错误路由
- 权限风险

所以它必须先做能力裁剪。

## 3. `utils/toolSearch.ts`：把工具可见性做成按需检索

关键函数包括：

- `getToolSearchMode()`
- `isToolSearchEnabled()`
- `getDeferredToolsDelta()`
- `extractDiscoveredToolNames()`

可以抽成：

```ts
function getToolSearchMode() {
  // standard / tst / tst-auto
}

async function isToolSearchEnabled(model, tools, messages) {
  if (mode === 'standard') return false
  if (mode === 'tst') return true
  if (mode === 'tst-auto') {
    chars = calculateDeferredToolDescriptionChars(tools)
    return chars > autoThresholdFor(model)
  }
}

function getDeferredToolsDelta(tools, messages) {
  discovered = extractDiscoveredToolNames(messages)
  return toolsThatNeedToBeSurfacedNow(tools, discovered)
}
```

这几段代码说明：

- 工具搜索并不是“有无”开关
- 它和模型上下文预算绑在一起
- 它会根据会话里已经发现过的工具动态增量注入

这就是为什么它的工具体系扩展性更好。

## 4. `skills/loadSkillsDir.ts`：Skill 已经被做成了元数据对象

这层的关键函数包括：

- `parseSkillFrontmatterFields()`

```ts
function loadSkills() {
  files = loadMarkdownFrom([
    userSkillDir,
    projectSkillDir,

    bundledSkillDir,
```

这意味着 skill 不是一段 prompt，而是：

- 有 agent / model 偏好

一旦做到这一步，skill 就从“文案片段”升级成了“高层策略对象”。

## 5. `autoCompact.ts`：上下文窗口被当成正式系统资源

关键函数：

- `getEffectiveContextWindowSize()`
- `getAutoCompactThreshold()`
- `shouldAutoCompact()`
- `autoCompactIfNeeded()`

可以抽成：

```ts
function shouldAutoCompact(messages, model, tracking) {
  window = getEffectiveContextWindowSize(model)
  threshold = getAutoCompactThreshold(model)
  usage = tokenCountWithEstimation(messages)

  if (!autoCompactEnabled) return false
  if (tracking.recentFailuresTooMany) return false
  if (usage + reserve < threshold) return false

  return true
}

async function autoCompactIfNeeded(messages, context) {
  if (!shouldAutoCompact(...)) return messages

  if (sessionMemoryCompactPossible(messages)) {
    return compactViaSessionMemory(messages)
  }

  return fullCompact(messages)
}
```

注意这里的思想：

- 先估算窗口
- 再算阈值
- 再看是否预留足够空间
- 还有 compact failure 的 circuit breaker

说明它非常清楚 compact 不是免费的、也不是永远可靠的。

## 6. `compact.ts`：压缩不是简单摘要，而是结构保真

可以把它理解成：

```ts
function compact(messages) {
  summary = summarizeImportantExecutionState(messages)
  stripped = stripHeavyBlocksLikeImages(messages)
  reinjected = reinjectEssentialAttachments(stripped)
  return addCompactBoundary(summary, reinjected)
}
```

这层的关键不是“让消息更短”，而是“让接下来的执行继续能跑”。

换句话说，它在保的是执行结构，而不是保表面自然语言。

## 7. `permissionSetup.ts`：先识别危险能力，再谈模式切换

这一层的关键函数包括：

- `isDangerousBashPermission()`
- `isDangerousPowerShellPermission()`
- `removeDangerousPermissions()`
- `stripDangerousPermissionsForAutoMode()`
- `restoreDangerousPermissions()`
- `transitionPermissionMode()`
- `prepareContextForPlanMode()`

可以抽成：

```ts
function transitionPermissionMode(current, targetMode) {
  if (targetMode === 'auto') {
    current = stripDangerousPermissionsForAutoMode(current)
  }

  if (targetMode === 'plan') {
    current = prepareContextForPlanMode(current)
  }

  if (leavingAutoMode) {
    current = restoreDangerousPermissions(current)
  }

  return {
    ...current,
    mode: targetMode,
  }
}
```

这说明 permission mode 不是 UI 状态，而是能力集合会跟着变化的运行时状态。

## 8. `bashPermissions.ts`：Shell 安全不是黑名单，而是结构分析

这一层非常深，核心函数包括：

- `getSimpleCommandPrefix()`
- `getFirstWordPrefix()`
- `stripSafeWrappers()`
- `matchingRulesForInput()`
- `checkCommandAndSuggestRules()`
- `bashToolHasPermission()`

可以抽成：

```ts
function bashToolHasPermission(command, permissionContext) {
  normalized = stripSafeWrappers(command)
  prefix = getSimpleCommandPrefix(normalized) ?? getFirstWordPrefix(normalized)

  if (checkEarlyExitDeny(normalized)) {
    return deny()
  }

  if (matchesExistingAllowRule(prefix, normalized)) {
    return allow()
  }

  semanticDecision = checkSemanticsDeny(normalized)
  if (semanticDecision === 'deny') {
    return denyWithReason()
  }

  suggestion = checkCommandAndSuggestRules(normalized)
  return askWithSuggestedPermissionRule(suggestion)
}
```

这套逻辑比常见的“字符串里含 rm 就拒绝”强太多，因为它在认真处理：

- wrapper 命令
- heredoc
- env vars
- prefix suggestion
- classifier
- sandbox auto allow

也就是说，它知道 shell 命令的风险来自组合性。

## 9. `pathValidation.ts`：PowerShell 安全被单独当成一类问题处理

关键函数包括：

- `isDangerousRemovalRawPath()`
- `dangerousRemovalDeny()`
- `validatePath()`
- `checkPathConstraints()`

可以抽成：

```ts
function checkPathConstraints(parsedPowerShellCommand, sandboxRules) {
  paths = extractPathsFromCommand(parsedPowerShellCommand)

  for (path of paths) {
    expanded = expandTilde(path)

    if (isDangerousRemovalRawPath(expanded)) {
      return dangerousRemovalDeny(expanded)
    }

    if (!isPathAllowed(expanded, sandboxRules)) {
      return deny(`path outside allowed roots: ${expanded}`)
    }

    if (matchesDenyRule(expanded)) {
      return deny(`denied by permission rule`)
    }
  }

  return allow()
}
```

这说明作者没有把 Bash 和 PowerShell 混为一谈，而是承认：

- PowerShell 有自己的 cmdlet 语义
- 参数位不一样
- path 读写行为也不一样

这是真正做桌面/终端 agent 时必须有的细粒度。

## 10. 这三层为什么合起来很强

如果只看某一个点，会觉得只是“设计挺细”。但三层合起来才是真优势：

### 10.1 扩展性层

```text
Tool / Skill / Command / MCP 都被对象化
```

### 10.2 性能层

```text
Deferred Tool Search + Compact + Budget + Prefetch
```

### 10.3 安全层

```text
Permission Mode + Shell Analysis + Path Validation + Edit Guard
```

这三层一组合，系统就具备了：

- 能长大
- 能长聊
- 不容易炸

## 11. 一份可以照抄的总伪代码

如果要把这三层压成一个总骨架，大概是：

```ts
function buildAgentRuntime() {
  tools = buildToolSet(environment, featureFlags, permissionMode)
  skills = loadSkillsFromAllSources()

  if (toolDescriptionsTooLong || contextTooLarge) {
    enableToolSearch()
  }

  messages = maybeCompact(messages)
  messages = maybeBudgetToolResults(messages)
  messages = maybePrefetchRelevantContext(messages)

  permissions = transitionPermissionModeAsNeeded(permissions)
  shellCalls = checkShellSemantics(shellCalls, permissions)
  pathCalls = checkPathConstraints(pathCalls, sandbox)

  return {
    tools,
    skills,
    messages,
    permissions,
  }
}
```

这个骨架本身就已经说明它不只是一个“套壳大模型”。
