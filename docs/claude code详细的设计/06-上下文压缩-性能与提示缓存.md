# 上下文压缩、性能与提示缓存

## 核心判断

很多 agent 项目只在功能层竞争，这个仓库明显不是。它对“上下文窗口是昂贵资源”这件事有很强的工程意识，因此做了大量上下文治理和性能优化。

关键代码主要在：

- `src/query.ts`
- `src/services/compact/autoCompact.ts`
- `src/services/compact/compact.ts`
- `src/services/compact/sessionMemoryCompact.ts`
- `src/utils/toolSearch.ts`
- `src/utils/forkedAgent.ts`
- `src/tools/AgentTool/forkSubagent.ts`
- `src/utils/sessionStorage.ts`

## 1. Auto Compact：不是超了再压，而是带阈值和预留的上下文控制

`autoCompact.ts` 会考虑：

- context window reserve
- proactive / reactive compact gating
- session memory first
- consecutive failure circuit breaker

这说明系统不是等上下文爆了再临时补救，而是在做持续的容量管理。

尤其 circuit breaker 很重要，说明作者知道 compact 并不是永远稳定成功的，因此需要失败熔断，避免不断在错误 compact 上浪费轮次。

## 2. Compact 不是简单摘要，而是结构化保真

`compact.ts` 做的不是一段普通总结，它会：

- 生成详细 summary prompt
- strip images
- reinject attachments
- 保留 transcript 和 boundary markers

这意味着 compact 的目标不是把聊天变短，而是尽量保留后续执行所需的结构信息。

对代码 agent 来说，最怕的是摘要后丢掉：

- 已经读过哪些文件
- 当前有哪些任务边界
- 哪些附件很关键

这个实现明显在避免这一点。

## 3. Session Memory Compact：优先压过程，保留工作状态

`sessionMemoryCompact.ts` 前面已经提到，它的价值在于：

- 不优先丢失工作记忆
- 优先把 transcript 压缩成结构化 session state

这是一种很成熟的长任务处理方式。因为对于复杂工程任务，真正重要的往往不是逐轮聊天原文，而是“现在的任务状态摘要”。

## 4. Tool Result Budget：工具结果也被纳入上下文配额管理

`query.ts` 在模型调用前会应用 tool result budget。这个细节很关键，说明系统明白：

- 不只是对话历史会撑爆上下文
- 工具输出同样可能膨胀 prompt

很多 agent 系统的真实瓶颈不是聊天本身，而是大段文件内容、日志、命令输出。这里已经把这类成本纳入预算。

## 5. Deferred Tool Discovery：用搜索替代全量注入

`utils/toolSearch.ts` 支持：

- `tst`
- `tst-auto`
- `standard`

这不仅是意图路由问题，也是性能工程。因为全量工具定义会占掉大量 prompt 空间，而延迟发现可以把“工具集合”从固定成本变成按需成本。

在工具越来越多时，这个收益会越来越大。

## 6. Forked Agent 的缓存安全前缀：在代理协作里复用 prompt cache

`utils/forkedAgent.ts` 和 `forkSubagent.ts` 的另一个亮点，是尽量保持 cache-safe prefix 一致。

这有两个好处：

- 子代理不用重新支付整段系统前缀成本
- 同一任务的多个代理更容易共享缓存和理解基底

很多多代理系统忽视这一点，结果并发一多，成本和延迟都迅速上升。这里已经在架构层考虑到。

## 7. Prefetch：把慢操作前移但不阻塞主路径

在 `query.ts` 里，relevant memory prefetch 和 skill prefetch 都说明了一个模式：

- 先启动可能会用到的检索型工作
- 不立刻阻塞主流程
- 在真正需要前尝试收割结果

这是一种很典型的吞吐优化。其本质是让“语义计算”和“上下文检索”并行化。

## 8. Session Storage：压缩和恢复必须有稳定边界

`sessionStorage.ts` 管理：

- compact boundaries
- snip removals
- transcript adoption
- sidechain transcript
- resume consistency

这说明上下文压缩不是一次性文本处理，而是和持久化、恢复、任务 sidechain 一起考虑的。

如果没有稳定边界，compact 之后的恢复和继续执行会很脆弱。

## 9. 这套性能思路的本质

它不是“做几处优化”，而是把上下文和推理成本视作正式系统资源，围绕这个资源做了四类治理：

- 少给：延迟暴露工具、预算工具输出
- 先压：auto compact、session memory compact
- 早取：prefetch memory / skill
- 复用：cache-safe prefix、session persistence

## 10. 结论

这个项目在性能上的竞争力，不在 benchmark 跑分，而在它知道真正昂贵的是：

- 长历史
- 大工具输出
- 重复系统前缀
- 检索阻塞
- 恢复不稳定导致的重复探索

它围绕这些点做的设计，使它更适合长会话和复杂工程任务。
