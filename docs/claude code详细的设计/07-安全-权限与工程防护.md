# 安全、权限与工程防护

## 核心判断

这个仓库的另一个明显优势，是它没有把安全当成外围补丁，而是把权限、路径校验、写入防线、危险操作检测内建进了执行路径。

关键代码主要在：

- `src/utils/permissions/permissionSetup.ts`
- `src/tools/BashTool/bashPermissions.ts`
- `src/tools/PowerShellTool/pathValidation.ts`
- `src/tools/FileEditTool/FileEditTool.ts`
- `src/utils/forkedAgent.ts`
- `src/tools/EnterPlanModeTool/EnterPlanModeTool.ts`
- `src/tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts`

## 1. Permission Setup：先决定什么危险，再决定能否执行

`permissionSetup.ts` 会对 Bash、PowerShell、Agent 等能力建立危险操作判断和 allow rule 过滤，尤其在 auto mode 里会主动剥离危险规则。

这说明权限体系不是简单“允许/拒绝”开关，而是按：

- 工具类别
- 模式
- 风险等级

做细分治理。

## 2. Bash 权限控制不是字符串黑名单，而是命令结构分析

`bashPermissions.ts` 里可以看到：

- AST / 安全解析
- dangerous shell prefix 检测
- command prefix extraction
- classifier integration
- sandbox 语义
- heredoc 等特殊情况处理

这很关键。因为 shell 命令最危险的地方在于它高度组合化，简单黑名单很容易失效。这里明显试图在结构层面识别危险命令。

## 3. PowerShell 路径校验体现了平台细节意识

`pathValidation.ts` 并不是照搬 Bash 逻辑，而是针对 PowerShell 做了：

- cmdlet-specific path config
- 读写检测
- canonical path validation
- sandbox write allowlist
- 危险删除路径检查

这说明作者没有把“命令安全”当成跨平台统一问题，而是尊重不同 shell 的语义差异。这个细节很见功力。

## 4. FileEditTool：最重要的安全不只是权限，而是防误改

`FileEditTool.ts` 的那些限制，本质上也是安全策略：

- 没读过不许改
- 只看了片段不许大改
- 文件变了不许基于旧状态改
- old_string 不唯一不许改

这是一种非常实用的工程安全观。因为代码 agent 在真实场景里最常见的问题，不一定是恶意，而是误伤。

而误伤一旦发生，代价往往比简单的拒绝执行更大。

## 5. 子代理默认避免权限提示，体现最小惊扰原则

`utils/forkedAgent.ts` 里 `shouldAvoidPermissionPrompts` 的默认策略很值得注意。

它的含义不是“子代理更自由”，恰恰相反，是：

- 子代理不应随意把权限交互扩散给用户
- 子代理最好在受限环境里完成可完成的事
- 真正需要升级权限时，应由上层更明确地处理

这符合最小权限和最小惊扰原则。

## 6. Plan Mode 也是一种安全机制

很多人会把 Plan Mode 只看成流程优化，但它其实也承担安全职责：

- 在不确定任务中先探索再执行
- 通过 ExitPlan 建立从设计到执行的边界
- 允许 leader approval / teammate coordination

也就是说，这套系统把“先规划再动手”做成了权限和风险治理的一部分。

这比直接让模型边想边改安全得多。

## 7. 工程防护的核心思路

综合来看，这个仓库的安全不是只防“恶意命令”，而是同时防三类风险：

- 真的危险操作
- 越权操作
- 基于错误前提的误操作

第三类最容易被忽视，但对代码 agent 恰恰最常见。

## 8. 为什么这层设计有竞争力

### 8.1 它更接近生产系统，而不是 demo

真正要落到团队日常使用，安全边界必须细。

### 8.2 它减少的是灾难性错误，而不是只减少警告

路径校验、旧内容匹配、文件变更检测，这些都会直接降低误删误改概率。

### 8.3 它把安全嵌进执行流，而不是事后审计

危险时拒绝执行，比事后补救更有价值。

## 9. 结论

这个仓库之所以更像一个可用的工程 agent runtime，而不是玩具 demo，很大一部分原因就在于它对安全的理解足够工程化：

- 有权限模式
- 有 shell 结构分析
- 有路径和写入边界
- 有文件事实校验
- 有规划态到执行态的门槛
- 有子代理的最小权限原则

这套防线让它在真实代码库中更可控。
