# 意图识别与输入路由

## 核心判断

这套代码的“意图识别”并不是经典意义上的 intent classification，而是一个多层输入路由系统。它不追求先给用户打一个标签，再决定动作；它做的是把不同形态的输入拆开、规整、补强、附加上下文，然后送进不同执行通道。

这和很多外部 agent 项目最大的不同在于：它把意图识别工程化了。

关键入口主要在这些位置：

- `src/utils/processUserInput/processUserInput.ts`
- `src/utils/processUserInput/processTextPrompt.ts`
- `src/utils/attachments.ts`
- `src/tools/ToolSearchTool/ToolSearchTool.ts`
- `src/tools/SkillTool/SkillTool.ts`
- `src/utils/toolSearch.ts`
- `src/commands.ts`

## 1. 第一层：输入规范化，而不是直接喂模型

`processUserInput.ts` 做的第一件事不是问模型“这句话啥意思”，而是先对输入做工程层的整理：

- 处理图片 block 和粘贴图片
- 识别斜杠命令
- 识别 bridge-safe command
- 识别是否需要抽取附件
- 识别 `ultraplan` 等特殊模式入口

这一步的意义很大。因为真实用户输入不是纯文本，它可能混着：

- 命令
- 文件引用
- 图片
- 外部资源
- 代理 mention
- 计划模式切换信号

如果一开始不做拆分，后面的模型推理就会被噪声污染。

## 2. 第二层：显式命令优先于隐式语义猜测

`parseSlashCommand`、`findCommand`、`isBridgeSafeCommand` 这一层表明，系统优先尊重显式操作信号。

这背后的思路很关键：

- 用户明确输入 `/plan`，不要再让模型“猜是不是想计划”
- 用户输入工具命令时，先走命令分发，不要用自然语言理解重做一遍
- 模型负责解释模糊请求，但对显式协议输入应尽量零歧义执行

这是一种非常成熟的 runtime 设计。它减少了“明明是命令，结果被模型理解偏了”的问题。

## 3. 第三层：附件系统才是真正的隐式意图增强器

真正有竞争力的地方在 `attachments.ts`。这不是简单的“顺手补几个附件”，而是一套大规模上下文增强管道。它会根据当前输入和会话状态，自动注入不同类型的附件信息。

它处理的内容包括：

- `@` 文件提及
- MCP resources
- agent mentions
- changed files
- nested memory
- dynamic skill
- skill listing
- plan mode / exit / reentry reminders
- todo / task reminders
- teammate mailbox
- pending agent messages
- critical reminders

这意味着“意图识别”在这里并不是一句话的语义归类，而是：

1. 看用户输入里出现了什么显式结构。
2. 看当前系统运行到什么状态。
3. 自动补上这次决策真正需要的上下文。

外面很多 agent 框架只做第 1 步，这里把第 2、3 步做成了正式机制。

## 4. Tool Search：把“我该用哪个工具”延后求值

`ToolSearchTool.ts` 和 `utils/toolSearch.ts` 很值得关注。很多系统会把所有工具定义直接塞进 prompt，让模型自己选。这个项目不是，它支持：

- `standard`
- `tst`
- `tst-auto`

这种 deferred tool discovery 思路非常有竞争力，原因有三点：

### 4.1 它降低了基础 prompt 膨胀

工具越来越多时，把完整 schema 全塞进 prompt，成本很高。这里通过搜索和条件展开，只在需要时暴露工具。

### 4.2 它把“选工具”变成检索问题

模型不再被迫从一大串工具里盲选，而是先根据关键词、别名、MCP 名称、`searchHint` 做初筛，再进入更小候选集。

### 4.3 它让系统更易扩展

工具越多，传统方式越容易退化。这里通过搜索层缓冲了扩展压力。

## 5. Skill 不是提示词模板，而是可发现、可执行的能力单元

`SkillTool.ts` 表明，skill 在这套系统里不是一段静态文案。它可以：

- 被动态发现
- 被 telemetry 记录
- 和 MCP skill 并列
- 运行在 forked subagent context 里

这很关键。因为它说明系统把“高层策略”封装成了能力对象，而不是临时拼接 prompt。

外部很多项目的 skill system 只是“预置几段 prompt”。这里的 skill 更像：

- 有元信息
- 有发现逻辑
- 有执行上下文
- 有工具边界

这使得意图识别的最终输出，不是“分类标签”，而可能是“把请求转交给某个 skill/agent/tool chain”。

## 6. 输入路由的真正结构

如果把这一层抽象成一个流程，大致是：

1. 用户输入进入 `processUserInput`
2. 先做内容级规范化
3. 检查是否是显式 command
4. 抽取文件、资源、agent、memory 等附件
5. 根据模式和上下文补充系统附件
6. 视配置决定是否做 tool search / skill discovery
7. 再进入主 query loop

这意味着“意图识别”本质上被拆到了多个阶段，而不是一次性判断。

## 7. 为什么这比常见做法更强

常见做法：

- 把原始文本直接发给模型
- 给一组工具定义
- 让模型自己猜用户意图和工具选择

这个仓库的做法：

- 先做输入协议化
- 再做上下文补强
- 再做模式与能力路由
- 最后才让模型做语义决策

差别在于，前者依赖模型“聪明”；后者依赖系统“有秩序”。

在工程系统里，后者的稳定性和可扩展性都明显更高。

## 8. 这层设计的核心亮点

### 亮点一：显式输入和隐式输入统一进同一条路由链

命令、自然语言、附件、资源、模式切换不是分裂处理，而是最终汇总为统一 query context。

### 亮点二：把“环境状态”也纳入意图识别

plan mode、pending agents、todo 状态、memory surfacing 这些都影响系统对当前输入的解释。这使得它的意图识别是 stateful 的。

### 亮点三：能力发现被正式系统化

Tool search 和 skill discovery 说明“系统能做什么”并不是静态 prompt 常量，而是可检索、可裁剪的。

## 9. 结论

如果只看表面，会以为这里的意图识别只是“处理一下 slash command”。但深入看会发现，它真正先进的地方是：

- 不把 intent 当分类问题
- 而是把 intent 当路由问题
- 再把路由建立在附件增强、能力检索、模式状态和执行上下文之上

这就是它和很多外部 agent 项目拉开差距的第一层原因。
