# 代码细读：全局写码与执行闭环

## 先给结论

这个项目比很多代码 agent 强的核心，不在于“生成 patch 的语言能力”，而在于它先建立一个仓库级事实世界，再在这个事实世界里做读写。

简单说，它做的是：

```text
建立仓库事实 -> 校验事实是否仍有效 -> 再允许编辑
```

而不是：

```text
看几段代码 -> 直接写 patch
```

---

## 1. `QueryEngine` 保存了一份持续演化的“仓库世界状态”

从 `QueryEngine.ts` 的字段可以直接看出这点：

```ts
class QueryEngine {
  private mutableMessages: Message[]
  private readFileState: FileStateCache
  private loadedNestedMemoryPaths = new Set<string>()
  private discoveredSkillNames = new Set<string>()
  private permissionDenials = []
  ...
}
```

这几个字段很关键：

- `mutableMessages`：会话事实流
- `readFileState`：代码事实缓存
- `loadedNestedMemoryPaths`：已加载的嵌套规则/记忆
- `discoveredSkillNames`：当前会话已发现策略能力

也就是说，主引擎并不是“收到一条消息就重新来过”，它在维护一份状态机。

---

## 2. `fetchSystemPromptParts()` + `context.ts` 先把仓库背景灌进去

`QueryEngine` 在启动 query 前会先拿：

```ts
const {
  defaultSystemPrompt,
  userContext,
  systemContext,
} = await fetchSystemPromptParts(...)
```

而 `context.ts` 这一层会把这些内容注入进来：

- 当前 branch
- main branch
- git status
- recent commits
- git user
- `CLAUDE.md`
- 当前日期

可以理解成：

```ts
systemContext = {
  git: currentRepoSnapshot(),
  environment: currentDate(),
}

userContext = {
  projectInstructions: loadClaudeMd(),
}
```

这一步很重要，因为模型开始写代码前，就已经知道：

- 我在什么仓库
- 当前工作区脏不脏
- 最近改动方向是什么
- 项目自己的工作规则是什么

## 3. `QueryEngine.ask()` 如何把这些状态串起来

从代码可以抽出这样一段框架：

```ts
async *ask(prompt) {
  { defaultSystemPrompt, userContext, systemContext } =
    await fetchSystemPromptParts(...)

  processUserInputContext = {
    messages: this.mutableMessages,
    readFileState: this.readFileState,
    loadedNestedMemoryPaths: this.loadedNestedMemoryPaths,
    discoveredSkillNames: this.discoveredSkillNames,
    ...
  }

  result = await processUserInput(...)
  this.mutableMessages.push(...result.messages)

  if (persistSession) {
    await recordTranscript(messages)
  }

  yield buildSystemInitMessage(...)

  for await (message of query(...)) {
    // 持续更新 mutableMessages / transcript / state
  }
}
```

最关键的点在于：`readFileState`、`loadedNestedMemoryPaths`、`discoveredSkillNames` 都是被带进 query loop 的。

所以主循环不是只拿聊天历史，而是拿着一整份运行状态。

## 4. `FileReadTool` 的真正作用：建立可验证的代码事实

虽然这里我没有逐行展开 `FileReadTool.ts`，但从系统行为和 `readFileState` 的后续使用方式可以反推出它的职责：

```ts
async function FileReadTool.call({ file_path, offset, limit }, context) {
  fullPath = normalize(file_path)
  content = readFileOrSlice(fullPath, offset, limit)

  context.readFileState.set(fullPath, {
    timestamp: nowOrFileMtime,
    content,
    offset,
    limit,
    isPartialView: offset !== undefined || limit !== undefined,
  })

  return { type: 'text', file: { path: fullPath, content } }
}
```

这意味着文件读取不仅仅是“拿到文本”，更是在登记一条事实：

```text
我在某个时间点读过这个文件的这一版内容。
```

后面所有安全编辑，都建立在这层事实之上。

## 5. `FileEditTool.validateInput()` 是最核心的工程防线之一

这段代码在 `tools/FileEditTool/FileEditTool.ts`，非常值得抄。它的验证流程几乎可以直接改写成：

```ts
async function validateInput({ file_path, old_string, new_string, replace_all }, context) {
  fullFilePath = expandPath(file_path)

  if (checkTeamMemSecrets(fullFilePath, new_string)) {
    return reject('team memory secret violation')
  }

  if (old_string === new_string) {
    return reject('old_string and new_string are exactly the same')
  }

  if (denyRuleMatches(fullFilePath)) {
    return reject('path denied by permission settings')
  }

  if (isUNCPath(fullFilePath)) {
    return allowEarly() // 交给权限层，不做额外 fs 操作，防止 NTLM 泄漏
  }

  if (file too large) {
    return reject('file too large')
  }

  fileContent = readFileIfExists(fullFilePath)

  if (file does not exist) {
    if (old_string === '') return allowCreate()
    return reject('file not found, maybe similar filename?')
  }

  if (old_string === '' && file already has content) {
    return reject('cannot create new file, file already exists')
  }

  if (isNotebook(fullFilePath)) {
    return reject('use notebook edit tool')
  }

  readTimestamp = context.readFileState.get(fullFilePath)
  if (!readTimestamp || readTimestamp.isPartialView) {
    return reject('file has not been read yet')
  }

  lastWriteTime = getFileModificationTime(fullFilePath)
  if (lastWriteTime > readTimestamp.timestamp) {
    if (!(fullRead && fileContent === readTimestamp.content)) {
      return reject('file modified since read, read again first')
    }
  }

  actualOldString = findActualString(fileContent, old_string)
  if (!actualOldString) {
    return reject('string to replace not found')
  }

  matches = countMatches(fileContent, actualOldString)
  if (matches > 1 && !replace_all) {
    return reject('multiple matches, provide more context or replace_all=true')
  }

  return allow()
}
```

这段代码几乎把代码 agent 最常见的误伤路径都堵上了：

- 没读过就改
- 只读了局部就全局改
- 基于过期文件版本改
- `old_string` 匹配不唯一
- 错把已有文件当新文件创建

所以它的竞争力不只是“会 edit”，而是“只在事实充分时才 edit”。

## 6. `readFileState` 和 `FileEditTool` 共同构成写码闭环

这两者合在一起，可以写成一句伪代码：

```ts
if (!I_have_read_the_current_version_of_this_file) {
  deny_edit()
}
```

这条原则看似简单，但其实是很多代码 agent 没有真正落实的。

也正因为有这层闭环，系统才能在多轮对话、多人协作、外部 lint/format 介入的情况下，仍然保持写入稳定性。

## 7. `getChangedFiles()` 负责主动告诉模型：你的事实过期了

`utils/attachments.ts` 里的 `getChangedFiles()` 可以抽成：

```ts
async function getChangedFiles(context) {
  changed = []

  for (filePath in context.readFileState) {
    if (file has changed since it was read) {
      latest = await FileReadTool.call({ file_path: filePath }, context)
      diffSnippet = buildDiffSnippet(oldContent, latest.content)
      changed.push(createEditedTextFileAttachment(filePath, diffSnippet))
    }
  }

  return changed
}
```

这层特别重要，因为它在主动修正模型的“旧世界模型”。

很多系统的问题不是模型没看过文件，而是：

- 看过
- 但后来文件变了
- 模型还当它没变

这里 runtime 会主动把这个失效事实回灌给模型。

## 8. `LSPTool`：全局写码不只靠 grep

`tools/LSPTool/LSPTool.ts` 支持：

- definition
- references
- hover
- document symbol
- workspace symbol
- implementation
- call hierarchy

可以把它理解成：

```ts
function getProgramStructure(question) {
  if (need definition) return goToDefinition(...)
  if (need impact analysis) return findReferences(...)
  if (need symbol map) return workspaceSymbol(...)
  if (need caller/callee graph) return callHierarchy(...)
}
```

这意味着系统维护的不是单纯文本上下文，还有程序结构上下文。

这对“全局写码”很重要。因为真正复杂的代码修改，通常都需要回答：

- 这个函数被谁调用
- 这个类型在哪里被实现
- 改这里会波及哪些文件

## 9. `toolOrchestration.ts`：工具执行顺序本身也是全局写码的一部分

这层代码的思想可以抽成：

```ts
async function* runTools(toolUseMessages) {
  groups = partitionToolCalls(toolUseMessages)

  for (group of groups) {
    if (group is serial) {
      yield* runToolsSerially(group)
    } else {
      yield* runToolsConcurrently(group)
    }
  }
}
```

`partitionToolCalls()` 会决定：

- 哪些工具可以并发
- 哪些工具必须串行
- 哪些写操作不该和别的调用混在一起

这很关键，因为代码任务通常不是一个动作，而是一串有依赖的动作：

- 先读文件
- 再查符号
- 再运行命令
- 再写文件
- 再验证

如果这一层没有 orchestration，模型很容易把动作顺序搞坏。

## 10. 真实的“全局写码闭环”长什么样

把上面的模块拼起来，系统大致是在做下面这件事：

```ts
function globalCodingLoop(userRequest) {
  // 1. 先建立仓库背景
  repoState = {
    git: currentBranchStatusAndRecentCommits(),
    projectRules: loadClaudeMd(),
  }

  // 2. 再收集本轮需要的附件和记忆
  runtimeContext = collectAttachmentsAndMemories(userRequest)

  // 3. 再进入工具驱动的探索
  files = readFilesAndRecordState()
  symbols = queryLSPIfNeeded()
  relatedChanges = surfaceChangedFilesIfNeeded()

  // 4. 只有在事实充分时才允许编辑
  if (fileWasRead && notPartial && notModifiedSinceRead && uniqueMatch) {
    editFile()
  } else {
    forceReReadOrAskForMoreContext()
  }

  // 5. 工具执行后再把结果纳回上下文
  updateMessages()
  maybeGenerateToolUseSummary()
  maybeCompact()
}
```

这才是它“基于全局去写代码”的真正含义。

---

## 11. 为什么这套闭环比外面的常见实现强

常见实现：

```text
用户需求 -> 读几个文件 -> 生成 patch
```

这个仓库：

```text
用户需求
  -> 仓库背景注入
  -> 记忆/附件增强
  -> 文件事实建立
  -> 结构关系查询
  -> 事实过期检测
  -> 编辑前校验
  -> 编排工具执行
  -> 再把结果纳回会话状态
```

所以它的“全局”不是一句营销词，而是 runtime 真的在维护全局状态。
