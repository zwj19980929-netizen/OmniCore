# 全局写码与仓库感知执行

## 核心判断

这个仓库最核心、也最有竞争力的能力，不是“会改代码”，而是它会先建立并维护一个仓库级世界模型，再基于这个世界模型进行修改。

这和很多外部 agent 的差别非常大。外部常见做法是：

- 看到用户需求
- 读几个文件
- 让模型直接生成 patch

而这里的做法更像：

- 不断收集仓库状态
- 记录哪些事实已经建立
- 跟踪这些事实是否过期
- 只在约束满足时才允许写入

关键代码主要在：

- `src/context.ts`
- `src/QueryEngine.ts`
- `src/query.ts`
- `src/tools/FileReadTool/FileReadTool.ts`
- `src/tools/FileEditTool/FileEditTool.ts`
- `src/utils/fileStateCache.ts`
- `src/utils/attachments.ts`
- `src/tools/LSPTool/LSPTool.ts`
- `src/tools/LSPTool/symbolContext.ts`
- `src/services/tools/toolOrchestration.ts`
- `src/services/tools/StreamingToolExecutor.ts`

## 1. 系统上下文不是空壳，它会注入仓库状态

`context.ts` 的 `getSystemContext()` 很关键。它会把这些信息注入系统上下文：

- 当前 branch
- main branch
- git status
- recent commits
- git user

此外 `getUserContext()` 还会注入：

- `CLAUDE.md`
- 当前日期

这意味着模型从一开始获得的就不是一个抽象“编程问题”，而是一个明确带着仓库状态的工程现场。

这对写代码有决定性作用。因为很多修改是否合理，取决于：

- 当前分支是不是 feature branch
- 工作区是否已经脏了
- 最近提交在改什么方向
- 仓库约定是什么

这就是“全局写码”的第一层基础。

## 2. QueryEngine 持有的是运行时世界状态，不只是消息数组

`QueryEngine.ts` 里保留了大量跨轮事实：

- `mutableMessages`
- permission denials
- total usage
- `readFileState`
- discovered skill names
- loaded nested memory paths

这很重要，因为它说明主引擎并不是每轮都在“重新开始思考”。它在持续维护一份世界状态。

尤其是 `readFileState`，它等于告诉模型：

- 哪些文件已经被确认过
- 当前对代码库的认知边界在哪里
- 后续读写动作是否还建立在有效前提上

这和普通聊天式 LLM 非常不同。

## 3. FileReadTool：读文件不是拿文本，而是在建立可信事实

`FileReadTool.ts` 的设计远比“read file”名字看起来更重要。它做了很多约束和记录：

- token / size limit
- offset / limit 分片读取
- 缓存读到的内容到 `readFileState`
- 检测未变化的重复读取
- 拒绝危险设备路径
- 支持图片 / PDF 等非纯文本场景

这说明一次文件读取的真正意义是：

> 在系统里登记一条“我已经看过这个版本的这个文件”的事实。

这就是后续安全编辑、变化检测、上下文压缩都能成立的基础。

## 4. FileEditTool：写代码前要证明自己看过、而且看的还是当前版本

`FileEditTool.ts` 是这个仓库工程质量很高的一块。它并不是“给出 old_string/new_string 就替换”，而是带着严格防线：

- 需要权限检查
- team memory 含 secret 的场景受限制
- `old_string !== new_string`
- 如果文件没读过，不能随便改
- 如果只看了文件片段，不能冒然改全局
- 如果文件在读取后已被别人修改，会拒绝写入
- `old_string` 必须唯一匹配
- notebook 有特殊处理

这背后是一个很成熟的观念：

> 代码修改必须建立在“已验证的文件事实”之上。

外面很多 agent 修改失败，不是模型不会写，而是它基于已经过期或不完整的文件认知在写。

这里把这种风险系统化地压掉了。

## 5. File State Cache：仓库世界模型的轻量记忆层

`utils/fileStateCache.ts` 提供了规范化路径和 LRU 缓存能力。这看起来像一个普通缓存模块，但它的实际价值在于：

- 帮系统稳定识别同一个文件
- 避免路径表示差异导致状态分裂
- 在多轮、子代理、恢复过程中复用文件事实

对代码 agent 来说，最危险的一件事就是“以为自己知道这个文件的状态”。file state cache 是防止这种幻觉的关键设施。

## 6. Changed Files Attachment：系统会主动告诉模型哪些事实过期了

`attachments.ts` 中 `getChangedFiles(...)` 很强。它会扫描 `readFileState`，找出那些自读取之后发生变化的文件，然后：

- 重新读取更新内容
- 生成差异片段
- 以 `edited_text_file` 附件重新注入

这解决了代码 agent 的一个经典问题：

- 模型之前看过文件 A
- 用户或别的工具后来改了 A
- 模型继续按旧内容操作，导致冲突

这里系统会主动把“你原来的事实过期了”反馈给模型。这个设计非常有竞争力。

## 7. LSP Tool：全局写码不只是读文件，更要读符号关系

`LSPTool.ts` 和 `symbolContext.ts` 让系统能够获取：

- go to definition
- find references
- hover
- document symbol
- workspace symbol
- implementation
- call hierarchy

这意味着系统维护的世界模型不仅有文件文本，还有符号级结构。

这对全局写码的意义很大：

- 改一个函数时，不只是改定义，还能看引用
- 分析影响范围时，不必全靠文本搜索
- 处理大型仓库时，可以借助 LSP 做结构导航

很多 agent 只能“看见文本”。这里的系统开始能“看见程序结构”。

## 8. Query Loop 的价值：在模型调用前不断补齐世界状态

`query.ts` 里主循环不是简单“发 prompt -> 收回复”。在每轮模型调用前，它会处理很多事情：

- 消费附件消息
- 消费 relevant memory prefetch
- 收集 skill prefetch
- 应用 tool result budget
- 处理 compact / microcompact / session memory compact
- 生成 tool use summary

这意味着模型真正决策时，看到的是一个经过 runtime 整理后的状态快照，而不是原始对话。

这就是为什么它更接近“受控执行环境”，而不是“聊天接口”。

## 9. Tool Orchestration：全局写码还要求工具执行顺序合理

`toolOrchestration.ts` 和 `StreamingToolExecutor.ts` 也很关键。它们会：

- 对工具调用做并发安全分组
- 把可并发的批量执行
- 保留必须串行的顺序
- 在 Bash 出错时中止同批次兄弟执行
- 保证结果按顺序输出

这个设计的价值在于，代码任务不是单工具调用，而是一连串互相依赖的读写动作。没有 orchestration，模型很容易：

- 读错顺序
- 并发写冲突
- 在前置失败后继续执行后续危险操作

这里 runtime 替模型承担了部分调度责任。

## 10. “全局写码”真正依赖的几类全局事实

综合来看，这个仓库在写代码时会尽量维护以下全局事实：

- 当前仓库分支与工作区状态
- 项目级约束和工作规则
- 已读取文件及其版本
- 当前哪些文件发生了外部变化
- 符号之间的引用和定义关系
- 当前会话中形成了哪些计划、任务、记忆和代理状态

这和“看一段代码改一段代码”完全不是一个量级。

## 11. 这套设计比常见 agent 强在哪

### 11.1 它把“读代码”升级成“建模代码库”

普通 agent 经常只在局部上下文里思考。这里则持续建立 repo state。

### 11.2 它把“改代码”升级成“在事实约束下编辑”

写入前要先满足读过、未过期、匹配唯一、权限允许等条件。

### 11.3 它把“全局理解”升级成多源联合

不是只靠 grep，而是联合：

- git 状态
- 文件状态缓存
- changed files
- LSP
- memory
- plan

### 11.4 它显著降低了长任务中的状态漂移

长时间修改仓库时，最大的敌人不是模型逻辑不强，而是上下文逐渐失真。这里通过状态缓存和变更回注显著压低了这个问题。

## 12. 更深一层：这其实是在做“代码世界模型”

如果把它再抽象一步，可以说这个项目在做的是一种轻量代码世界模型。它并不真正构建 AST graph database，也不做完整语义索引平台，但它已经具备了几个关键要素：

- 当前代码世界的上下文快照
- 文件级事实记忆
- 符号级结构访问
- 变更导致的事实失效检测
- 编辑前事实校验

对一个终端型代码 agent 来说，这是非常强的一条路线。

## 13. 结论

这个仓库的核心竞争力之一，就是它不是在做“文本补丁生成器”，而是在做“带仓库状态感知、事实验证、结构导航和执行编排能力的代码操作 runtime”。

所以它真正厉害的不是模型回答得多漂亮，而是：

- 什么时候该读
- 读了什么算有效事实
- 什么时候这些事实已经失效
- 在什么约束下才能写
- 写之前如何重新校正整个仓库世界状态

这就是它“基于全局去写代码”的本质。
