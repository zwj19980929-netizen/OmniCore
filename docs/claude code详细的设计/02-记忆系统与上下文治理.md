# 记忆系统与上下文治理

## 核心判断

这个仓库的记忆系统非常强，不是因为它“能记住很多历史”，而是因为它明确区分了多种不同类型的记忆，并给每种记忆安排了不同的载体、生命周期和注入时机。

关键代码主要在：

- `src/utils/claudemd.ts`
- `src/memdir/memdir.ts`
- `src/memdir/paths.ts`
- `src/memdir/memoryScan.ts`
- `src/memdir/findRelevantMemories.ts`
- `src/utils/attachments.ts`
- `src/services/extractMemories/extractMemories.ts`
- `src/services/SessionMemory/sessionMemory.ts`
- `src/services/compact/sessionMemoryCompact.ts`
- `src/utils/sessionStorage.ts`
- `src/QueryEngine.ts`
- `src/query.ts`

如果概括成一句话，这套系统的思路是：

> 不把所有历史都叫 memory，而是把不同时间尺度、不同可信度、不同用途的上下文拆层治理。

## 1. 第一层记忆：`CLAUDE.md` 是静态项目记忆，不是聊天历史

`utils/claudemd.ts` 这部分设计很成熟。它不是简单读取一个项目根目录文件，而是支持：

- managed / user / project / local 多层来源
- 向上遍历目录树
- `@include`
- frontmatter `paths`
- 条件嵌套加载

这意味着 `CLAUDE.md` 承担的是“项目常识”和“工作约定”的角色，例如：

- 代码风格
- 仓库结构
- 团队偏好
- 当前项目特殊规则

它不是运行时经验，而是静态背景知识。把这类知识和对话历史分开，是非常重要的。

## 2. 第二层记忆：`MEMORY.md` / memdir 是长期经验库

`memdir/memdir.ts` 这一层，才更接近真正的长期记忆系统。

它的几个关键点：

- 用文件系统而不是数据库做记忆载体
- `MEMORY.md` 是索引入口，而不是把所有内容硬塞一个文件
- 对入口内容做截断控制
- 明确区分记忆分类与保存规范
- 明确强调“当前任务状态不要滥存到长期记忆里”

这一点非常重要。很多 agent 会把当前任务细节和长期经验混在一起，结果长期记忆越来越脏。这里的设计在结构上就试图避免这件事。

它的潜台词是：

- 长期记忆应该保存“下次还值得用”的东西
- 当前任务细节更适合放在 plan、task、session memory

这是一种“记忆治理”思路，而不是“能存就多存”。

## 3. 记忆目录和自动记忆路径不是小细节，而是操作系统级架构选择

`memdir/paths.ts` 里面有一套完整的路径和启用逻辑，包括：

- auto-memory 是否启用
- base dir 如何确定
- override path
- remote memory dir
- daily log path

这说明系统不是临时在 prompt 里加一段“remember this”。它把 memory 当成正式的磁盘资源和运行时设施。

好处有两个：

### 3.1 记忆是可检查、可迁移、可审计的

文件型存储比黑盒向量库更适合工程环境，因为：

- 能直接看
- 能版本化
- 能排查错误
- 能做团队协作

### 3.2 记忆可以天然被别的机制复用

比如 agent memory、daily log、relevant memory surfacing 都能建立在同一套路径体系上。

## 4. Relevant Memories：不是全量回灌，而是相关记忆检索

`findRelevantMemories.ts` 很关键。它不是把 memdir 全部塞回 prompt，而是：

- 扫描记忆元数据
- 基于 manifest 和 sideQuery 选择相关条目
- 控制返回数量
- 避免把最近工具使用类噪声文档过度注入

这背后的竞争力很明显：

- 它不假设“记忆越多越好”
- 它假设“只有跟当前问题相关的记忆才值得占上下文”

这是成熟 agent 系统必须有的观念。否则长期记忆最后只会演化成上下文污染源。

## 5. Relevant Memory Prefetch：把记忆检索放到主循环旁路异步执行

`query.ts` 和 `attachments.ts` 里还有一个很强的设计点：`startRelevantMemoryPrefetch(...)`。

也就是说，系统不是每次都同步等待记忆检索完成，而是：

1. 用户提问后先启动相关记忆预取
2. 主流程继续推进
3. 在真正发起模型调用前，尽量消费已经完成的 memory prefetch 结果

这个思路很像现代系统里的 speculative execution。

它的好处是：

- 降低等待时间
- 保留相关记忆增强
- 把“检索”从阻塞路径上尽量移走

这不是单纯的 memory feature，而是性能工程。

## 6. 自动记忆提炼不是同步做，而是后台 agent 做

`services/extractMemories/extractMemories.ts` 是很能体现工程功力的一块。

它的思路不是“每轮结束让主模型顺手总结一下”，而是：

- 在完整 query 结束后后台执行
- 用 forked agent 去做 durable memory extraction
- 明确限制它能用的工具
- Read / Grep / Glob 可以用
- Bash 只允许只读场景
- Edit / Write 只允许写 auto memory 目录
- 如果主 agent 已经写了 memory 文件，则跳过重复提炼

这非常强，因为它实现了两个分离：

### 6.1 主执行与记忆沉淀分离

主 agent 专注当前任务，记忆提炼作为 sidecar 异步进行，不打断主链路。

### 6.2 记忆生成权限与业务权限分离

提炼记忆的 agent 没有全仓库修改权限，它只能在限定目录工作。

这极大降低了“为了记忆而误改业务代码”的风险。

## 7. Session Memory：这才是短期工作记忆

`services/SessionMemory/sessionMemory.ts`、`prompts.ts`、`sessionMemoryUtils.ts` 构成了另一层记忆：会话工作记忆。

它保存的不是长期经验，而是当前长任务中的结构化状态，包括：

- Session Title
- Current State
- Task specification
- Files and Functions
- Workflow
- Errors & Corrections
- Codebase and System Documentation
- Learnings
- Key results
- Worklog

这个设计特别成熟，因为它把“工作记忆”从聊天 transcript 里抽离出来了。

很多 agent 做长任务时，真正缺的不是长期记忆，而是：

- 当前做到哪了
- 哪些文件已经分析过
- 哪些假设被证伪了
- 哪些坑已经踩过

session memory 恰好在解决这个问题。

## 8. Session Memory 的触发不是随意的，而是阈值治理

这一层还有几个很强的工程细节：

- 最低 token 数才初始化
- 两次更新之间有最小 token 间隔
- 工具调用数量参与判断
- 如果最后一轮 assistant 还在工具调用态，暂不抽取

这说明它非常清楚：

- 过度频繁更新 session memory 会浪费上下文和算力
- 太早抽取会得到不稳定状态
- 工具调用进行中的阶段，不适合沉淀“当前结论”

这不是一个“有就行”的功能，而是经过成本和稳定性权衡的产物。

## 9. Session Memory Compact：压缩时先保工作记忆，而不是直接硬摘要

`sessionMemoryCompact.ts` 是另一个关键点。长会话快超窗口时，它不是一上来把整个历史粗暴摘要，而是优先尝试 session memory compact。

原因很明确：

- transcript 是原始过程
- session memory 是结构化工作状态

在长期任务里，后者往往比完整聊天历史更有价值。因此先保 session memory，相当于优先保“可执行状态”。

这比常见的“把前面对话总结成一段文字”更聪明。

## 10. Transcript Persistence：恢复能力本身也是记忆系统的一部分

`QueryEngine.ts` 和 `sessionStorage.ts` 这块很值得重视。它会：

- 早期持久化 transcript
- 记录 compact 边界
- 处理 snip removal
- 恢复 sidechain transcript
- 校验 resume 一致性
- 恢复 plan、agent metadata 等运行信息

这里的关键思想是：

> 记忆不只存在于 prompt 中，也存在于可恢复的运行轨迹里。

很多系统把“恢复会话”理解成“把聊天记录读回来”。这里不是，它恢复的是一整套运行状态。

## 11. File Read State 其实也是一种记忆

`QueryEngine.ts`、`fileStateCache`、`FileReadTool` 和 `attachments.ts` 共同维护了已读文件状态。

这层信息包括：

- 哪些文件读过
- 读的是哪一版内容
- 之后文件是否被改动
- 是否需要把 changed files 再注入给模型

它不是人类意义上的“记忆”，但对代码 agent 来说，这是最关键的事实记忆之一。

因为模型真正要避免的是：

- 记得文件读过，但忘了已经变了
- 基于旧内容继续修改
- 对同一文件产生互相冲突的操作

所以文件状态缓存是一种非常实用、非常工程化的 memory。

## 12. 这套记忆系统为什么比常见 agent 强

常见 agent 的 memory 问题是把所有东西都混在一起：

- 系统规则
- 项目背景
- 用户历史
- 当前任务进度
- 代码状态
- 长期经验

这里把它拆成了：

- 静态项目记忆：`CLAUDE.md`
- 长期经验记忆：memdir / `MEMORY.md`
- 相关记忆检索：relevant memories
- 会话工作记忆：session memory
- 运行轨迹记忆：transcript persistence
- 文件事实记忆：read file state

这意味着每层都能用不同策略处理：

- 加载时机不同
- 摘要策略不同
- 写入权限不同
- 保鲜周期不同

这是少数真正理解“上下文治理”而不是“上下文堆叠”的实现。

## 13. 核心亮点总结

### 亮点一：把记忆做成多层体系，而不是单点功能

系统能区分什么该长期保存，什么只是当前工作状态，什么只是运行轨迹。

### 亮点二：把检索、提炼、压缩、恢复全部制度化

不是只有“读 memory”这一件事，而是完整覆盖：

- 记忆写入
- 记忆筛选
- 记忆预取
- 记忆压缩
- 会话恢复

### 亮点三：把磁盘文件、会话状态、代码事实统一纳入记忆范畴

这使得 memory 不再只是 prompt engineering，而变成 runtime engineering。

## 14. 结论

这个仓库在记忆上的真正竞争力，不是“存更多”，而是“分得更清、用得更准、写得更稳、恢复得更完整”。

它把 memory 从一个模糊概念，拆成了：

- 项目知识层
- 长期经验层
- 当前工作层
- 代码事实层
- 运行恢复层

这套分层，是整个系统能做复杂长任务的基础。
