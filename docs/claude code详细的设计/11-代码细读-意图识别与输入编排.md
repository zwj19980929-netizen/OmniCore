# 代码细读：意图识别与输入编排

## 先给结论

这一层最值得抄的不是某个 prompt，而是它把“意图识别”拆成了一条正式的输入编排链：

1. 先清洗输入。
2. 再识别显式协议输入。
3. 再补隐式附件和上下文。
4. 再做技能与工具发现。
5. 最后才进入主 query loop。

所以它真正做的不是 intent classification，而是 input orchestration。

---

## 1. 主调用链长什么样

从大框架看，代码路径大致是：

```text
QueryEngine.ask()
  -> processUserInput()
    -> processUserInputBase()
      -> getAttachmentMessages()
        -> getAttachments()
  -> query()
    -> queryLoop()
      -> startRelevantMemoryPrefetch()
      -> startSkillDiscoveryPrefetch()
      -> model sampling / tool loop
```

对应的主要文件：

- `QueryEngine.ts`
- `utils/processUserInput/processUserInput.ts`
- `utils/attachments.ts`
- `query.ts`
- `utils/toolSearch.ts`
- `tools/ToolSearchTool/ToolSearchTool.ts`
- `tools/SkillTool/SkillTool.ts`

这条链条意味着：输入不是进来就直接请求模型，中间要先经历一层运行时改写。

## 2. `processUserInput()` 是最外层调度壳

从 `utils/processUserInput/processUserInput.ts` 可以抽出这样一个骨架：

```ts
async function processUserInput(args) {
  if (mode === 'prompt' && input is string && !isMeta) {
    setUserInputOnProcessing(input)
  }

  result = await processUserInputBase(...)
  if (!result.shouldQuery) return result

  for await (hookResult of executeUserPromptSubmitHooks(...)) {
    if (hookResult.blockingError) {
      return warningMessageOnly()
    }
    if (hookResult.preventContinuation) {
      return stopButKeepOriginalPrompt()
    }
    if (hookResult.additionalContexts.length > 0) {
      result.messages.push(createAttachmentMessage(...))
    }
    if (hookResult.message) {
      result.messages.push(convertHookMessageToRuntimeMessage(...))
    }
  }

  return result
}
```

这一层有三个很重要的设计点：

### 2.1 它先处理“输入生命周期”，不是先处理语义

一进来就会把用户输入显示到 UI，说明这里先考虑的是“交互过程”和“当前轮是否继续”，而不是立刻理解语义。

### 2.2 它给 hooks 留了正式插槽

`executeUserPromptSubmitHooks()` 让这条链可插拔。也就是说，输入进入主循环前，还有一层可扩展的阻断、改写、注入机会。

### 2.3 它把“继续执行”当成显式布尔状态

`shouldQuery` 很关键。它意味着：

- 有些输入只需要本地处理，不需要调用模型
- 有些输入可能被 hook 拦截
- 有些输入经过 slash command 后直接返回

这里已经不是简单的文本聊天管线了。

---

## 3. `processUserInputBase()` 才是真正的第一层路由器

这个函数虽然长，但可以抽成一个比较清晰的伪代码：

```ts
async function processUserInputBase(...) {
  normalizeInputBlocks()
  handlePastedTextAndImages()

  if (input startsWith '/' and slash commands allowed) {
    command = parseSlashCommand(...)
    if (command exists and bridge conditions satisfied) {
      return runLocalOrPromptSlashCommand(command)
    }
  }

  if (preExpansionInput contains ultraplan keyword) {
    routeToUltraPlan()
  }

  attachmentMessages = skipAttachments
    ? []
    : await getAttachmentMessages(...)

  userMessage = await processTextPrompt(...)

  return {
    messages: [...attachmentMessages, userMessage],
    shouldQuery: true,
    allowedTools,
    maybeUpdatedModel,
  }
}
```

可以看到，它在做的是“把一段输入翻译成 runtime 能消费的消息序列”。

这层不是把一句话分类成：

- 写代码
- 查问题
- 解释代码

而是把输入拆成：

- 斜杠命令
- 纯文本 prompt
- 附件消息
- 模式切换
- 特殊关键词信号

这就是为什么我说它做的是路由，不是分类。

---

## 4. `processTextPrompt()` 只负责把普通文本变成“可追踪的用户消息”

`utils/processUserInput/processTextPrompt.ts` 这层的职责很克制，大致可以理解为：

```ts
function processTextPrompt(input, context) {
  logPromptTelemetry()
  detectContinueSignals()
  detectNegativeSignals()
  return createUserMessage({
    content: normalizedPrompt,
    metadata: promptState,
  })
}
```

这说明作者把不同职责切得很开：

- `processUserInputBase()` 负责路由
- `processTextPrompt()` 负责把自然语言 prompt 编码成消息对象

这种分层有一个很大的好处：后续要增强“纯文本 prompt 的特征提取”时，不会污染 slash command 和附件逻辑。

## 5. `getAttachments()` 是第二层、更关键的意图增强器

`utils/attachments.ts` 才是真正让这套系统强起来的地方。可以把 `getAttachments()` 简化成下面这种结构：

```ts
async function getAttachments(input, toolUseContext, ideSelection, queuedCommands, messages) {
  if (attachments disabled) {
    return getQueuedCommandAttachments(queuedCommands)
  }

  abortController = timeout(1000ms)
  context = { ...toolUseContext, abortController }
  isMainThread = !toolUseContext.agentId

  userInputAttachments = [
    processAtMentionedFiles(input),
    processMcpResourceAttachments(input),
    processAgentMentions(input),
    maybeTurnZeroSkillDiscovery(input),
  ]
  await Promise.all(userInputAttachments)

  allThreadAttachments = [
    queued_commands,
    date_change,
    ultrathink_effort,
    deferred_tools_delta,
    agent_listing_delta,
    mcp_instructions_delta,
    changed_files,
    nested_memory,
    dynamic_skill,
    skill_listing,
    plan_mode,
    plan_mode_exit,
    auto_mode,
    todo_or_task_reminders,
    teammate_mailbox,
    team_context,
    agent_pending_messages,
    critical_system_reminder,
    compaction_reminder,
    context_efficiency,
  ]

  if (isMainThread) {
    add [
      ide_selection,
      ide_opened_file,
      output_style,
      diagnostics,
      lsp_diagnostics,
    ]
  }

  return flatten(await Promise.all(all attachments))
}
```

这里非常关键的一点是：附件不是配菜，而是主路由的一部分。

因为系统在这里做的其实是：

- 识别用户显式提及了什么
- 识别当前运行时缺什么上下文
- 自动补上这次决策需要的状态信息

这比“把用户原话交给模型”强得多。

---

## 6. 这个附件体系为什么等于意图识别升级版

传统意图识别通常会问：

- 用户这句话属于哪个 intent

这里的系统会问：

- 用户提到了哪些文件
- 需要不要补相关 memory
- 当前是不是 plan mode
- 有没有 pending agent 消息
- 该不该展示 deferred tools delta
- 当前是否已有 changed files 需要回灌
- 当前是否已经有 task/todo 应当提醒

换句话说，它不是识别“句子的意图”，而是在识别“当前执行现场的缺口”。

这是更强也更稳定的路径。

## 7. `queryLoop()` 里还有第三层意图增强

很多人会忽略 `query.ts`。其实它不是模型调用壳，它也在继续做输入编排。

从 `queryLoop()` 可以抽出这一段关键流程：

```ts
async function* queryLoop(params) {
  state = {
    messages,
    toolUseContext,
    autoCompactTracking,
    pendingToolUseSummary,
    ...
  }

  using pendingMemoryPrefetch = startRelevantMemoryPrefetch(messages, toolUseContext)

  while (true) {
    pendingSkillPrefetch = startSkillDiscoveryPrefetch(null, messages, toolUseContext)

    messagesForQuery = getMessagesAfterCompactBoundary(messages)
    messagesForQuery = applyToolResultBudget(messagesForQuery, contentReplacementState)
    messagesForQuery = maybeHistorySnip(messagesForQuery)
    messagesForQuery = maybeMicrocompact(messagesForQuery)
    messagesForQuery = maybeAutocompact(messagesForQuery)

    // 然后才真正发起模型调用
    response = sample(messagesForQuery, systemPrompt, userContext, systemContext)
    ...
  }
}
```

这个 loop 告诉我们一个重要事实：

> 真正送进模型的上下文，是 runtime 多次加工后的结果，而不是用户输入的直接投影。

所以意图识别不是在用户输入的那一刻完成，而是在主循环里持续重算。

## 8. Tool Search 把“要用哪个工具”从 prompt 常量变成检索问题

`utils/toolSearch.ts` 和 `tools/ToolSearchTool/ToolSearchTool.ts` 配合后，可以把这部分理解成：

```ts
function getToolSearchMode() {
  // standard / tst / tst-auto
}

async function isToolSearchEnabled(model, tools, messages) {
  if (mode === 'standard') return false
  if (mode === 'tst') return true
  if (mode === 'tst-auto') {
    return toolDescriptionsTooLong() || contextOverThreshold()
  }
}

class ToolSearchTool {
  search(query) {
    return tools
      .matchByName()
      .matchByKeyword()
      .matchBySearchHint()
      .matchByMcpName()
  }
}
```

它的核心竞争力是：

- 不把全部工具 schema 永久塞进 prompt
- 先缩成“工具检索问题”
- 真正需要时再暴露工具定义

这解决的是提示词规模和工具扩展规模的双重问题。

## 9. Skill discovery 不是锦上添花，而是策略层能力路由

`SkillTool.ts` 和 `loadSkillsDir.ts` 配合起来看，大致是这样：

```ts
if (turnZero && input carries strong signal) {
  prefetchSkillDiscovery(input, messages, context)
}

if (interTurn && writePivotDetected) {
  startSkillDiscoveryPrefetch(messages, context)
}

if (skill selected) {
  run skill in current context or forked subagent context
}
```

这意味着 skill 并不只是一个静态模板，而是：

- 会被发现
- 会被路由
- 会被执行

系统不是简单问“这句话什么意思”，而是问“有没有一套现成策略比让主模型临时思考更合适”。

## 10. 这一层最值得抄的几段伪代码

### 10.1 输入路由总骨架

```ts
function routeUserTurn(input) {
  normalized = normalize(input)

  if (isExplicitCommand(normalized)) {
    return runCommand(normalized)
  }

  attachments = collectAttachments(normalized, runtimeState)
  promptMessage = buildUserPrompt(normalized)

  return queryLoop({
    messages: [...attachments, promptMessage],
    runtimeState,
  })
}
```

### 10.2 附件增强骨架

```ts
function collectAttachments(input, state) {
  explicit = [
    atMentionedFiles(input),
    mcpMentions(input),
    agentMentions(input),
  ]

  implicit = [
    changedFiles(state),
    nestedMemory(state),
    relevantPlanState(state),
    taskReminders(state),
    pendingAgentMessages(state),
    deferredToolsDelta(state),
  ]

  return explicit + implicit
}
```

### 10.3 查询前再整形一次上下文

```ts
function prepareMessagesForSampling(messages, state) {
  messages = applyToolResultBudget(messages, state)
  messages = maybeSnip(messages, state)
  messages = maybeMicrocompact(messages, state)
  messages = maybeAutocompact(messages, state)
  messages = mergeMemoryPrefetch(messages, state)
  messages = mergeSkillPrefetch(messages, state)
  return messages
}
```

## 11. 为什么这套输入编排比外面的常见实现更强

很多系统的输入流：

```text
用户文本 -> 模型 -> 工具选择
```

这个仓库的输入流：

```text
用户文本
  -> 规范化
  -> 显式命令分流
  -> 附件增强
  -> 运行时状态补全
  -> 技能/工具发现
  -> 上下文预算整形
  -> 模型
  -> 工具
```

这就是它更稳定的原因。它把大量原本靠模型猜的事情，前移成了 runtime 责任。
