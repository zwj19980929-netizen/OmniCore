# 代码细读：记忆系统与上下文治理

## 先给结论

这个项目的记忆系统最强的地方，不是“记得久”，而是把不同类型的记忆拆开治理。它至少拆了这几层：

1. 静态项目记忆：`CLAUDE.md`
2. 长期经验记忆：`memdir` / `MEMORY.md`
3. 相关记忆检索：relevant memories
4. 会话工作记忆：session memory
5. 运行轨迹记忆：transcript persistence / resume
6. 文件事实记忆：`readFileState`

如果你从代码角度看，这其实已经是一套 context operating system。

---

## 1. `CLAUDE.md` 负责“项目常识层”

这一层主要在 `utils/claudemd.ts`。虽然这里我不逐行展开，但可以把它的加载逻辑抽成下面的伪代码：

```ts
function loadClaudeMdContext(cwd) {
  result = []

  result += managedClaudeMd()
  result += userClaudeMd()
  result += projectClaudeMd()

  for (dir from cwd upward to project root) {
    if (dir contains CLAUDE.md) {
      result += parseClaudeMd(dir / CLAUDE.md)
    }
  }

  result += resolveIncludes()
  result += applyFrontmatterPathsRules()
  result += conditionalNestedRules()

  return result
}
```

这层存的是“规则”和“背景”，不是任务过程。比如：

- 项目约束
- 团队偏好
- 代码风格
- 仓库特殊结构

换句话说，`CLAUDE.md` 更像长期稳定的项目操作手册。

## 2. `memdir/memdir.ts` 才是真正的长期记忆内核

这一层最关键的函数有：

- `truncateEntrypointContent()`
- `buildMemoryLines()`
- `buildMemoryPrompt()`
- `loadMemoryPrompt()`

### 2.1 `buildMemoryLines()` 在做什么

从代码看，这个函数不是简单拼几句说明，而是在构造一整套“如何使用长期记忆”的行为协议：

```ts
function buildMemoryLines(displayName, memoryDir, extraGuidelines, skipIndex) {
  lines = [
    `You have a persistent, file-based memory system at ${memoryDir}`,
    'save memories when user explicitly asks',
    'forget by deleting relevant entry',
    TYPES_SECTION,
    WHAT_NOT_TO_SAVE_SECTION,
    HOW_TO_SAVE_SECTION,
    WHEN_TO_ACCESS_SECTION,
    TRUSTING_RECALL_SECTION,
    'use plan/tasks for current-conversation state instead of memory',
  ]

  lines += buildSearchingPastContextSection(memoryDir)
  return lines
}
```

最值得注意的是这一句思想：

```text
当前任务里的计划变化、执行步骤，不应该优先写进 memory，而应该去 plan / task。
```

这意味着作者非常清楚 memory 容易被滥用成“任务暂存区”，所以在 prompt 级别就强制做分工。

### 2.2 `buildMemoryPrompt()` 在做什么

它的逻辑大致是：

```ts
function buildMemoryPrompt({ displayName, memoryDir }) {
  entrypoint = memoryDir + '/MEMORY.md'
  entrypointContent = readFileIfExists(entrypoint)

  lines = buildMemoryLines(displayName, memoryDir)

  if (entrypointContent not empty) {
    t = truncateEntrypointContent(entrypointContent)
    logMemoryDirCounts(...)
    lines += ['## MEMORY.md', t.content]
  } else {
    lines += ['## MEMORY.md', 'Your MEMORY.md is currently empty']
  }

  return lines.join('\n')
}
```

注意这里的关键点：

- `MEMORY.md` 是索引入口
- 内容会被截断
- 系统会记录长度和是否截断

这说明 `MEMORY.md` 不是随便乱长的知识库，而是受预算控制的“长期记忆入口页”。

## 3. 长期记忆为什么不是数据库，而是文件系统

从 `memdir/paths.ts`、`memdir/memoryScan.ts`、`memdir/findRelevantMemories.ts` 来看，这套系统坚持把 memory 设计成文件系统资源。

大致等于：

```ts
autoMemDir = resolveAutoMemoryPath()
ensurePathSafe(autoMemDir)
scanMemoryFiles(autoMemDir, limit=200)
readFrontmatterOnlyForRanking()
selectRelevantMemories()
```

它的优点非常实际：

- 人类可读
- 可版本化
- 可审计
- 可迁移
- 可被子代理共享

这比黑盒 memory store 更适合工程现场。

## 4. Relevant Memory 不是全量回灌，而是相关性筛选

`findRelevantMemories.ts` 的核心思想可以抽成：

```ts
async function findRelevantMemories(query, memoryManifest, recentTools) {
  candidates = scanRecentMemoryFrontmatters(limit=200)
  filtered = removeObviousNoise(candidates, recentTools)
  selected = sideQueryRank(query, filtered)
  return selected.slice(0, 5)
}
```

注意这个思路很成熟：

- 先扫描元数据，不急着读全文
- 先去掉噪声
- 再让 sideQuery 帮忙筛相关性
- 最后强行限制返回数量

这和“把 memory 全都塞回 prompt”是完全不同的系统哲学。

## 5. `startRelevantMemoryPrefetch()` 把 memory 检索变成旁路异步任务

`query.ts` 里有一段非常值得注意：

```ts
using pendingMemoryPrefetch = startRelevantMemoryPrefetch(
  state.messages,
  state.toolUseContext,
)
```

可以抽成伪代码：

```ts
function startRelevantMemoryPrefetch(messages, context) {
  if (!autoMemoryEnabled) return noopPrefetch

  return new MemoryPrefetch(async () => {
    surfaced = collectSurfacedMemories(messages)
    recentTools = collectRecentSuccessfulTools(messages)
    candidates = await findRelevantMemories(messages, surfaced, recentTools)
    return await readMemoriesForSurfacing(candidates)
  })
}
```

主循环里再在合适时机消费：

```ts
if (pendingMemoryPrefetch.settled()) {
  messages += pendingMemoryPrefetch.result()
}
```

这是一种非常现代的做法：把昂贵检索前移，但不阻塞主线程。

## 6. `SessionMemory` 负责“当前长任务工作记忆”

这部分主要在：

- `services/SessionMemory/sessionMemory.ts`
- `services/SessionMemory/prompts.ts`
- `services/SessionMemory/sessionMemoryUtils.ts`

这一层的目标不是保存长期偏好，而是保存：

- 当前任务目标
- 当前做到哪
- 看过哪些文件
- 走过哪些错误路径
- 得到了哪些阶段结论

### 6.1 `shouldExtractMemory()` 是如何决定要不要提炼

这段代码非常值得抄。它的逻辑几乎可以直接写成：

```ts
function shouldExtractMemory(messages) {
  currentTokenCount = tokenCountWithEstimation(messages)

  if (!sessionMemoryInitialized) {
    if (!hasMetInitializationThreshold(currentTokenCount)) {
      return false
    }
    markSessionMemoryInitialized()
  }

  hasMetTokenThreshold = hasMetUpdateThreshold(currentTokenCount)
  toolCallsSinceLastUpdate = countToolCallsSince(messages, lastMemoryMessageUuid)
  hasMetToolCallThreshold = toolCallsSinceLastUpdate >= getToolCallsBetweenUpdates()
  hasToolCallsInLastTurn = hasToolCallsInLastAssistantTurn(messages)

  shouldExtract =
    (hasMetTokenThreshold && hasMetToolCallThreshold) ||
    (hasMetTokenThreshold && !hasToolCallsInLastTurn)

  if (shouldExtract) {
    lastMemoryMessageUuid = lastMessage.uuid
    return true
  }

  return false
}
```

这一段的设计特别成熟，原因是它同时控制了三件事：

- 初始化门槛
- 两次更新之间的最小代价
- 不要在工具调用中途乱提炼

换句话说，它很清楚“记忆提炼本身也是昂贵操作”。

### 6.2 `extractSessionMemory()` 如何工作

主逻辑大致是：

```ts
async function extractSessionMemory(context) {
  if (querySource !== 'repl_main_thread') return
  if (!sessionMemoryGateEnabled) return
  initSessionMemoryConfigIfNeeded()
  if (!shouldExtractMemory(messages)) return

  markExtractionStarted()

  setupContext = createSubagentContext(toolUseContext)
  { memoryPath, currentMemory } = await setupSessionMemoryFile(setupContext)

  userPrompt = await buildSessionMemoryUpdatePrompt(currentMemory, memoryPath)

  await runForkedAgent({
    promptMessages: [createUserMessage({ content: userPrompt })],
    cacheSafeParams: createCacheSafeParams(context),
    canUseTool: createMemoryFileCanUseTool(memoryPath),
    querySource: 'session_memory',
    forkLabel: 'session_memory',
    overrides: { readFileState: setupContext.readFileState },
  })
}
```

这段实现说明了两个非常强的点：

#### 6.2.1 会话记忆提炼由独立 forked agent 完成

主线程不自己边做任务边顺手总结，而是交给 sidecar agent。

#### 6.2.2 这个 sidecar agent 的权限被刻意收窄

它不是拿全仓库写权限去乱写，而是只允许写 session memory 文件。

这就是为什么这套系统能长期跑而不容易自污染。

## 7. 自动 durable memory 提炼和 session memory 是两条线

`services/extractMemories/extractMemories.ts` 和 Session Memory 不一样。

可以简单理解为：

- Session memory：面向“当前长任务”
- durable memory extraction：面向“未来轮次也有价值的长期经验”

其骨架大致是：

```ts
async function extractDurableMemoryAfterQuery(queryRange, context) {
  if (mainAgentAlreadyWroteMemoryFiles(queryRange)) return

  await runForkedAgent({
    promptMessages: [memoryExtractionPrompt],
    canUseTool: allowOnly([
      Read,
      Grep,
      Glob,
      readOnlyBash,
      Edit/Write within autoMemoryDir,
    ]),
    querySource: 'memory_extraction',
  })
}
```

这两种记忆分开，是这个仓库非常成熟的地方。因为：

- 当前任务状态不一定值得长期保留
- 长期经验也不该污染当前工作记忆

## 8. `QueryEngine` 的 transcript 持久化也是记忆体系的一部分

`QueryEngine.ts` 里有一段非常关键的逻辑：在进入 query loop 之前，先把用户消息落盘。

其逻辑大概是：

```ts
messagesFromUserInput = await processUserInput(...)
mutableMessages.push(...messagesFromUserInput)

if (persistSession && messagesFromUserInput.length > 0) {
  transcriptPromise = recordTranscript(messages)
  if (bareMode) {
    fireAndForget(transcriptPromise)
  } else {
    await transcriptPromise
    maybeFlushSessionStorage()
  }
}
```

这段代码的意义很大：

- 就算模型请求还没返回
- 只要用户输入被接受
- transcript 就已经可恢复

也就是说，这里记住的不是“对话文本”，而是“运行轨迹边界”。

## 9. `readFileState` 其实也是记忆，只是很多人没意识到

从 `QueryEngine.ts`、`FileReadTool.ts`、`attachments.ts` 可以看出，系统一直在维护：

- 文件是否读过
- 读的是哪个时间点
- 是否只读了 partial view
- 文件后来有没有变化

这可以抽成：

```ts
readFileState[filePath] = {
  timestamp,
  content,
  offset,
  limit,
  isPartialView,
}
```

这层记忆和“用户偏好”完全不是一回事，但对代码 agent 反而更关键。因为它决定了：

- 当前代码事实是否还成立
- 编辑动作是否安全

## 11. 为什么这套记忆系统比外面强

很多系统把 memory 理解成一句话：

```text
把旧历史加回 prompt
```

这个仓库的理解更像：

```text
不同层次的事实，需要不同持久化方式、不同注入时机、不同压缩策略、不同权限边界
```

这就是它能做复杂长任务的真正原因。
