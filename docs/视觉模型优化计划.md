# 视觉模型优化计划

> 创建日期: 2026-03-24
> 状态: ✅ 全部完成 (2026-03-25)

## 一、现状分析

### 当前架构

视觉模型在浏览器 Agent 中有 6 个集成点，但全部是**被动兜底**角色：

| 集成点 | 触发条件 | 实际触发率 | 问题 |
|--------|----------|-----------|------|
| 感知描述 | 页面复杂度 > 0.4 | 极低 | 阈值太保守，绝大多数页面不触发 |
| 视觉行动决策 | DOM 决策返回 WAIT/None | 低 | 系统已经卡住才调，往往来不及 |
| 视觉数据提取 | DOM + 文本提取都失败 | 极低 | 第三层兜底，很少走到 |
| 页面相关性 | step>=2 且无数据且 page_type=unknown | 低 | 条件过于严格 |
| Critic 视觉验证 | LLM 审查打低分 | 低 | 只做二次确认 |
| 验证码识别 | 检测到验证码 | 视场景而定 | 功能正常 |

### 核心问题

1. **视觉模型太被动** — 只在"系统已经卡住"时才介入，这时状态已经乱了，视觉模型也难救回来
2. **感知描述几乎不触发** — `compute_complexity_score()` 基于 a11y 元素命名率算复杂度，阈值 0.4 在实际运行中几乎不会被触发，导致决策 prompt 里的 `{vision_description}` 几乎永远是空字符串
3. **没有操作验证** — 点击、填写后仅靠 DOM diff 判断是否生效，无法检测视觉变化（toast、颜色变化、动画等）
4. **WAIT 是盲等** — 等待期间不知道页面是否在变化，可能在静态页面上白等

## 二、优化方案

按优先级排列，共 4 个优化项。

### 优化 1: 新页面自动视觉感知（优先级: P0）

**目标**: 让决策 prompt 里的 `{vision_description}` 不再是空字符串

**现状**:
- `observe()` 中仅在 `complexity > 0.4 or page_type == "unknown"` 时才调 `get_vision_description()`
- 实际运行中绝大多数页面复杂度低于阈值，vision_description 几乎永远为空

**方案**:
在 `browser_perception.py` 的 `observe()` 方法中，增加"新 URL 首次访问"触发条件：

```python
# 现有逻辑
if complexity > settings.VISION_PERCEPTION_COMPLEXITY_THRESHOLD or page_type == "unknown":
    vision_description = await self.get_vision_description(page)

# 改为
is_new_page = self._is_new_url(url)  # 对比上一次 observe 的 URL
if is_new_page or complexity > settings.VISION_PERCEPTION_COMPLEXITY_THRESHOLD or page_type == "unknown":
    vision_description = await self.get_vision_description(page)
```

**改动范围**:
- `agents/browser_perception.py`:
  - 新增 `_last_vision_url` 实例变量
  - 新增 `_is_new_url()` 方法（对比 normalized URL）
  - 修改 `observe()` 中的触发条件（约 466-477 行）
- 新增配置项 `VISION_ON_NEW_PAGE = true`（可通过环境变量关闭）

**成本**: 每个新页面增加一次轻量视觉调用（gemini-2.0-flash, max_tokens=300, ~0.3s）
**收益**: 决策层（deepseek-reasoner）能看到页面视觉描述，减少对纯 DOM 理解的依赖

### 优化 2: 操作后视觉验证（优先级: P0）

**目标**: 关键操作后用截图对比确认操作是否生效

**现状**:
- `_verify_action_effect()` 委托给 execution 层，仅做 DOM 状态对比
- 无法检测纯视觉变化：toast 提示、颜色变化、loading 动画消失、弹窗出现等
- 验证失败时进入 recovery 流程，可能在已经成功的操作上反复重试

**方案**:
在 `_verify_action_effect()` 中，当 DOM diff 不确定时，增加视觉验证：

```python
async def _verify_action_effect(self, before: Dict, action: BrowserAction) -> bool:
    dom_verified = await self.execution.verify_action_effect(before, action, self._get_semantic_snapshot)
    if dom_verified:
        return True
    # DOM 不确定时，用视觉对比
    after_screenshot = await self._capture_screenshot()
    if self._before_screenshot and after_screenshot:
        changed = self._screenshots_differ(self._before_screenshot, after_screenshot)
        if changed:
            return True  # 视觉上页面变了，认为操作生效
    return False
```

**改动范围**:
- `agents/browser_agent.py`:
  - 在操作执行前保存截图 `self._before_screenshot`
  - 修改 `_verify_action_effect()` 加视觉对比分支
  - 新增 `_screenshots_differ()` 方法（像素级差异检测，不需要调 LLM）
- `utils/image_diff.py`（新文件）:
  - `compute_pixel_diff(img_a: bytes, img_b: bytes) -> float` — 返回差异比例 0.0-1.0
  - 阈值 > 0.02（2% 像素变化）视为有变化
  - 使用 Pillow 实现，不引入新依赖

**成本**: 每次操作后额外一次截图（~50ms），像素对比（~10ms），无 LLM 调用
**收益**: 避免在已成功的操作上做无意义的 recovery 重试

### 优化 3: WAIT 期间视觉变化检测（优先级: P1）

**目标**: 用截图对比判断页面是否在变化，避免在静态页面上白等

**现状**:
- 决策返回 WAIT 时，固定等待后重新 observe
- 不知道页面是否真的在加载/变化
- 可能在完全静态的页面上反复等待浪费步骤

**方案**:
在 WAIT 执行逻辑中加入截图对比循环：

```python
if action.action_type == ActionType.WAIT:
    before_img = await self._capture_screenshot()
    await asyncio.sleep(float(action.value or "1"))
    after_img = await self._capture_screenshot()
    if not self._screenshots_differ(before_img, after_img):
        # 页面没变化，不用继续等，直接进入下一步决策
        log_agent_action(self.name, "WAIT", "页面无视觉变化，跳过等待")
```

**改动范围**:
- `agents/browser_agent.py`: 主循环中 WAIT 处理逻辑（约 5291-5298 行）
- 复用优化 2 中的 `_screenshots_differ()` 方法

**成本**: 每次 WAIT 额外两次截图 + 一次像素对比（~110ms），无 LLM 调用
**收益**: 在静态页面上快速跳过无意义等待，节省步骤

### 优化 4: 连续截图进度感知（优先级: P2）

**目标**: 检测 Agent 是否陷入视觉循环（连续操作但页面无变化）

**现状**:
- 靠 `_action_signature()` 去重防止重复同一操作
- 但不同操作可能导致相同结果（点不同按钮都没反应）
- Agent 可能在视觉上已经陷入循环但 action signature 不同

**方案**:
维护最近 N 步的截图指纹，检测连续无变化：

```python
class VisualProgressTracker:
    def __init__(self, window_size: int = 3):
        self._recent_hashes: deque[str] = deque(maxlen=window_size)

    def record(self, screenshot: bytes) -> bool:
        """记录截图并返回是否有进度。连续 window_size 步截图相同则认为卡住。"""
        img_hash = self._perceptual_hash(screenshot)
        is_stuck = len(self._recent_hashes) >= self.window_size and all(
            h == img_hash for h in self._recent_hashes
        )
        self._recent_hashes.append(img_hash)
        return not is_stuck
```

**改动范围**:
- `agents/browser_agent.py`:
  - 新增 `VisualProgressTracker` 类（或放 `utils/` 下）
  - 主循环每步结束时调用 `tracker.record(screenshot)`
  - 检测到卡住时提前终止或切换策略
- 使用感知哈希（perceptual hash）而非精确像素对比，容忍微小差异

**成本**: 每步一次截图 + 哈希计算（~60ms），无 LLM 调用
**收益**: 更早发现 Agent 卡住，减少无效步骤消耗

## 三、实施计划

| 阶段 | 优化项 | 预计改动量 | 依赖 |
|------|--------|-----------|------|
| 第一阶段 | 优化 1: 新页面自动视觉感知 | ~30 行 | 无 |
| 第一阶段 | 优化 2: 操作后视觉验证 | ~60 行 + 新文件 image_diff.py ~40 行 | 无 |
| 第二阶段 | 优化 3: WAIT 视觉变化检测 | ~20 行 | 依赖优化 2 的 _screenshots_differ |
| 第三阶段 | 优化 4: 连续截图进度感知 | ~50 行 | 独立 |

### 测试策略

每个优化项需要：
1. 单元测试：验证触发条件、边界情况
2. 集成测试：在实际页面上验证视觉调用正常工作
3. 性能验证：确认额外截图/LLM 调用不显著拖慢整体速度
