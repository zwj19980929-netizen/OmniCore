# OmniCore Runtime 架构优化规划

> 编写时间：2026-04-01
>
> 灵感来源：Claude Code 智能体架构深度分析（见 `docs/claude code详细的设计/`）
> 
> 定位：不是新功能，而是**执行内核的工程化升级**——让已有功能跑得更稳、更省、更可控。

> 注意每个子功能优化完成后都要在本文档记录下，以免下个协作者做重复了
---

## 核心判断

OmniCore 当前的功能覆盖已经很广（MCP、Skill Library、RAG、多模态、事件驱动、多 Agent 协作），但 runtime 治理层还比较薄。对比 Claude Code 的设计，主要差距不在"能做什么"，而在"怎么做得稳"：

| 维度 | OmniCore 现状 | Claude Code 做法 | 差距 |
|------|-------------|-----------------|------|
| 执行循环 | graph.py 2200 行，编排+业务混杂 | 独立 AgentLoop，显式 LoopState 跨轮传递 | 可维护性、可恢复性 |
| 状态管理 | 30+ 字段扁平 dict，shared_memory 双写 | 分域子状态，每层有清晰边界 | 耦合度、并发安全 |
| 上下文成本 | 依赖模型窗口硬扛，无压缩策略 | 五级渐进压缩（budget→snip→micro→collapse→auto） | 长会话成本、稳定性 |
| 规划控制 | 计划是 LLM 输出的 JSON，执行后即丢 | Plan Mode 状态机 + 计划文件持久化 + reminder | 长任务质量 |
| 工具执行 | 字符串匹配工具类型，无能力声明 | 工具有结构化能力标签，并发安全标记 | 调度精度 |
| System Prompt | 所有信息混入 prompt | 静态前缀 + 动态 attachment lane 分离 | 缓存友好性 |

## R1: 上下文成本控制（Tool Result Budget + History Snip）

**优先级**：P0 — 直接降成本，防长会话崩溃

**问题**：当前工具返回结果（网页抓取、文件读取）原样塞入上下文，长会话消息历史无上限增长。一次浏览器抓取可能返回 50KB+ 文本，几轮下来 token 消耗暴增。

**Claude Code 做法**：五级渐进压缩，优先保细粒度上下文，逐级降级。其中 Tool Result Budget 和 History Snip 是投入产出比最高的两级。

**实现方案**：

### R1-1: Tool Result Budget（工具返回截断）

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | 新增 `utils/context_budget.py`，实现 `truncate_tool_result(text, max_chars=8000)` | `utils/context_budget.py`（新建） | ✅ |
| 2 | 截断策略：保留头部 60% + 尾部 30% + 中间插入 `[...truncated {n} chars...]` | 同上 | ✅ |
| 3 | 在 `core/task_executor.py` `_apply_task_outcome()` 中对工具输出应用截断 | `core/task_executor.py` | ✅ |
| 4 | `parallel_executor_node()` 通过 `_apply_task_outcome()` 间接覆盖，无需单独修改 | `core/graph.py` | ✅ |
| 5 | 配置项 `TOOL_RESULT_MAX_CHARS`（默认 8000） | `config/settings.py` | ✅ |
| 6 | 单元测试 | `tests/test_context_budget_unit.py`（新建） | ✅ |

### R1-2: History Snip（历史消息裁剪）

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | 在 `utils/context_budget.py` 新增 `snip_history(messages, max_messages=20, keep_recent=10)` | `utils/context_budget.py` | ✅ |
| 2 | 裁剪策略：保留最近 K 条完整消息 + 更早的消息只保留 role+摘要（前 200 字符） | 同上 | ✅ |
| 3 | 在 `core/graph.py` 每轮 LLM 调用前（`route_node`、`replanner_node`）应用 history snip | `core/graph.py` | ✅ |
| 4 | 配置项 `HISTORY_MAX_MESSAGES`（默认 20）、`HISTORY_KEEP_RECENT`（默认 10） | `config/settings.py` | ✅ |
| 5 | 单元测试 | `tests/test_context_budget_unit.py` | ✅ |

### R1 完成记录（2026-04-01）

**新建文件**
- `utils/context_budget.py` — 核心实现，含 `truncate_tool_result`、`truncate_result_dict`、`snip_history`
- `tests/test_context_budget_unit.py` — 17 个单元测试，全部通过

**修改文件**
- `config/settings.py` — 新增 `TOOL_RESULT_MAX_CHARS`（8000）、`HISTORY_MAX_MESSAGES`（20）、`HISTORY_KEEP_RECENT`（10）
- `core/task_executor.py` — `_apply_task_outcome()` 使用 `truncate_tool_result` 替代原 50KB 硬截断；同时对 raw result dict 的文本字段截断
- `core/graph.py` — `route_node`、`replanner_node` 入口处调用 `snip_history`

**实现说明**
- `truncate_result_dict` 额外对工具返回的 raw result dict（`content`/`output`/`text`/`extracted_text`/`stdout`/`stderr` 字段）截断，比原方案多一层保护
- `snip_history` 保留消息原始 ID，确保 LangGraph `add_messages` reducer 正确更新（覆盖而非追加）
- `state["messages"]` 目前主要作为审计日志（LLM 调用走 `shared_memory["conversation_history"]`），snip 限制其无限增长、为未来直接传 LLM 做准备

**验收标准**：
- 单次工具返回超过 `TOOL_RESULT_MAX_CHARS` 时自动截断，不丢关键信息（头尾保留）
- 长会话（>20 轮）的 token 消耗相比无截断降低 40%+
- 现有所有测试不受影响

## R2: 废弃 shared_memory，统一 MessageBus

**优先级**：P0 — 消除双写混乱，降低维护负担

**问题**：`shared_memory`（plain dict）和 `MessageBus` 并存，graph 节点有的写 shared_memory，有的写 message_bus，有的两边都写。`MessageBus.to_shared_memory()` 桥接逻辑复杂且有损。

**目标**：所有节点间通信统一走 `MessageBus`，彻底移除 `shared_memory` 字段。

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | 全局搜索 `shared_memory` 的所有读写点，建立迁移清单 | 全仓库 | 🔲 |
| 2 | 逐个将 `state["shared_memory"][key] = val` 替换为 `bus.publish(...)` | `core/graph.py`、`core/router.py`、`agents/*.py` | 🔲 |
| 3 | 逐个将 `state["shared_memory"].get(key)` 替换为 `bus.query(...)` 或 `bus.get_latest(...)` | 同上 | 🔲 |
| 4 | 移除 `MessageBus.to_shared_memory()` / `from_shared_memory()` 桥接方法 | `core/message_bus.py` | 🔲 |
| 5 | 从 `OmniCoreState` 中移除 `shared_memory` 字段 | `core/state.py` | 🔲 |
| 6 | 给 MessageBus 加 TTL（默认 30 分钟）和最大容量（默认 500 条） | `core/message_bus.py` | 🔲 |
| 7 | 全量回归测试 | `pytest tests -q` | 🔲 |

**验收标准**：
- `shared_memory` 在代码中零出现（除历史文档）
- `MessageBus` 内存增长可控（TTL + 容量限制）
- 全量测试通过

### R2 完成记录（2026-04-01）

**核心变更**

- `core/message_bus.py`：移除 `to_shared_memory()` / `from_shared_memory()` 桥接方法；新增 TTL（默认 1800s）+ 最大容量（默认 500 条）；新增 `to_snapshot()` 方法，将 MessageBus 内容导出为 worker 可消费的扁平 dict（含 `conversation_history`、`_approved_actions` 等 legacy key 映射）；`from_dict()` 从 settings 读取 TTL/容量配置
- `config/settings.py`：新增 `MESSAGE_BUS_TTL`（1800）、`MESSAGE_BUS_MAX_CAPACITY`（500）配置项
- `core/state.py`：移除 `shared_memory: Dict[str, Any]` 字段及 `create_initial_state()` 中的 `shared_memory={}`；注释说明 `message_bus` 为唯一节点间通信通道
- `core/graph.py`：`_bus_get_str()` / `_bus_get()` 移除 `legacy_key` 参数；所有 `state["shared_memory"]` 读写替换为 MessageBus publish/get_latest；replanner_node 的 `_replan_history` 写入 `MSG_REPLAN_HISTORY`；finalize_node 完全去除 `shared_memory` 变量
- `core/runtime.py`：`_execute_submitted_job()` 初始化改为 `bus.publish()` 批量写入 time/location/os/history 等上下文；`resume_job_from_checkpoint()` 和 `approve_waiting_job()` 也改用 MessageBus
- `core/router.py`：读取改用 `_bus_val(MSG_*)` 方式；`MSG_DIRECT_ANSWER` 写入走 `bus.publish()`，不再双写 shared_memory
- `core/task_executor.py`：`_apply_task_outcome()` 移除 `state["shared_memory"][task_id]` 写入；`run_ready_batch_async()` 改用 `bus.to_snapshot()` 生成 worker 快照
- `core/tool_adapters.py`：`_base_outcome()` 移除 `shared_memory` 参数及所有调用处的 `"shared_memory": None` 占位
- `utils/runtime_state_store.py`：checkpoint 序列化 key 由 `"shared_memory"` 改为 `"message_bus"`
- `agents/file_worker.py`、`system_worker.py`、`web_worker.py`、`terminal_worker.py`：`execute()` / `execute_async()` 入参由 `state["shared_memory"]` 改为 `bus.to_snapshot()`

**新增 MSG_* 常量**（均在 `core/message_bus.py`）

`MSG_TIME_CONTEXT`、`MSG_LOCATION_CONTEXT`、`MSG_OS_CONTEXT`、`MSG_CONVERSATION_HISTORY`、`MSG_RELATED_HISTORY`、`MSG_SESSION_ARTIFACTS`、`MSG_RESOURCE_MEMORY`、`MSG_SUCCESSFUL_PATHS`、`MSG_FAILURE_PATTERNS`、`MSG_WORK_SCOPE`、`MSG_MEMORY_SCOPE`、`MSG_RESUME_CHECKPOINT_ID`、`MSG_RESUME_REQUESTED_AT`、`MSG_APPROVAL_RESUMED_AT`

**测试迁移**（13 个测试文件）

`test_graph_policy_decision_unit.py`、`test_graph_approval_flow_unit.py`、`test_task_executor_runtime_loop_unit.py`、`test_tool_registry_unit.py`、`test_graph_replanner_unit.py`、`test_graph_finalize_unit.py`、`test_runtime_metrics_unit.py`（state dict 中 `shared_memory: {}` → `message_bus: []`）；`test_router_unit.py`（bus.publish 写入 / bus.get_latest 断言）；`test_mcp_unit.py`（`assert "shared_memory" not in outcome`）

**验收结果**
- `grep -rn 'state\["shared_memory"\]' core/ agents/ utils/`：**零匹配**（仅注释提及）
- R2 专项测试（9 个文件，98 个用例）：**98 通过，1 跳过**（那 1 个失败 `test_replanner_preserves_direct_user_target_url` 是 R4 之前的预存缺陷）
- `to_snapshot()` 保持 worker 接口不变，无破坏性改动

## R3: graph.py 拆分 + AgentLoop 独立

**优先级**：P1 — 可维护性大幅提升

**问题**：`core/graph.py` 2200+ 行，graph 编排逻辑与业务逻辑（mojibake 修复、Markdown 格式化、结构化发现提取）混杂。replanner_node 200 行、finalize_node 160 行内嵌在同一文件。节点间无显式接口契约。

**Claude Code 做法**：AgentLoop 是独立类，内置 LoopState 跨轮传递；辅助逻辑全部在 utils 里；每类错误有显式状态跳转。

| 2 | 新建 `core/graph_utils.py`，迁移辅助函数（`_repair_mojibake_text`、`_format_discovery_*`、`_should_skip_for_resume`、`_apply_adaptive_skip`） | `core/graph_utils.py`（新建） | 🔲 |
| 3 | 新建 `core/replanner.py`，将 `replanner_node` + `dynamic_replan_node` 逻辑抽成 `Replanner` 类 | `core/replanner.py`（新建） | 🔲 |
| 4 | `core/graph.py` 精简为纯粹的图定义（节点声明 + 边连接 + 条件路由），目标 <300 行 | `core/graph.py` | 🔲 |
| 5 | 引入 `LoopState` dataclass，追踪 replan_count、error_recovery_count、compact_applied 等跨轮状态 | `core/graph_nodes.py` | 🔲 |
| 6 | 节点间传参引入类型化接口（`NodeInput` / `NodeOutput` TypedDict） | `core/graph_nodes.py` | 🔲 |
| 7 | 全量回归测试 + 补充节点级单元测试 | `tests/` | 🔲 |

**验收标准**：
- `core/graph.py` < 300 行，只含图定义
- 各节点函数可独立 import、独立测试
- 现有功能不受影响

## R4: 工具能力标签化

**优先级**：P1 — 调度更智能，为未来并发执行打基础

**问题**：`_extract_typed_output()` 用 `"web" in lower_tool` 判断工具类型，脆弱易误判。工具无并发安全标记，无资源需求声明，调度器无法做资源感知。

**Claude Code 做法**：工具有结构化能力标签（concurrent_safe、needs_network 等），StreamingToolExecutor 据此决定并行/串行。

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | `ToolSpec` 扩展 `capabilities` 字段：`concurrent_safe: bool`、`output_type: str`、`needs_network: bool`、`destructive: bool` | `core/tool_protocol.py` | ✅ |
| 2 | 内置工具补充能力声明（web_search: needs_network+concurrent_safe, browser: needs_network+非concurrent_safe, file: destructive, terminal: destructive+非concurrent_safe） | `core/tool_registry.py` | ✅ |
| 3 | MCP 工具注册时，从 MCP Server 元数据推断或配置能力标签 | `core/tool_registry.py` | ✅ |
| 4 | `_extract_typed_output()` 改为读取 `ToolSpec.output_type`，替代字符串 heuristic | `core/task_executor.py` | ✅ |
| 5 | `_select_batch_indexes()` 利用 `concurrent_safe` 标记决定并行批次 | `core/task_executor.py` | ✅ |
| 6 | 单元测试 | `tests/test_tool_registry_unit.py` | ✅ |

### R4 完成记录（2026-04-01）

**新建文件**：无

**修改文件**：
- `core/tool_protocol.py`：`ToolSpec` 新增 4 个能力字段（`concurrent_safe`、`output_type`、`needs_network`、`destructive`），默认值保证向后兼容
- `core/tool_registry.py`：
  - 10 个内置工具全部补充能力声明
  - MCP 工具注册时按 `risk_level` 推断 `concurrent_safe`/`destructive`/`needs_network`
- `core/task_executor.py`：
  - `_extract_typed_output()` 改为通过 `_resolve_registered_tool()` 查取 `spec.output_type`，完全消除字符串 heuristic；未注册工具退化为通用兜底
  - `_select_batch_indexes()` 新增 `non_concurrent_safe_type` 追踪：`concurrent_safe=False` 的工具同类可并行、异类互斥（不影响原有 `serialized` 逻辑）
- `tests/test_tool_registry_unit.py`：新增 8 个 R4 专项测试
- `tests/test_multi_agent_unit.py`：修正虚构工具名 `terminal.shell` → `terminal.execute`（旧名依赖字符串 heuristic，R4 后需使用真实注册名）

**语义说明**：
- `concurrent_safe=False`：同类工具可并行（受 `max_parallelism` 约束），但不与其他工具混入同一批次；`serialized=True` 比 `concurrent_safe=False` 更严格（任何任务都不能共批）
- MCP high-risk → `serialized=True` + `concurrent_safe=False`；medium-risk → `destructive=True`；low-risk → 无额外限制

**注意事项**：
- `terminal.*` 工具共享同一 `task_type="terminal_worker"`，`get_by_task_type()` 只返回最后注册的（`terminal.search`）；调用时务必通过 `tool_name` 明确指定工具，不能依赖 task_type 解析
- `tests/test_graph_replanner_unit.py::test_replanner_preserves_direct_user_target_url` 在 R4 之前就已失败，属预存缺陷，与本次改动无关

**验收标准**：
- 所有内置工具有结构化能力声明
- `_extract_typed_output()` 零字符串 heuristic
- 并发调度基于能力标签而非硬编码

## R5: Plan Mode 持久化 + Reminder 机制

**优先级**：P2 — 长任务质量显著提升

**问题**：当前计划只是 Router LLM 输出的 JSON task_queue，执行过程中不再回顾，也无法恢复。模型在长任务中容易忘记原始目标、跳步、忘更新状态。

**Claude Code 做法**：
1. Plan Mode 是显式状态切换，规划态禁止写操作
2. 计划持久化为文件（`plans/{session}.md`），支持 resume/fork 恢复
3. Runtime 基于轮次数主动插 reminder，不信任模型自律

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | `OmniCoreState` 新增 `plan_mode: bool`、`plan_file_path: str` 字段 | `core/state.py` | 🔲 |
| 2 | 新建 `core/plan_manager.py`，实现 `save_plan(job_id, plan_json)`、`load_plan(job_id)`、`plan_to_markdown(plan_json)` | `core/plan_manager.py`（新建） | 🔲 |
| 3 | `route_node` 结束后调用 `save_plan()`，计划写入 `data/plans/{job_id}.md` | `core/graph_nodes.py` 或 `core/graph.py` | 🔲 |
| 4 | `replanner_node` 触发时同步更新 plan 文件 | 同上 | 🔲 |
| 5 | 新建 `core/plan_reminder.py`，实现 `generate_reminder(state) -> Optional[str]`：超过 N 轮未更新 → 注入提醒；任务完成率低 → 建议重规划 | `core/plan_reminder.py`（新建） | 🔲 |
| 6 | 在 graph 的 `dynamic_replan_node` 之前插入 reminder 检查，将提醒注入 messages | `core/graph.py` | 🔲 |
| 7 | resume 场景：`submit_job` 时如果检测到同 session 的 plan 文件存在，自动加载 | `core/runtime.py` | 🔲 |
| 8 | 配置项 `PLAN_PERSISTENCE_ENABLED`（默认 true）、`PLAN_REMINDER_INTERVAL`（默认 5 轮） | `config/settings.py` | 🔲 |
| 9 | 单元测试 | `tests/test_plan_manager_unit.py`（新建） | 🔲 |

**验收标准**：
- 每次任务规划自动生成 `data/plans/{job_id}.md`
- 超过 5 轮未更新任务状态时，system message 中出现提醒
- resume 场景能从文件恢复计划

## R6: System Prompt 静态/动态分离

**优先级**：P2 — 缓存友好，长期降成本

**问题**：`prompts/router_system.txt` 中角色定义、工具说明、动态工具列表、当前任务状态全部混在一起。每次任务状态变化，整个 prompt 都变了，无法利用 LLM API 的 prompt cache。

**Claude Code 做法**：System prompt 分为静态前缀（角色+能力说明，命中全局 cache）+ 动态边界标记之后的内容（当前状态、reminder 等）。动态信息尽量走 attachment lane（作为 user message 前缀注入）。

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | 将 `prompts/router_system.txt` 拆分为 `prompts/router_system_static.txt`（角色定义+固定规则）和 `prompts/router_system_dynamic.txt`（模板，含 `{{AGENT_CAPABILITIES}}`、`{{KNOWLEDGE_CONTEXT}}` 等占位符） | `prompts/`（拆分） | 🔲 |
| 2 | `core/router.py` 修改 prompt 组装逻辑：static 部分作为 system message，dynamic 部分作为 user message 前缀 | `core/router.py` | 🔲 |
| 3 | `core/llm.py` 支持 `system_prefix`（可缓存）和 `system_dynamic`（不缓存）分离传入 | `core/llm.py` | 🔲 |
| 4 | 其他 prompt（planner、critic、browser 等）同理拆分 | `prompts/*.txt` | 🔲 |
| 5 | 验证 LiteLLM 的 cache 行为（确认静态前缀不变时是否命中缓存） | 联调验证 | 🔲 |

**验收标准**：
- system prompt 静态部分跨请求不变
- 动态信息（工具列表、知识上下文、任务状态）不污染静态前缀
- 现有功能不受影响

## R7: Session Memory 后台提炼

**优先级**：P3 — 长会话连贯性终极方案

**问题**：当前会话没有"工作记忆"概念。上下文压缩（R1）只能丢信息，无法提炼关键状态。长会话中模型逐渐失去对早期决策的记忆。

**Claude Code 做法**：后台 fork 一个低权限 agent，定期（非每轮）提炼当前会话的工作记忆摘要。Compact 时优先保留 session memory 而非对话历史。Session memory agent 只有 Edit 权限，只能写指定文件。

| 步骤 | 任务 | 涉及文件 | 状态 |
|------|------|---------|------|
| 1 | 新建 `core/session_memory.py`，实现 `SessionMemoryManager`：维护 `data/sessions/{session_id}_memory.md` | `core/session_memory.py`（新建） | 🔲 |
| 2 | 实现提炼 prompt：输入为最近 N 轮对话 + 当前 session memory → 输出为更新后的 session memory | `prompts/session_memory_extract.txt`（新建） | 🔲 |
| 3 | 触发策略：每 `SESSION_MEMORY_INTERVAL` 轮（默认 8）触发一次，工具执行中不触发，最低 token 门槛 | `core/session_memory.py` | 🔲 |
| 4 | 提炼调用使用低成本模型（`complexity=low`） | `core/session_memory.py` | 🔲 |
| 5 | 在 R1 的 history snip 逻辑中：裁剪时将 session memory 作为首条系统消息保留 | `utils/context_budget.py` | 🔲 |
| 6 | 配置项 `SESSION_MEMORY_ENABLED`（默认 false，需手动开启）、`SESSION_MEMORY_INTERVAL`（默认 8） | `config/settings.py` | 🔲 |
| 7 | 单元测试 | `tests/test_session_memory_unit.py`（新建） | 🔲 |

**验收标准**：
- 长会话中每 8 轮生成一次 session memory 文件
- history snip 后 session memory 仍存在于上下文中
- 提炼过程不阻塞主流程
